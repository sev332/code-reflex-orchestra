# The Boundary Instrument Thesis

## Toward a Post-Magnetic Lasso: Human-Led, Field-Driven, Ambiguity-Aware Selection as a New Class of Tool

### Abstract

This document argues that the true evolution of the lasso is not object intelligence, semantic completion, or AI-driven segmentation disguised as manual work. The next paradigm is subtler, broader, and ultimately more durable: the lasso must become a boundary instrument.

A boundary instrument is a human-led selection system that does not attempt to understand objects in the semantic sense. Instead, it constructs and navigates a local contour field: a mathematically grounded landscape of edge attraction, tangent flow, curvature, ambiguity, and reversible chronology. The user remains the author of the path. The system acts as a physically legible guide, not as a covert substitute for judgment.

This thesis develops the conceptual, mathematical, interaction, architectural, and product foundations for such a system. It defines what the lasso is, what it is not, why AI is the wrong axis of evolution for this class of tool, how local assistance should work, how ambiguity must be handled, how path authorship can become temporally editable, and how post-path operations can transform the lasso from a mere outlining tool into a full selection construction system.

The central claim is simple:

**The future of the lasso is not greater autonomy. It is greater fidelity of cooperation.**

---

# 1. Introduction

## 1.1 The problem with the current framing of selection tools

Selection tools have largely diverged into two camps:

1. **Manual path tools** that offer precision but demand effort and constant correction.
2. **Automated segmentation tools** that offer convenience but collapse authorship into prediction.

The first camp often feels brittle, low-bandwidth, and technologically stale. The second often feels alienating, semantically presumptuous, and contrary to the reason many users reach for a lasso in the first place.

The lasso occupies a very particular niche in creative work. It is not merely a path tool. It is a negotiation interface between perception and intention. The user sees a boundary, feels where it should go, and needs a tool that helps them express that boundary without overruling them.

If the tool becomes too weak, it is tedious. If it becomes too strong, it ceases to be a lasso.

That tension is the entire design problem.

## 1.2 The wrong answer: semantic intelligence

There is a seductively lazy answer to this tension: make the lasso smarter. Add AI. Let it infer objectness. Let it guess intent. Let it bridge gaps by recognizing what the object probably is.

This is the wrong axis.

Once a lasso begins inferring semantics, it begins to leave the domain of boundary following and enter the domain of segmentation. That may be useful in a different tool, but it breaks the core identity of the lasso. The user no longer authors a path through visual evidence. The tool authors a belief about a region.

This changes the phenomenology, the control contract, the failure modes, and the trust model.

A lasso should fail locally and legibly.
A semantic segmenter fails globally and opaquely.

The former says, "I need your hand here."
The latter says, "I assumed a dog. Sorry about your bicycle spokes."

## 1.3 The correct answer: a boundary instrument

The lasso should evolve into a **boundary instrument**:

* human-led
* edge-biased
* locally assistive
* ambiguity-aware
* physically interpretable
* temporally reversible
* composition-ready

This is not a giant leap into automation. It is a deep refinement of cooperation.

The paradigm shift is not that the tool becomes sentient. The paradigm shift is that the tool becomes *structurally honest*.

It should only help where there is local evidence.
It should only resist when resistance is justifiable.
It should retreat when ambiguity rises.
It should expose uncertainty instead of disguising it.
It should make revision as fluid as drawing.

# 2. A definition of the lasso

## 2.1 What the lasso is

A lasso is a boundary-path authoring tool.

Its job is to let a user express a contour through space with partial assistance from local image structure. The user is not merely indicating a region. They are tracing a relationship to a boundary.

That means a lasso is fundamentally about:

* edges
* turns
* continuity
* ambiguity resolution
* path authorship
* local intent under uncertainty

It is not fundamentally about object detection.

## 2.2 What the lasso is not

A lasso is not:

* a semantic classifier
* a region completion oracle
* an object detector
* a mask predictor in disguise
* a one-click segmenter with an overly polite UI

These are different tools. They may coexist in the same product, but they must not be conflated.

## 2.3 The design covenant

Any true lasso evolution must obey the following covenant.

### The tool may:

* detect local contour structure
* estimate edge strength and orientation
* bias motion toward plausible nearby boundaries
* move more easily along than across strong contours
* weaken assistance at corners and junctions
* visualize confidence and ambiguity
* support graceful backtracking and revision
* convert paths into masks with exact, composable algebra

### The tool may not:

* infer object identity
* invent hidden contours because they are semantically likely
* jump across missing evidence using category knowledge
* silently replace path authorship with segmentation
* conceal uncertainty behind confident completion

If a system violates these rules, it may still be useful, but it is no longer a lasso.

# 3. Why subtle evolution is the true paradigm shift

## 3.1 Why giant leaps are usually category errors

When people imagine a major evolution of the lasso, they often imagine dramatic intelligence. But that usually means replacing the tool instead of improving it.

The most profound innovations are often not giant jumps in automation. They are expansions in **bandwidth**, **fidelity**, and **editability**.

A boundary instrument becomes revolutionary not by taking over the act, but by increasing the quality of the human-tool conversation.

That includes:

* better attraction to real edges
* better behavior at ambiguity
* better continuity under speed variation
* better revision mechanics
* better field awareness
* better post-path operations
* better diagnostics and control surfaces

The result feels radically better without becoming categorically different.

## 3.2 The lasso should remain imperfect in the right way

This is not a flaw. It is essential.

A perfect lasso would be one that always finds the intended object boundary without needing the user. But that is just segmentation.

The lasso must remain imperfect because its value lies in *negotiated precision*. It must preserve the user’s role in determining the path. It should make that role easier, cleaner, and more expressive, but never obsolete.

The best lasso is not one that knows too much.
It is one that refuses to pretend.

# 4. The field-driven paradigm

## 4.1 From snapping to topography

Traditional magnetic lasso metaphors tend to imply discrete snapping: the cursor passes near an edge, the tool finds a likely point, and the path is pulled there.

This model is crude. It often feels point-like, brittle, and discontinuous.

A boundary instrument should instead construct a **local contour topography**.

In this model, the image is transformed into a continuous field with at least the following elements:

* edge magnitude
* contour tangent direction
* corner or junction intensity
* a scalar height or energy surface
* local gradient toward valley floors
* confidence and ambiguity measures derived from recent motion

The lasso then moves not through arbitrary pixel space, but through a shaped physical landscape.

Strong edges become valleys or channels.
Tangents become preferred directions of travel.
Corners become instability zones.
Flat areas become low-information plateaus.

This is a far richer and more legible model than snap-to-nearest-edge.

## 4.2 Why local fields beat semantic inference

A local field offers several advantages:

1. **Interpretability**: every motion bias comes from nearby evidence.
2. **Speed**: the computations are local, deterministic, and GPU-friendly.
3. **Responsiveness**: the system can run at high frequency without semantic bottlenecks.
4. **Trust**: the user can learn the tool because it behaves according to image structure, not mysterious guesses.
5. **Failure honesty**: when evidence weakens, assistance weakens.

This is exactly how the lasso should evolve: not toward understanding, but toward better local mechanics.

# 5. The mathematical anatomy of a boundary instrument

## 5.1 The contour field

At the core of the system lies a contour field extracted from the image. The exact implementation may vary, but the conceptual stack is stable.

### 5.1.1 Edge evidence

The system should estimate edge strength from local gradients, ideally with rotationally stable operators and some degree of pre-smoothing to suppress noise.

### 5.1.2 Tensor structure

A structure tensor or related local orientation model is especially valuable because it does more than detect edges. It distinguishes between:

* coherent contour structure
* isotropic texture or noise
* corner or junction-like events

This allows the field to know not only where strength exists, but how organized that strength is.

### 5.1.3 Tangent flow

A contour is not only a place to move toward. It is also a direction to move along.

This means the field should encode a tangent orientation so that once the lasso is near a contour, travel along the contour is energetically favored while motion across it becomes more difficult.

### 5.1.4 Scalar terrain

The field should also be converted into a scalar terrain or energy landscape.

One useful conceptualization is that strong contours become low valleys. The lasso is gently pulled downhill toward these valleys and then flows along them.

This produces an intuitive dynamical system:

* descent toward boundaries
* travel along boundaries
* reduced stability at corners and junctions
* edge search when lost in weak-information areas

## 5.2 Sub-pixel continuity

A serious lasso cannot remain trapped at pixel resolution. The field must be sampled continuously or near-continuously.

This means:

* sub-pixel interpolation of edge strength
* sub-pixel interpolation of tangent flow
* analytical or near-analytical gradients of the scalar terrain
* smooth transitions across pixel boundaries

Without this, the system will chatter, jitter, and produce the familiar crunchy feel of outdated tools.

## 5.3 The motion equation

A useful lasso state update is not a single snap rule but a composition of forces.

At minimum, four forces are conceptually justified:

### 5.3.1 Edge attraction

A gradient-descent force that pulls the head downhill toward nearby contour valleys.

### 5.3.2 Tangent flow

A directional force that moves the head along the contour direction, with sign determined by user motion or user pull intent.

### 5.3.3 Cursor spring

A tether force that ensures the tool remains human-led. This is essential. The path must remain bound to the cursor’s broader intention and not become self-propelled.

### 5.3.4 Path stiffness or inertia

A continuity force that preserves smooth forward motion and prevents every tiny perturbation from causing jagged local chaos.

These forces must not be fixed. They must be modulated by context.

## 5.4 Contextual modulation

The motion system should adapt based on local conditions.

Examples:

* strong edge, cursor moving parallel to contour: spring weakens, flow strengthens
* strong corner: momentum weakens sharply
* weak edge: attraction and confidence fall
* high ambiguity: automation recedes, user authority rises
* rapid motion: spacing and responsiveness curves adapt

This is where the lasso becomes intelligent in the right way: not semantic intelligence, but dynamical intelligence.

# 6. Ambiguity as a first-class design problem

## 6.1 The central truth

The most important moments in lassoing are not clear edges. Clear edges are easy. The real design battle is fought at ambiguity.

Ambiguity appears in:

* corners
* junctions
* T-intersections
* X-crossings
* weak-contrast regions
* fur and hair
* textured backgrounds
* parallel edge families
* internal texture that competes with boundary texture
* broken or intermittent contours

A mediocre tool treats these cases as edge-detection failures.
A great tool treats them as changes in the control contract.

## 6.2 The principle of graceful retreat

When ambiguity rises, assistance should not become more authoritarian. It should become more deferential.

This means the system should:

* reduce edge-lock intensity
* reduce contour-following authority
* reduce momentum preservation
* increase manual dominance
* communicate uncertainty visibly or tactically

At ambiguity, the tool should not say, "I will decide for you."
It should say, "This is your turn. I will steady the instrument, not choose the branch."

## 6.3 Ambiguity grammar

A mature lasso should not merely expose one scalar confidence value. It should model a grammar of ambiguity.

Examples include:

### 6.3.1 Junctionness

How likely is the region to support multiple viable outgoing boundary directions?

### 6.3.2 Side ambiguity

Are there two nearby parallel edges competing for ownership?

### 6.3.3 Texture ambiguity

Is the lasso traversing repeated small structures rather than a stable silhouette?

### 6.3.4 Continuity break

Has the previously stable boundary evidence weakened or fragmented?

### 6.3.5 Intent conflict

Is the cursor pull direction increasingly inconsistent with the contour flow the system is proposing?

A full ambiguity model enables smarter handoff without requiring semantic knowledge.

## 6.4 Proposed ambiguity mechanisms

This thesis proposes several major additions.

### 6.4.1 Confidence deadbands

Rather than continuously scaling help linearly, define confidence regimes:

* **high confidence**: strong contour bias, low spring dominance
* **mid confidence**: balanced assistance
* **low confidence**: assistance fades sharply, manual control dominates

Deadbands make the tool behavior more legible and less mushy.

### 6.4.2 Junction brake cones

At strong corners or crossings, create angular zones where momentum is damped and contour-following cannot overshoot recklessly. This prevents the classic wide swing past a corner.

### 6.4.3 Contour-family hysteresis

When the lasso has been riding a particular side of an edge family, introduce a small resistance to jumping laterally to a neighboring parallel contour unless the cursor commits strongly.

This solves the zipper-chatter problem without semantic reasoning.

### 6.4.4 Side-lock bias

Once the tool identifies that the user is tracing the outer or inner side of a boundary family, maintain a weak directional preference to remain on that side.

This is especially powerful in fur, foliage, clothing folds, and anti-aliased silhouette bands.

### 6.4.5 Manual clutch zones

Allow the user to temporarily suppress assistance by pressure, key clutch, or stylus gesture so they can resolve difficult transitions without fighting the field.

# 7. The temporal thesis: path as chronology, not just geometry

## 7.1 The hidden poverty of conventional path tools

Most lasso tools treat the path as a static growing polyline. Once added, points become inert historical facts. Backtracking is clumsy, often modal, and psychologically expensive.

This is primitive.

The lasso should not merely accumulate geometry. It should accumulate **editable chronology**.

## 7.2 The floating tail

A superior model is to split the path into:

* a baked historical body
* a floating recent tail

The floating tail remains fluid, revisable, and dissolvable. The user can retreat through it without explicit undo commands. This makes the path feel alive.

## 7.3 Organic unzip backtracking

Backtracking should not require mode switches or button rituals. If the user physically moves backward over recent trajectory, the path should unzip naturally and kill local momentum.

This is one of the most important experiential upgrades available to the lasso. It changes revision from an administrative action into a motor gesture.

That is a paradigm shift of feel, not just engineering.

## 7.4 Manual anchors as declarations of intent

Anchors should remain available, but their role changes.

In the old model, anchors were a crude way to force certainty.
In the new model, anchors become intentional declarations:

* commit this branch
* pin this corner
* seal this ambiguous transition
* collapse the floating tail into path history

Anchors are no longer just technical fixes. They are semantic acts by the human, while the system remains non-semantic.

## 7.5 Closure as gestalt, not guesswork

Closure snapping can be useful, but it must remain local and geometric. A boundary instrument may detect return-to-origin conditions and gently close, but it must not infer hidden enclosure from semantic objectness.

# 8. The tactile thesis: the lasso must feel like an instrument

## 8.1 Why feel matters

The most important quality of a lasso is not raw correctness but control feel.

Users do not experience the underlying tensor or force balance directly. They experience drag, stickiness, release, obedience, resistance, and revision.

A world-class lasso must therefore be designed as a tactile system.

## 8.2 Desired tactile states

A mature lasso should be able to inhabit multiple tactile states without becoming confusing.

Examples:

* **glide**: low resistance in open space
* **catch**: gentle attraction toward a plausible boundary
* **ride**: stable travel along a strong contour
* **brace**: reduced drift near corners
* **release**: quick manual override when confidence drops
* **unzip**: frictionless revision of recent path history
* **pin**: manual commitment at decisive points

These are not marketing adjectives. They should map directly to physical parameter regimes.

## 8.3 Pressure and stylus as continuous control channels

Pressure should not be wasted on simple brush-like metaphors. It can become a true control dimension for the lasso.

Examples:

* increase edge search radius
* increase or decrease attraction bandwidth
* temporarily widen ambiguity tolerance
* increase clutch suppression of automation
* alter anchor commitment behavior

This turns the lasso into something closer to a precision surgical instrument than a flat cursor mechanic.

## 8.4 Dynamic response curves

Static parameters are blunt. A modern lasso should support dynamic parameter graphs driven by live metrics such as:

* cursor speed
* head speed
* edge strength
* confidence
* pressure
* local ambiguity

These curves should be editable, inspectable, and even surfaceable in the UI. This makes the lasso not only more powerful, but tunable to different artistic styles and image domains.

---

# 9. Beyond luminance: richer boundary cues without AI

## 9.1 Why luminance alone is insufficient

Many classic edge systems over-rely on luminance. This is inadequate for modern workflows.

Boundaries often exist in:

* chroma shifts
* saturation changes
* texture discontinuities
* alpha edges
* depth discontinuities
* normal changes in layered or 3D-aware contexts

A lasso that only sees grayscale gradients is half-blind.

## 9.2 A multi-channel boundary field

The next generation should build not a single edge map but a composite boundary field from multiple local evidence channels.

Potential channels include:

* luminance gradient
* chroma gradient in perceptual color space
* saturation change
* local entropy or texture discontinuity
* alpha edge energy
* depth edge energy
* normal discontinuity
* user-painted hint channels

These channels need not be semantically interpreted. They can simply contribute to a fused local contour evidence model.

## 9.3 Domain-specific profiles

Different images demand different cue weighting.

Examples:

* portrait cutout: skin-hair-background chroma contrast matters greatly
* product photography: specular and silhouette edges dominate
* matte painting: texture boundaries and atmospheric transitions matter
* compositing from rendered assets: alpha and depth cues may be strongest

The tool can expose profiles or adaptive local weighting without invoking AI semantics.

# 10. The mask thesis: the lasso must become a selection construction system

## 10.1 The lasso should not end at path closure

Historically, the lasso has been treated as an input mechanism whose job ends once a region is enclosed.

This undersells its true potential.

A mature lasso should continue seamlessly into exact mask generation and mask algebra.

## 10.2 Path-to-mask exactness

The conversion from path to mask should be mathematically rigorous:

* strict closure handling
* parity-safe scanline logic
* sub-pixel edge coverage
* anti-aliased boundary fill
* stable behavior at shared vertices and degenerate segments

This matters because a brilliant pathing system can be ruined by sloppy rasterization.

## 10.3 Boolean composition

Selections are rarely one-pass monoliths. Users construct them iteratively.

Therefore the lasso should integrate naturally with:

* union
* intersection
* subtraction
* xor-like difference

This is not an afterthought. It is core to turning the lasso into a compositional system.

## 10.4 Morphological refinement

Expansion, contraction, feathering, and related refinements should be fast enough to feel native and immediate.

This matters because it lets the lasso remain the center of selection creation even when final adjustment requires region-level refinement.

## 10.5 The philosophical shift

At this point, the lasso ceases to be merely a contour drawer.
It becomes a **selection constructor**.

The path is one expression layer.
The mask is another.
The boolean graph is another.
The morphological refinement stage is another.

This is how a humble lasso becomes infrastructure.

# 11. Architecture for responsiveness

## 11.1 Responsiveness is not optional

The lasso lives or dies by latency.

A brilliant field model running sluggishly is worthless. The tool must feel immediate. That means:

* field generation must be efficient
* per-frame updates must be lightweight
* UI must not interrupt the simulation loop
* telemetry and configuration must be decoupled from the motion step
* heavy preprocessing should live on GPU where practical

## 11.2 Separation of concerns

A high-performance boundary instrument should be partitioned into distinct layers:

### 11.2.1 Field engine

Builds the contour field and associated scalar and directional data.

### 11.2.2 Motion engine

Runs the continuous update loop and integrates forces.

### 11.2.3 Temporal engine

Maintains trails, history, floating tail, backtracking logic, and anchors.

### 11.2.4 Mask engine

Converts paths to masks and performs compositional refinement.

### 11.2.5 Parameter graph engine

Evaluates live response curves at frame rate outside expensive UI state systems.

### 11.2.6 UI and diagnostics layer

Displays telemetry, ambiguity, confidence, and editable response surfaces without owning the real-time loop.

## 11.3 GPU acceleration where it belongs

GPU work is especially well suited for:

* gradient extraction
* tensor construction
* blurs
* local field passes
* possibly auxiliary diagnostic buffers

But the crucial principle remains:

**GPU acceleration should strengthen responsiveness, not justify semantic bloat.**

The purpose of compute here is not to sneak in a giant model. It is to make local mathematics fast enough to disappear into feel.

# 12. Diagnostics as creative power

## 12.1 Most tools hide too much

Selection tools usually hide their internal state. Users see only the final path and maybe a cursor icon. This is a mistake.

A boundary instrument becomes much more powerful when it reveals selected parts of its internal dynamics.

## 12.2 Useful live diagnostics

Potential diagnostics include:

* edge strength under head
* confidence level
* ambiguity class
* cursor-lasso alignment
* local corner intensity
* side-lock state
* current force contributions
* local field preview
* candidate contour family visualization
* hysteresis state

These are not only for debugging. They help expert users build intuition and tune the instrument to image types and personal preference.

## 12.3 Diagnostics for testing and research

The same visibility is invaluable for building a world-class test platform.

One can create benchmark scenes and diagnostic overlays for:

* clean hard edges
* anti-aliased silhouettes
* transparent hair
* repeating textures
* fur and foliage
* crossings and intersections
* low-contrast contours
* motion-speed stress tests
* device-specific latency testing

This transforms the lasso project from a clever tool into a research platform.

# 13. Product differentiation: why this can shatter expectations

## 13.1 The false dichotomy in current tooling

Most products implicitly force a choice:

* use old manual tools for control
* use AI tools for convenience

That leaves a huge neglected zone in between: **high-fidelity assisted authorship**.

This is where the boundary instrument lives.

## 13.2 Why it can beat legacy magnetic lasso

Legacy magnetic lasso is generally limited by:

* weak field models
* poor ambiguity behavior
* crude point acquisition
* bad backtracking ergonomics
* low diagnostic visibility
* insufficient adaptability
* limited post-path integration

A boundary instrument surpasses it by making every layer more intelligent in the non-semantic sense.

## 13.3 Why it does not compete directly with AI segmentation

This is critical. The goal is not to defeat AI segmentation at its own game. That would be foolish and unnecessary.

The goal is to own a different category:

* high-trust
* high-control
* low-latency
* ambiguity-honest
* composition-centric
* craftsmanship-preserving

AI segmentation is for one kind of workflow.
A boundary instrument is for another.

The right product does not collapse them into one confused blob. It makes both available, clearly distinguished, and philosophically coherent.

# 14. A critique of anthropomorphic framing

## 14.1 Why words matter

It is tempting to describe tool behavior with words like sentience, soul, intelligence, posterior belief, or understanding. These phrases can be evocative during ideation, but they become dangerous if they blur architectural truth.

A lasso should not be framed as alive or semantically aware.

It should be framed as:

* field-sensitive
* dynamically coupled
* ambiguity-aware
* tactilely responsive
* temporally editable

This preserves conceptual rigor.

## 14.2 Use poetry in vision, not in invariants

Poetic names can still exist at the UI or project mythology level. But the implementation and design doctrine should use precise terms.

For example:

* "texture soul" might become **micro-structure response mode**
* "sentience" might become **local evidence adaptation**
* "Bayesian posterior proxy" might become **confidence score** unless it is truly probabilistic

This is not pedantry. It protects the architecture from self-mystification.

# 16. Success criteria

## 16.1 What success looks like experientially

A successful boundary instrument should feel like this:

* it catches the contour without lunging
* it follows without dragging the user into nonsense
* it releases gracefully when the image becomes ambiguous
* it rounds difficult corners without overshooting
* it lets the user revise by motion rather than bureaucracy
* it produces masks that need less cleanup
* it rewards skill without demanding masochism

## 16.2 What success looks like empirically

The system should be evaluated on:

* path deviation from human-approved target contours
* overshoot rate at corners and junctions
* branch-switch rate in parallel-edge regions
* revision efficiency during backtracking
* latency under real workloads
* number of manual corrections per contour length
* user preference in expert workflows versus legacy magnetic lasso

## 16.3 The deeper success criterion

The deepest success criterion is not simply speed or accuracy. It is this:

**Does the user feel more authorship, not less, while getting more help, not less?**

That is the whole game.

* guided by physically interpretable forces
* respectful of ambiguity

This is the subtle revolution.

# Part II. Execution Stack

## From Thesis to Build System

This part translates the Boundary Instrument Thesis into a concrete execution stack that can guide implementation, evaluation, and product discipline. The purpose is not merely to describe the ideal tool, but to establish a layered system of invariants, equations, interaction rules, diagnostics, benchmarks, and module boundaries that make the ideal tool buildable.

The execution stack is divided into five layers:

1. Product doctrine spec
2. Math and physics spec
3. Interaction and control spec
4. Benchmark and diagnostics platform
5. Implementation roadmap

Each layer should be treated as necessary. If the project skips any one of them, the result will become fragile, confused, or regress toward legacy lasso behavior.

---

# 18. Product Doctrine Spec

## 18.1 Purpose of the doctrine layer

The doctrine layer exists to prevent category drift.

Without an explicit doctrine, a powerful lasso project slowly deforms under pressure from feature requests, comparative marketing, and engineering temptation. One developer adds gap completion. Another adds semantic region hints. Another adds auto-closure based on likely object continuation. Before long, the lasso has become a confused hybrid of contour follower and object segmenter.

The doctrine layer protects the project from this fate.

Its purpose is to answer, before implementation decisions are made:

* what the tool fundamentally is
* what it is fundamentally not
* what kinds of assistance are lawful
* what kinds of assistance are forbidden
* what tradeoffs always resolve in favor of authorship
* what user promises the system must keep under all conditions

## 18.2 Product identity statement

### Canonical statement

The Boundary Instrument is a human-led contour authoring system that helps users trace real image boundaries through local field guidance, ambiguity-aware assistance, reversible chronology, and exact mask construction.

### Prohibited reinterpretation

The Boundary Instrument is not an object segmentation system, not a semantic predictor, not a hidden-region inference engine, and not an autonomous selection mechanism disguised as a manual tool.

## 18.3 Non-negotiable user promises

The product must keep the following promises.

### Promise 1: Human authorship remains primary

The path must always remain meaningfully attributable to the user’s gesture and local decisions. Assistance may refine, stabilize, and guide, but it may not replace authorship.

### Promise 2: Assistance is justified only by local evidence

When the system exerts force, snaps, bias, or resistance, that behavior must be explainable by nearby contour evidence, not by semantic priors or distant inference.

### Promise 3: Ambiguity never becomes silent overreach

When the image becomes ambiguous, the tool must reduce confidence and yield authority back to the user rather than hallucinate certainty.

### Promise 4: Revision must be fluid

The user must be able to change their mind continuously. Backtracking, re-steering, and partial re-commitment are not secondary features; they are fundamental.

### Promise 5: The output must compose cleanly

The lasso is not finished at path closure. It must feed robustly into exact mask construction, boolean composition, and refinement workflows.

## 18.4 Allowed feature classes

The product doctrine explicitly permits the following categories of evolution.

### 18.4.1 Local contour analysis

Examples:

* better gradient extraction
* richer tensor coherence logic
* curvature-sensitive corner detection
* better edge continuity metrics
* multi-channel evidence fusion

### 18.4.2 Dynamic assistance mechanics

Examples:

* stronger tangent riding
* corner braking
* ambiguity deadbands
* side-lock hysteresis
* clutch modulation
* pressure-sensitive attraction bandwidth

### 18.4.3 Temporal editing mechanics

Examples:

* floating tail refinement
* organic unzip variants
* branch-safe re-entry
* timeline visualization
* soft anchors versus hard anchors

### 18.4.4 Exact output operations

Examples:

* higher-quality path rasterization
* better anti-aliasing
* morphology improvements
* boolean graph composition
* local feather profiles

### 18.4.5 Diagnostics and tuning

Examples:

* force contribution overlays
* ambiguity class indicators
* response curve editing
* replay tooling
* failure taxonomy dashboards

## 18.5 Forbidden feature classes

The doctrine explicitly forbids the following categories inside the lasso core.

### 18.5.1 Semantic object inference

The lasso core must not identify categories such as person, dog, bicycle, cup, leaf, building, hair, or sky and alter contour behavior because of those categories.

### 18.5.2 Hidden contour completion from semantic priors

The lasso core must not bridge gaps because it “knows” how an object boundary probably continues.

### 18.5.3 Autonomous branch resolution at ambiguous intersections

At branch points, the tool may bias weakly according to local continuity and user motion history, but it may not silently commit major branch choices without strong user steering.

### 18.5.4 Semantic mask post-correction disguised as contour assistance

The system must not draw one path and then invisibly replace it with a semantically cleaner mask while pretending that the pathing logic did the work.

## 18.6 Boundary Instrument modes should remain within one philosophy

Different modes may exist, but they must all obey doctrine.

Possible lawful modes:

* free lasso with minimal assistance
* field lasso with standard assistance
* precision mode with stronger corner braking
* stylus mode with pressure-modulated control
* texture-guard mode for repeated microstructure
* compositing mode with alpha/depth cue fusion

These are all still boundary instruments. They differ in local mechanics, not product identity.

## 18.7 Relationship to AI tools in the larger product

The product may still offer AI segmentation elsewhere. But doctrine requires a bright line.

### Required separation principles

* AI tools and Boundary Instrument tools must have distinct naming and UI framing.
* The lasso core must not silently call semantic models.
* If a hybrid workflow exists, the transition between manual boundary authorship and AI suggestion must be explicit and reversible.
* Selection provenance should remain knowable: the user should be able to tell whether a region arose from path authorship, boolean composition, or semantic mask suggestion.

## 18.8 Product acceptance criteria

A feature should only be admitted into the Boundary Instrument if it satisfies all of the following.

1. It improves contour authorship rather than replacing it.
2. Its effects are explainable by local evidence and recent user interaction.
3. It degrades gracefully under ambiguity.
4. It preserves or improves revision fluidity.
5. It keeps latency low enough to feel instrument-like.
6. It integrates honestly with path-to-mask workflows.

# 19. Math and Physics Spec

## 19.1 Purpose of the math layer

The math layer translates doctrine into mechanics. It defines what the field is, how it is sampled, what state the lasso maintains, what forces act on it, how ambiguity is represented, and how updates are integrated without turning the system into jitter soup.

The emphasis should remain on:

* locality
* determinism
* interpretability
* sub-pixel continuity
* bounded dynamics
* controllable feel

## 19.2 State representation

A minimal but serious lasso head state should include:

* head position `x`
* head velocity `v`
* preferred tangent direction `t_prev`
* recent arc-length parameterization
* floating tail sample ring
* current anchor state
* confidence state `c`
* ambiguity vector `a`
* side-lock state `s`
* contour-family hysteresis state `h`
* clutch state `u`
* current dynamic parameters sampled from response curves

Optional advanced state may include:

* local curvature estimate history
* branch-candidate scores
* local field sample cache
* short-horizon user intent vector
* reversible chronology markers

## 19.3 Field representation

The field should be represented as a set of sampled textures or analytical approximants containing:

* edge magnitude field `E(x)`
* contour tangent field `T(x)`
* scalar terrain or energy field `Phi(x)`
* corner/junction field `J(x)`
* coherence field `K(x)`
* texture ambiguity field `Q(x)`
* chroma boundary field `C(x)` and optional fused channels

The system should treat these as distinct but related maps. Over-compressing them into one scalar destroys too much information.

## 19.4 Multi-channel contour evidence fusion

A generalized contour evidence model may be written as:

`B(x) = wL * BL(x) + wC * BC(x) + wS * BS(x) + wT * BT(x) + wA * BA(x) + wD * BD(x) + wN * BN(x)`

Where:

* `BL` is luminance boundary evidence
* `BC` is chroma boundary evidence
* `BS` is saturation boundary evidence
* `BT` is texture discontinuity evidence
* `BA` is alpha boundary evidence
* `BD` is depth discontinuity evidence
* `BN` is normal discontinuity evidence

The weights may be global, mode-specific, or locally adaptive, but adaptation must remain based on signal quality rather than semantics.

## 19.5 Suggested field extraction pipeline

### Stage A: preconditioning

* convert source to perceptual working space where useful
* build luminance and chroma representations
* optionally denoise lightly while preserving edges
* build texture descriptor pyramid for texture discontinuity channels

### Stage B: derivative estimation

* compute robust gradient estimates for relevant channels
* compute magnitude and orientation fields
* produce sub-pixel-friendly filtered derivative maps

### Stage C: tensor analysis

* build local structure tensor
* derive principal orientation
* derive coherence or anisotropy
* derive cornerness or junctionness

### Stage D: scalar terrain synthesis

Produce `Phi(x)` such that strong reliable contours form traversable low-energy channels. Suggested components:

* negative contribution from fused boundary evidence
* positive penalty for corner/junction instability
* positive penalty for texture ambiguity
* regularization term to keep terrain smooth enough for stable gradients

### Stage E: tangent orientation stabilization

Compute `T(x)` as orientation along contours with sign chosen at runtime according to user motion continuity, not fixed globally.

## 19.6 Force model

The head update should combine multiple force components. A suitable generalized model is:

`F_total = alpha_e * F_edge + alpha_t * F_tangent + alpha_s * F_spring + alpha_i * F_inertia + alpha_b * F_brake + alpha_h * F_hysteresis + alpha_u * F_useroverride`

### 19.6.1 Edge force

`F_edge = -grad Phi(x)`

This is the primary attraction into contour valleys.

### 19.6.2 Tangent flow force

`F_tangent = sigma * normalize(T(x))`

Where sign `sigma` is chosen from the projected user-intent direction and recent head motion consistency.

### 19.6.3 Cursor spring force

`F_spring = k_s * (x_cursor_proxy - x)`

Where `x_cursor_proxy` may be a filtered cursor position or intentional pull point, not necessarily raw cursor location.

### 19.6.4 Inertial smoothing force

This term preserves motion continuity and reduces jitter. It may be implemented as velocity memory, path stiffness, or a curvature-penalizing continuation bias.

### 19.6.5 Brake force

This term becomes active near corners, strong ambiguity, or when user and contour predictions diverge sharply. It reduces overshoot.

### 19.6.6 Hysteresis force

This is the mechanism that resists lateral chatter between nearby competing contour families once a local commitment has been established.

### 19.6.7 User override force

This becomes dominant when clutching or when confidence falls into the low deadband.

## 19.7 Dynamic coefficient modulation

The force coefficients `alpha_*` must be functions of context, not fixed constants.

Examples:

* `alpha_e = f1(B, K, c, pressure)`
* `alpha_t = f2(K, c, speed_along_contour)`
* `alpha_s = f3(cursor_distance, low_confidence, clutch)`
* `alpha_b = f4(J, ambiguity_peak, curvature_predicted)`
* `alpha_h = f5(side_lock_strength, parallel_edge_risk)`

The response curves should be editable and inspectable.

## 19.8 Confidence model

Confidence should not be a vague scalar. It should be a composed estimate derived from local evidence and motion alignment.

One possible factorization:

`c = g(edge_strength, coherence, user_alignment, roughness_penalty, corner_penalty, continuity_score)`

Where:

* edge strength rewards strong boundaries
* coherence rewards organized flow
* user alignment rewards agreement between user direction and contour direction
* roughness penalty reduces confidence on noisy or jagged micro-trajectories
* corner penalty reduces confidence near branch-heavy regions
* continuity score rewards staying on a stable boundary family

## 19.9 Ambiguity vector

Instead of a single uncertainty number, define:

`a = [a_junction, a_side, a_texture, a_continuity, a_intent]`

Where each component is normalized to a common range.

This allows the system to respond differently depending on the type of ambiguity.

Examples:

* high `a_junction`: activate brake cones and manual dominance
* high `a_side`: strengthen side-lock hysteresis
* high `a_texture`: reduce edge attraction bandwidth and warn of microstructure trap
* high `a_continuity`: lower tangent confidence and widen manual control
* high `a_intent`: trust the user over the contour flow immediately

## 19.10 Confidence deadbands

Define three primary behavioral regimes.

### High-confidence regime

* strong contour attraction
* strong tangent riding
* reduced spring dominance
* low jitter tolerance
* stable edge-lock feel

### Mid-confidence regime

* balanced forces
* moderate contour guidance
* more explicit user ownership
* no hard locking

### Low-confidence regime

* assistance collapses quickly
* spring and direct user influence dominate
* edge-lock nearly disabled
* diagnostic signaling strengthened

These deadbands create crisp, learnable behavior rather than mushy interpolation across all states.

## 19.11 Junction brake cones

At positions with high junctionness or cornerness, define angular cones centered on unstable outgoing branches. Inside these cones:

* tangent flow authority drops
* braking rises
* inertial continuation weakens
* overshoot distance cap tightens

This prevents the head from swinging past corners simply because momentum was previously correct.

## 19.12 Side-lock and contour-family hysteresis

Parallel or near-parallel contour families are one of the classic failure modes of edge-following tools. The solution should remain local and mechanical.

### Side-lock state

Estimate which side of a contour family the user is tracing from local normal direction, early approach geometry, and recent path-side consistency.

### Hysteresis mechanism

Introduce a switching cost when moving laterally between comparable nearby contour channels unless the cursor intent provides clear evidence of desired transfer.

This cost may be implemented as:

* an energy penalty in `Phi`
* an additional lateral damping term
* an explicit contour-family state machine with persistence decay

## 19.13 Integration scheme

The update integrator must emphasize stability. A simple explicit Euler scheme may suffice if the timestep is small and forces are bounded, but semi-implicit integration or bounded-step projection may produce better stability.

Key requirements:

* clamp maximum velocity
* clamp maximum per-step displacement
* filter high-frequency sign flips in tangent direction
* enforce bounded response when entering corners
* gracefully handle field singularities and flat regions

## 19.14 Lost-state behavior

When the head exits coherent contour guidance entirely, the system should not panic.

Lost-state behavior should include:

* fading tangent authority
* widening search visualization if desired
* stronger user spring dominance
* optional weak re-acquisition attempts inside a small radius
* no long-range snapping

The tool should become a clean manual lasso again, not a desperate treasure hunter.

## 19.15 Numerical testing requirements

The math layer should include automated tests for:

* convergence toward isolated ideal contours
* stable travel along long edges
* overshoot at controlled corners
* branch-switch resistance in parallel-edge fields
* ambiguity deadband transitions
* re-acquisition after lost-state events
* reproducibility across frame rates and timestep jitter

# 20. Interaction and Control Spec

## 20.1 Purpose of the interaction layer

The interaction layer defines how the tool feels in the hand. This is where raw mechanics become usable craft.

The most important principle is that interaction should map continuously onto the math layer rather than existing as arbitrary UI glue.

## 20.2 Core interaction states

The system should explicitly model the following states.

### State A: free glide

The user is moving through low-information space or intentionally approaching a contour. Assistance is minimal.

### State B: contour catch

The tool begins to feel local evidence and gently aligns the head toward a plausible boundary.

### State C: contour ride

The head travels stably along a coherent contour while remaining spring-bound to user direction.

### State D: brace

The head nears a corner, junction, or ambiguity pocket. Speed and momentum reduce. Manual steering becomes more consequential.

### State E: release

Confidence drops or the user clutches. Assistance fades quickly.

### State F: unzip

Recent path history becomes fluid and retractable through reverse motion.

### State G: anchor commit

The user explicitly commits a point, branch, or local contour decision.

These states need not be visually labeled, but they should be mechanically real.

## 20.3 Cursor-to-head relationship

The cursor should not always be identical to the lasso head. A direct lock can make the system too twitchy, while too much separation makes it feel disobedient.

A better model introduces a proxy relationship:

* raw cursor input
* filtered intent vector
* head state influenced by spring and contour forces

The key is that separation must remain small, interpretable, and reversible.

## 20.4 Manual clutching

The user needs a temporary override channel that is not equivalent to abandoning the tool.

Possible clutch semantics:

* hold modifier key to suppress contour attraction
* stylus pressure above threshold reduces assistance
* stylus barrel button forces manual dominance
* touch gesture expands free-glide radius

Clutching should be immediate, not modal.

## 20.5 Anchors and commits

Anchors should be redesigned from crude control points into a richer vocabulary.

### Soft anchor

Lightly increases local commitment and stabilizes recent path memory without fully freezing behavior.

### Hard anchor

Freezes the committed point as a firm path milestone.

### Corner pin

Declares that a local corner decision is intentionally authored by the user and should resist later smoothing or reinterpretation.

### Branch commit

States that the chosen branch at a junction is deliberate. Hysteresis around that choice becomes much stronger.

## 20.6 Floating tail behavior

The floating tail should be defined explicitly.

Possible rules:

* maintain most recent `N` samples or recent `L` arc-length as mutable tail
* tail remains editable until committed by distance, anchor, or explicit event
* reverse motion within a tolerance corridor causes tail unzip
* unzip should damp local inertia rapidly
* branch-safe logic should prevent accidental destruction of older committed path body

## 20.7 Unzip mechanics

Unzip is not merely deleting points. It is a state transition.

When reverse intent is detected:

* local head velocity decays
* tail samples become candidate retract targets
* contour attraction weakens to avoid fighting the retreat
* spring dominance increases toward reverse cursor motion
* committed body remains protected unless retreat exceeds threshold or anchor semantics permit crossing back

## 20.8 Corner interaction grammar

Corners are where most tools embarrass themselves.

A mature corner grammar should include:

* approach damping
  n- optional sticky corner pin suggestion
* reduced smoothing through the vertex
* immediate manual priority after apex if ambiguity is high
* resumed contour ride only after stable outgoing edge evidence forms

## 20.9 Side-lock interaction semantics

Side-lock should be mostly invisible but diagnosable.

User-facing rules:

* when tracing a silhouette edge family, the tool tends to stay on the same side unless the user deliberately crosses
* a transient UI cue may reveal that side persistence is active
* clutching temporarily weakens side-lock
* explicit branch commit can strengthen side-lock persistence through short ambiguous runs

## 20.10 Pressure grammar

Stylus pressure should not simply scale brush width. It can encode contour intent.

Possible mappings:

* low pressure = stronger assistance and contour ride
* medium pressure = balanced standard mode
* high pressure = clutch-like manual dominance
* pressure spikes = temporary anchor insertion or corner pin suggestion

Alternative mappings may be offered, but they should remain coherent and discoverable.

## 20.11 Tactile presets

The system should support named feel profiles while preserving the same doctrine.

Examples:

* **Classic Field**: balanced all-purpose mode
* **Precision Edge**: stronger brake cones and tighter corner handling
* **Texture Guard**: lower texture sensitivity, stronger side-lock hysteresis
* **Painterly Freehand**: weaker attraction, more fluid motion
* **Compositor**: stronger alpha/depth channel weighting where available

These presets should alter dynamics, not identity.

## 20.12 Failure transparency

When the tool is uncertain, it should communicate gracefully.

Possible channels:

* subtle color shift in the head or tail
* confidence ring
* slight reduction in edge-lock visual intensity
* optional diagnostic panel for advanced users

The point is not to burden casual users, but to avoid false certainty.

## 20.13 Interaction acceptance tests

The interaction layer should be judged on whether users can:

* enter and leave contours fluidly
* negotiate corners without fighting momentum
* backtrack without mode-switch frustration
* maintain path ownership in ambiguous regions
* learn the tactile states within a short session
* prefer the tool over legacy magnetic lasso after practice

# 21. Benchmark and Diagnostics Platform

## 21.1 Purpose of the benchmark layer

A project of this ambition cannot rely on vibes, isolated demos, or cherry-picked examples. It needs a research-grade benchmark and diagnostics platform.

The benchmark layer exists to:

* quantify progress
* expose regressions
* compare presets and algorithms
* stress-test corner cases
* support record-and-replay development
* build trust in claims of superiority

## 21.2 Benchmark philosophy

The benchmark suite should evaluate the lasso as an instrument, not merely as an edge detector.

That means measuring both:

* geometric outcomes
* interactional experience proxies

A system that produces decent contours but feels awful is still a failure.

## 21.3 Benchmark corpus design

The corpus should include synthetic and real-world categories.

### Synthetic classes

* isolated high-contrast edges
* anti-aliased silhouette bands
* sharp corners at controlled angles
* T-junctions and X-junctions
* parallel contour families at varying spacing
* broken contours with controlled gaps
* repeated texture patterns
* noisy backgrounds versus clean boundaries
* sub-pixel diagonal edges
* hair-like filament structures

### Real-world classes

* portrait cutouts with hair and soft transitions
* product silhouettes with reflections
* animals with fur and whiskers
* foliage and branches
* clothing folds against similar backgrounds
* transparent or semi-transparent materials
* architectural edges with perspective and clutter
* compositing plates with alpha/depth channels where available

## 21.4 Ground truth philosophy

Ground truth should not be naïvely reduced to one perfect contour line. For many images, acceptable contours fall within a band.

Therefore define:

* ideal contour centerline where clear
* tolerance band where boundary is inherently soft or anti-aliased
* ambiguity zones where multiple paths are defensible

The benchmark should penalize nonsense strongly but allow human-valid variation where appropriate.

## 21.5 Replay harness

A critical component is input replay.

The platform should support deterministic playback of:

* cursor trajectories
* stylus pressure curves
* clutch events
* anchor insertion events
* mode changes
* frame timing profiles

This enables apples-to-apples comparison between algorithm revisions.

## 21.6 Metrics

### Geometric metrics

* mean signed contour deviation
* max contour deviation
* corner overshoot distance
* branch-switch frequency
* contour-family chatter rate
* gap-jump incidence
* anti-aliased band placement consistency

### Temporal metrics

* number of corrections per path length
* average backtrack distance per contour task
* time to completion for fixed replay trajectories
* frequency of lost-state events
* recovery latency after lost-state

### Interaction proxy metrics

* spring-head separation statistics
* confidence regime occupancy
* clutch usage frequency in benchmark tasks
* ambiguity-triggered brake activation frequency
* anchor insertion frequency and location patterns

### Performance metrics

* end-to-end latency
* frame time under representative image sizes
* GPU preprocessing cost
* CPU motion-step cost
* UI-thread interference

## 21.7 Expert evaluation program

Benchmarks should not be purely synthetic. A panel of expert users should compare the tool against legacy magnetic lasso and possibly other path tools.

Structured evaluation topics:

* perceived obedience
* perceived overreach
* trust under ambiguity
* corner handling quality
* backtracking quality
* cleanup burden after selection
* overall preference for real work

## 21.8 Diagnostics architecture

Diagnostics should operate at three levels.

### Level 1: user-safe subtle diagnostics

* confidence tint
* side-lock indication
* clutch indicator
* anchor state indicator

### Level 2: advanced visible overlays

* edge field visualization
* tangent vectors
* scalar terrain slices
* ambiguity heat map
* force contribution bars
* tail mutability visualization

### Level 3: developer research panels

* per-frame state dumps
* coefficient traces
* event timelines
* benchmark replay inspector
* failure classifier output

## 21.9 Failure taxonomy

The benchmark platform should classify failures, not just score them.

Suggested taxonomy:

* overshoot corner failure
* zipper chatter failure
* wrong-side lock failure
* texture trap failure
* premature release failure
* over-assist failure
* under-assist failure
* bad unzip failure
* lost-state re-acquisition failure
* rasterization mismatch failure

A mature tool learns faster from named failures than from one giant average score.

## 21.10 Benchmark acceptance criteria

A candidate release should satisfy:

* no major regression on core synthetic classes
* no latency regression above defined threshold
* measurable improvement over legacy baseline on expert tasks
* reduced correction burden on ambiguous real-world sets
* no doctrine violations introduced by hidden automation

# 24. System Invariants

These invariants are global laws. Every subsystem must preserve them.

## 24.1 Identity invariants

1. **Human authorship invariant**
   The active path must remain attributable to current or recent user motion. No subsystem may autonomously create large contour jumps or branch commitments without direct local user support.

2. **Local evidence invariant**
   Any assistance force, lock, or contour preference must be traceable to local field evidence and short-horizon interaction state. No long-range semantic prior may influence the core path solution.

3. **Ambiguity humility invariant**
   Increased ambiguity must decrease automated authority. No ambiguity source may increase autonomous commitment.

4. **Reversibility invariant**
   Recent path history must remain fluid until committed. Reverse motion must be able to retract recent path state without destructive global undo rituals.

5. **Determinism invariant**
   Given the same image field, input replay, parameter graph, and timing model, the simulation must produce materially identical results within accepted floating-point tolerance.

6. **Latency invariant**
   The real-time step must remain bounded and predictable enough to preserve instrument feel. Diagnostic and UI layers may never block the motion loop.

7. **Mask honesty invariant**
   The produced mask must correspond to authored path geometry and declared post-path operations. No hidden semantic clean-up may silently rewrite the result.

## 24.2 Failure-handling invariants

8. **Lost-state fallback invariant**
   When contour confidence collapses, the tool must degrade toward manual lasso behavior, not long-range snapping or guesswork.

9. **Branch conservatism invariant**
   At ambiguous branch points, uncommitted switching must remain reversible and weak. Strong branch commitment requires explicit user intent or explicit commit semantics.

10. **Diagnostic truth invariant**
    Any visible confidence, ambiguity, side-lock, or field overlay must reflect actual internal state, not UI approximation for vibes.

# 28. State Machines

## 28.1 Session-level state machine

```text
IDLE
  -> ARMED (pointer down, start request)
ARMED
  -> DRAWING (first valid motion sample)
  -> IDLE (cancel)
DRAWING
  -> CLOSING (close request or closure condition)
  -> COMMITTING (selection apply request)
  -> IDLE (cancel/reset)
CLOSING
  -> COMMITTING (path closed)
  -> DRAWING (closure rejected or reopened)
COMMITTING
  -> IDLE (selection applied)
  -> DRAWING (commit aborted, path retained)
```

## 28.2 Draw-mode interaction state machine

```text
FREE_GLIDE
  -> CONTOUR_CATCH (fused boundary rises above threshold and user approaches)
CONTOUR_CATCH
  -> CONTOUR_RIDE (confidence enters MID/HIGH and tangent stable)
  -> FREE_GLIDE (boundary fades)
CONTOUR_RIDE
  -> BRACE (junctionness/cornerness or ambiguity spike)
  -> RELEASE (confidence drops or clutch)
BRACE
  -> CONTOUR_RIDE (stable outgoing contour forms)
  -> RELEASE (manual dominance or ambiguity remains high)
RELEASE
  -> FREE_GLIDE (manual traversal)
  -> CONTOUR_CATCH (new boundary acquisition)
ANY
  -> UNZIP (reverse intent over mutable tail)
UNZIP
  -> FREE_GLIDE (tail cleared or reverse motion ends)
  -> CONTOUR_CATCH (new boundary reacquired during retreat)
ANY
  -> ANCHOR_COMMIT (anchor request)
ANCHOR_COMMIT
  -> prior state (after commit processing)
```

## 28.3 Confidence regime machine

```text
HIGH
  -> MID (confidence falls below high exit threshold)
MID
  -> HIGH (confidence rises above high enter threshold)
MID
  -> LOW (confidence falls below low enter threshold)
LOW
  -> MID (confidence rises above low exit threshold)
```

Use hysteresis thresholds so regime transitions do not chatter.

## 28.4 Side-lock machine

```text
UNLOCKED
  -> PENDING (stable normal-side evidence appears)
PENDING
  -> LOCKED (persistence duration met)
PENDING
  -> UNLOCKED (evidence lost)
LOCKED
  -> WEAKENED (high side ambiguity or clutch)
WEAKENED
  -> LOCKED (evidence restored)
WEAKENED
  -> UNLOCKED (persistence exhausted or explicit cross intent)
```

# 29. Reference Frame Step Blueprint

The following frame step is normative unless profiling proves a different ordering is materially superior.

## 29.1 Frame step pseudocode

```ts
function frameStep(ctx: FrameContext): FrameResult {
  // 1. Normalize and filter input
  const pointer = inputController.consumeLatestPointer(ctx.rawInput);
  const intent = intentFilter.update(pointer, ctx.prevIntent);

  // 2. Sample dynamic runtime coefficients
  const coeffs = parameterGraph.evaluate({
    pointerSpeed: pointer.speed,
    headSpeed: ctx.head.kinematics.speed,
    pressure: pointer.pressure,
    edgeStrength: ctx.head.currentFieldSample?.edgeStrength ?? 0,
    confidence: ctx.head.confidence.value,
    ambiguity: ctx.head.ambiguity,
    cursorDistance: distance(intent.proxyPosition, ctx.head.kinematics.position),
    clutchActive: intent.clutchActive,
    lostState: ctx.head.lostState,
  }, ctx.activePresetId);

  // 3. Advance motion state
  const motion = motionEngine.step({
    dtSec: ctx.dtSec,
    head: ctx.head,
    pointer,
    intent,
    field: ctx.field,
    coeffs,
    settings: ctx.motionSettings,
  });

  // 4. Advance temporal path state
  const temporal = temporalEngine.update({
    state: ctx.temporal,
    head: motion.nextHead,
    intent,
    pointer,
    samplingRules: ctx.samplingRules,
    pendingAnchor: inputController.consumePendingAnchor(),
  });

  // 5. Optional closure detection
  const closure = closureController.evaluate(temporal.state, motion.nextHead, pointer);

  // 6. Emit telemetry
  diagnostics.pushFrame({
    frameIndex: ctx.frameIndex,
    timestampMs: ctx.timestampMs,
    headPosition: motion.nextHead.kinematics.position,
    cursorPosition: pointer.filteredPosition,
    confidence: motion.nextHead.confidence.value,
    regime: motion.nextHead.confidence.regime,
    ambiguity: motion.nextHead.ambiguity,
    sideLockActive: motion.nextHead.sideLock.active,
    lostState: motion.nextHead.lostState,
    forceBreakdown: motion.forces,
    tailLength: temporal.state.tail.points.length,
    committedLength: temporal.state.body.committed.length,
  });

  // 7. Return render-ready state
  return {
    pointer,
    intent,
    head: motion.nextHead,
    temporal: temporal.state,
    motionEvents: motion.events,
    temporalEvents: temporal.events,
    closure,
  };
}
```

## 29.2 Ordering rationale

* coefficient evaluation must see recent state but not future motion
* motion must precede path sampling so geometry reflects latest head
* diagnostics must observe finalized per-frame state, not partial mid-step internals
* closure logic must inspect updated path and head state

# 30. Path Sampling and Commit Rules

## 30.1 Tail insertion rules

A new tail point may be inserted only when at least one condition is true:

* arc-length since last tail point exceeds threshold
* curvature or heading change exceeds threshold
* confidence regime changes
* explicit anchor event occurs
* time since last sample exceeds timeout

This avoids oversampling stable straight motion while preserving fidelity at turns and state transitions.

## 30.2 Tail commit rules

Tail points may be committed into the body when:

* distance from tail root exceeds commit distance
* a hard anchor occurs
* a branch commit occurs
* closure is requested
* mutable tail exceeds maximum arc-length and oldest mutable segment ages out

## 30.3 Unzip detection rules

Unzip should engage only when all are true:

* reverse intent is active
* pointer/head motion projects sufficiently opposite recent tail direction
* cursor/head lies within unzip corridor around mutable tail
* clutch or low-confidence condition does not conflict with reverse signal semantics

## 30.4 Closure rules

Closure may be suggested when:

* head approaches valid start neighborhood
* tangent agreement is not catastrophically inconsistent
* closure does not require long-range leap

Closure must remain explicit or at least easily rejectable. No hidden region completion allowed.

# 34. Acceptance Tests by Phase

## 34.1 Phase 1 acceptance tests: minimal viable field lasso

* head converges toward an isolated straight edge under bounded cursor pull
* head travels along a long edge without excessive oscillation
* path remains reproducible across replayed input
* rasterized mask matches path geometry on simple closed shapes
* latency remains under declared frame budget on target device class

## 34.2 Phase 2 acceptance tests: temporal sophistication

* reverse motion unzips mutable tail without corrupting committed body
* hard anchor preserves committed corner under continued motion
* soft anchor stabilizes recent history without freezing all movement
* closure commits exactly once and produces valid mask parity

# Boundary Instrument — Algorithmic Appendix

## Formal Estimators, Force Logic, State Updates, Raster Rules, and Benchmark Computation

### Purpose

This document is the implementation-facing algorithmic appendix for the Boundary Instrument system. It is written as the next layer after the doctrine, execution stack, and engineering blueprints. Its purpose is to define the core computational logic clearly enough that an implementation team or coding agent can build the system without inventing hidden semantics, smearing together unrelated heuristics, or quietly mutating the lasso into an autonomous segmentation goblin.

This appendix is organized around one rule:

**Every assistive behavior must arise from local evidence, explicit user interaction, and bounded temporal state.**

No semantic inference is assumed anywhere in the algorithms below.

# 1. Notation and Coordinate Conventions

## 1.1 Image coordinates

Let the image domain be `Omega ⊂ R^2`.

We use two coordinate systems:

* **pixel coordinates**: `p = (x_px, y_px)` in image pixel units
* **normalized coordinates**: `u = (x, y)` in `[0,1]^2`

All internal field samplers should support normalized coordinates. Local radii may be specified in pixels and converted through image dimensions.

## 1.2 Basic operators

For a vector `v = (vx, vy)`:

* norm: `|v| = sqrt(vx^2 + vy^2)`
* normalize: `n(v) = v / max(|v|, eps)`
* perpendicular: `perp(v) = (-vy, vx)`
* dot: `dot(a,b)`
* cross scalar in 2D: `cross2(a,b) = ax*by - ay*bx`

For stability, all divisions use an epsilon floor:

`safeDiv(a,b) = a / max(|b|, eps)`

## 1.3 Time discretization

Let frame time be `dt > 0`.

State updates are evaluated on each frame or simulation tick. The implementation should assume variable `dt` within bounded jitter and must remain stable under modest frame-rate variation.

# 2. Field Construction Algorithms

## 2.1 Goal

Construct a fused local contour field from image evidence such that:

* strong, coherent contours form low-energy channels
* local tangent directions are stable and sub-pixel sampleable
* corners and junctions are explicitly represented
* texture ambiguity is represented separately from ordinary edge strength
* all channels remain interpretable and independently inspectable

## 2.2 Working-space conversion

### Recommended conversion steps

1. decode source image into linear or perceptually appropriate working space
2. derive:

   * luminance `L`
   * chroma axes `C1, C2`
   * saturation-like scalar `S`
3. preserve alpha, depth, normals if available as auxiliary layers

### Suggested luminance

If using linear RGB:

`L = 0.2126 R + 0.7152 G + 0.0722 B`

### Suggested chroma

Use either perceptual Lab-like components or a stable opponent transform. Exact choice may vary, but the principle is to separate luminance from chromatic discontinuity.

## 2.3 Derivative estimation

For each enabled scalar channel `X`, compute smoothed gradients:

`Gx_X = dX/dx`
`Gy_X = dX/dy`

### Recommended practical method

* light pre-blur or edge-preserving denoise
* Scharr or high-quality derivative operator
* store gradients at full resolution in float textures if possible

### Channel boundary magnitude

`B_X = sqrt(Gx_X^2 + Gy_X^2)`

## 2.4 Texture discontinuity channel

A boundary instrument needs a way to detect “busy nonsense” separately from “clean contour.”

### Suggested local texture descriptor

For each pixel, compute a local structure descriptor over a small window:

* local gradient histogram
* local variance / entropy
* local orientation spread

Then estimate a texture discontinuity signal:

`B_T(x) = difference( descriptor(left neighborhood), descriptor(right neighborhood) )`

The exact descriptor may vary. The key is that texture discontinuity should capture abrupt change in local microstructure, not merely raw gradient strength.

## 2.5 Fused boundary evidence

Let enabled channels be weighted by nonnegative weights `w_i`.

`B_fused = wL * B_L + wC1 * B_C1 + wC2 * B_C2 + wS * B_S + wT * B_T + wA * B_A + wD * B_D + wN * B_N`

### Normalization rule

Either:

* weights sum to 1, or
* the system stores unnormalized contributions and separately stores total weight

Do not silently renormalize in ways that destroy tuning intent.

## 2.6 Structure tensor and coherence

Build a structure tensor from luminance or fused directional evidence:

`J = [ <Gx^2>   <Gx Gy> ]`
`    [ <Gx Gy>  <Gy^2>  ]`

where `<...>` denotes local smoothing.

Compute eigenvalues `lambda1 >= lambda2 >= 0` and eigenvector `e1` corresponding to `lambda1`.

### Coherence

`K = (lambda1 - lambda2) / max(lambda1 + lambda2, eps)`

Interpretation:

* `K մոտ 1`: strong, coherent oriented structure
* `K մոտ 0`: isotropic, noisy, or corner-like region

## 2.7 Tangent field

If `g = n(grad B_fused)` is the local normal-like direction, then contour tangent candidate is:

`t_raw = perp(g)`

Alternatively, use the minor eigenvector of the structure tensor as the tangent direction.

### Tangent stabilization

Because orientation is sign-ambiguous, the stored tangent field should encode only unsigned orientation or store both signs as equivalent. The actual signed tangent is resolved at runtime from user motion continuity.

## 2.8 Cornerness and junctionness

Corners and junctions must explicitly suppress overconfidence.

### Suggested cornerness proxy

Use either Harris-like response, Shi-Tomasi score, or coherence inversion combined with local boundary intensity.

One usable local proxy:

`cornerness = saturate( (lambda2 / max(lambda1, eps)) * B_fused * kc )`

This rises when the second eigenvalue is nontrivial and boundary evidence is strong.

### Junctionness refinement

A stronger junction measure can additionally inspect multi-modal tangent distribution in a neighborhood.

For neighborhood `N(x)`, gather tangent samples and build an orientation histogram. If multiple strong peaks exist, raise `junctionness`.

## 2.9 Texture ambiguity

Texture ambiguity should be distinct from texture discontinuity.

* **texture discontinuity**: a plausible boundary cue
* **texture ambiguity**: a sign that many similar local micro-edges compete

### Suggested texture ambiguity estimator

Over neighborhood `N(x)`:

* compute variance of boundary magnitude
* count local maxima of `B_fused`
* measure orientation spread
* measure periodicity or repeated edge spacing

One coarse estimator:

`Q_tex = saturate( a1 * peakDensity + a2 * orientationSpread + a3 * repeatScore - a4 * coherence )`

## 2.10 Scalar terrain synthesis

The scalar terrain `Phi(x)` must create low-energy channels along reliable contours and higher-energy regions in ambiguous or unstable areas.

### Suggested synthesis

`Phi = c0 - gb * smooth(B_fused) + gc * smooth(cornerness) + gt * smooth(Q_tex) + gr * regularization`

Where:

* `gb > 0` lowers energy at strong contours
* `gc > 0` raises energy near corners/junctions
* `gt > 0` raises energy in texture-ambiguous zones
* `regularization` keeps the field smooth enough for stable gradients

### Regularization

Possible regularization terms:

* Laplacian smoothing of terrain
* Gaussian blur on the assembled field
* anisotropic smoothing aligned with contours

The terrain should remain smooth enough to yield stable gradients but not so blurred that contour valleys disappear.

## 2.11 Continuous field sampling

At runtime, sample all scalar and vector fields with bilinear or better interpolation.

### Terrain gradient

Either:

* precompute gradient of `Phi`, or
* estimate from central differences at sample time

Preferred for performance: precompute terrain gradient textures if memory budget allows.

# 3. Confidence Composition

## 3.1 Design requirement

Confidence must not be a mystical single number excreted by vibes. It should be a composed score with interpretable factors, each of which can be logged and tuned.

## 3.2 Factor definition

Let:

* `E = normalized edge strength at head`
* `K = coherence at head`
* `A = alignment between user intent and local tangent`
* `R = roughness penalty`
* `C = corner penalty`
* `U = continuity factor`
* `L = lost-state penalty`

### 3.2.1 Edge factor

`f_edge = saturate(remap(E, E_low, E_high))`

### 3.2.2 Coherence factor

`f_coh = saturate(remap(K, K_low, K_high))`

### 3.2.3 Alignment factor

Let `i = n(userForwardIntent)` and `t = signedTangentCandidate`.

`f_align = 0.5 + 0.5 * dot(i, t)`

This maps `[-1,1]` to `[0,1]`.

If intent is weak, blend toward a neutral value instead of overtrusting noise.

### 3.2.4 Roughness penalty

Estimate local path roughness from recent heading changes over the floating tail.

If recent heading sequence is `{h_k}`, define:

`roughness = mean_k (1 - dot(h_k, h_{k-1}))`

Then:

`p_rough = saturate(remap(roughness, R_low, R_high))`

### 3.2.5 Corner penalty

`p_corner = saturate(remap(cornerness, C_low, C_high))`

### 3.2.6 Continuity factor

Continuity measures whether the head is remaining on a plausible continuation of the recent contour family.

Possible components:

* signed tangent stability
* local normal-side consistency
* low lateral drift relative to recent path tangent

Define:

`f_cont = saturate( w1 * tangentConsistency + w2 * sideConsistency + w3 * contourChannelPersistence )`

### 3.2.7 Lost-state penalty

If lost-state is active, confidence must collapse hard.

`p_lost = 1 if lostState else 0`

## 3.3 Composite confidence

A robust form is multiplicative-positive with subtractive penalties:

`c_raw = (f_edge^ae) * (f_coh^ak) * (f_align^aa) * (f_cont^au)`

`c_pen = 1 - saturate( wr * p_rough + wc * p_corner + wl * p_lost )`

`confidence = saturate(c_raw * c_pen)`

This structure avoids high confidence when one critical factor is near zero.

## 3.4 Confidence deadbands

Use hysteretic bands, not a single threshold.

### Example thresholds

* enter HIGH if `confidence >= 0.72`
* exit HIGH if `confidence < 0.63`
* enter LOW if `confidence <= 0.28`
* exit LOW if `confidence > 0.36`
* otherwise MID

These values are illustrative and should be benchmark-tuned.

## 3.5 Confidence pseudocode

```ts
function composeConfidence(input: ConfidenceInputs): HeadConfidence {
  const fEdge = saturate(remap(input.edgeStrength, input.edgeLow, input.edgeHigh));
  const fCoh  = saturate(remap(input.coherence, input.cohLow, input.cohHigh));
  const fAlign = input.intentMagnitude > input.intentEps
    ? 0.5 + 0.5 * dot(normalize(input.intent), normalize(input.signedTangent))
    : 0.5;

  const roughness = computeTailRoughness(input.recentHeadings);
  const pRough = saturate(remap(roughness, input.roughLow, input.roughHigh));
  const pCorner = saturate(remap(input.cornerness, input.cornerLow, input.cornerHigh));
  const fCont = saturate(
    input.wTangentConsistency * input.tangentConsistency +
    input.wSideConsistency * input.sideConsistency +
    input.wChannelPersistence * input.channelPersistence
  );

  const pLost = input.lostState ? 1 : 0;

  const cRaw = Math.pow(fEdge, input.aEdge)
             * Math.pow(fCoh, input.aCoh)
             * Math.pow(fAlign, input.aAlign)
             * Math.pow(fCont, input.aCont);

  const cPen = 1 - saturate(
    input.wRough * pRough +
    input.wCorner * pCorner +
    input.wLost * pLost
  );

  const value = saturate(cRaw * cPen);
  const regime = updateConfidenceRegime(value, input.prevRegime, input.thresholds);

  return {
    value,
    regime,
    edgeFactor: fEdge,
    coherenceFactor: fCoh,
    alignmentFactor: fAlign,
    roughnessPenalty: pRough,
    cornerPenalty: pCorner,
    continuityFactor: fCont,
  };
}
```

# 4. Ambiguity Vector Estimation

## 4.1 Philosophy

Ambiguity is not one thing. A system that collapses all uncertainty into a single scalar becomes clumsy because different ambiguity types require different behavioral responses.

We define:

`a = [a_junction, a_side, a_texture, a_continuity, a_intent]`

Each component lies in `[0,1]`.

## 4.2 Junction ambiguity

This rises when multiple plausible outgoing contour directions exist.

### Suggested estimator

Use local orientation histogram in neighborhood `N(x)`.

* build histogram over contour tangent angles modulo pi
* detect number and strength of significant peaks
* combine with cornerness

One form:

`a_junction = saturate( b1 * cornerness + b2 * multiPeakScore + b3 * (1 - coherence) )`

## 4.3 Side ambiguity

This rises when parallel or nearby contour families compete laterally.

### Suggested estimator

Sample along local normal direction `n = perp(tangent)` at offsets `{-r,...,+r}`.

* find multiple local minima in terrain or multiple high boundary ridges of similar strength
* measure lateral spacing and strength similarity

Example:

`a_side = saturate( s1 * parallelPeakScore + s2 * strengthSimilarity + s3 * familyDensity )`

## 4.4 Texture ambiguity

This rises when the head sits inside repeated microstructure with many equally plausible tiny edges.

`a_texture = Q_tex`

where `Q_tex` came from the field build.

## 4.5 Continuity ambiguity

This measures breakdown of the previously stable contour trajectory.

Factors:

* drop in edge strength versus recent running average
* drop in tangent stability
* increased lateral drift from expected continuation

Example:

`a_cont = saturate( c1 * edgeDrop + c2 * tangentVariance + c3 * lateralDeviation )`

## 4.6 Intent conflict ambiguity

This rises when the user’s current motion disagrees with contour-following predictions.

Let `i = user intent`, `t = signed tangent`, `g = -n(grad Phi)`.

Then define:

`intentTangentConflict = 0.5 - 0.5 * dot(i, t)`
`intentEdgeConflict = max(0, dot(i, -g))` if the user is pulling away from the valley

Then:

`a_intent = saturate( q1 * intentTangentConflict + q2 * intentEdgeConflict )`

## 4.7 Ambiguity pseudocode

```ts
function estimateAmbiguity(input: AmbiguityInputs): AmbiguityVector {
  const aJunction = saturate(
    input.b1 * input.cornerness +
    input.b2 * input.multiPeakScore +
    input.b3 * (1 - input.coherence)
  );

  const aSide = saturate(
    input.s1 * input.parallelPeakScore +
    input.s2 * input.strengthSimilarity +
    input.s3 * input.familyDensity
  );

  const aTexture = saturate(input.textureAmbiguity);

  const aContinuity = saturate(
    input.c1 * input.edgeDrop +
    input.c2 * input.tangentVariance +
    input.c3 * input.lateralDeviation
  );

  const tangentConflict = 0.5 - 0.5 * dot(normalize(input.intent), normalize(input.signedTangent));
  const aIntent = saturate(
    input.q1 * tangentConflict +
    input.q2 * input.edgeEscapeScore
  );

  return {
    junction: aJunction,
    side: aSide,
    texture: aTexture,
    continuity: aContinuity,
    intentConflict: aIntent,
  };
}
```

# 5. Tangent Sign Resolution

## 5.1 Problem

The tangent field is orientation-like, not direction-like. Each tangent has two possible signs. Choosing the wrong sign every few frames turns the lasso into a jittering clown shoe.

## 5.2 Inputs for sign choice

Use:

* user forward intent `i`
* previous stable tangent `t_prev`
* recent head velocity `v`
* local unsigned tangent `t0`

Candidates are:

`t_plus = n(t0)`
`t_minus = -n(t0)`

## 5.3 Score function

Define a directional score for candidate `t`:

`S(t) = wp * dot(t, t_prev) + wi * dot(t, i) + wv * dot(t, n(v))`

Choose the sign with greater score.

If both scores are weak because intent and velocity are tiny, preserve previous sign if available.

## 5.4 Stabilization rule

To prevent flip chatter, require a margin before switching sign.

Only switch if:

`S(new) > S(old) + hysteresisMargin`

## 5.5 Pseudocode

```ts
function resolveSignedTangent(input: TangentSignInput): Vec2 {
  const tPlus = normalize(input.unsignedTangent);
  const tMinus = scale(tPlus, -1);

  const score = (t: Vec2) =>
    input.wPrev * dot(t, input.prevStableTangent ?? t) +
    input.wIntent * dot(t, input.intentDir ?? t) +
    input.wVel * dot(t, input.velocityDir ?? t);

  const sPlus = score(tPlus);
  const sMinus = score(tMinus);

  const current = input.prevStableTangent ?? tPlus;
  const sCurrent = score(current);
  const candidate = sPlus >= sMinus ? tPlus : tMinus;
  const sCandidate = Math.max(sPlus, sMinus);

  if (dot(candidate, current) < 0 && sCandidate < sCurrent + input.hysteresisMargin) {
    return current;
  }

  return candidate;
}
```

# 6. Force Model and Dynamic Coefficients

## 6.1 Total force

The lasso head evolves under:

`F_total = a_e F_edge + a_t F_tangent + a_s F_spring + a_i F_inertia + a_b F_brake + a_h F_hyst + a_u F_user`

Each coefficient `a_*` is dynamic.

## 6.2 Edge force

`F_edge = -grad Phi(x)`

Optionally normalize and scale nonlinearly to avoid absurd acceleration on steep terrain spikes:

`F_edge = n(-grad Phi) * f_mag(|grad Phi|)`

## 6.3 Tangent force

`F_tangent = t_signed * m_t`

Where `m_t` may depend on:

* coherence
* confidence
* alignment
* edge strength

### Suggested magnitude

`m_t = mt0 * f_edge * f_coh * (0.5 + 0.5 f_align)`

## 6.4 Spring force

The system must remain user-led.

`F_spring = k_s * (x_proxy - x_head)`

### Cursor proxy

The proxy may be a lightly filtered pointer position rather than raw input:

`x_proxy(t) = lerp(x_proxy(t-dt), x_pointer, alpha_proxy)`

This reduces twitch without severing user authorship.

## 6.5 Inertial force

The lasso should preserve short-horizon continuity but not bulldoze corners.

One formulation:

`F_inertia = k_i * n(v_prev) * continuityWeight`

Where `continuityWeight` shrinks near corners and ambiguity spikes.

## 6.6 Brake force

Brake force opposes dangerous continuation. It should rise in corners, junctions, and high intent conflict.

`F_brake = - n(v_prev) * m_brake`

with

`m_brake = kb * max(a_junction, p_corner, a_intent)`

## 6.7 Hysteresis force

This force resists lateral switching between contour families.

Let `n_side` be preferred side normal. Compute lateral drift along that normal:

`d_lat = dot(x_head - x_familyRef, n_side)`

Then:

`F_hyst = - kh * d_lat * n_side`

applied only when side-lock is active and competing family risk is high.

## 6.8 User override force

When clutching or in low-confidence regime, user intent should dominate.

`F_user = ku * i`

where `i` is normalized pull direction or raw cursor delta scaled appropriately.

## 6.9 Dynamic coefficient mapping

Each coefficient should be sampled from response curves.

### Example mapping logic

* `a_e` high when edge and confidence are high, lower when clutch is active
* `a_t` high when coherence is high and junction ambiguity is low
* `a_s` rises when cursor-head distance grows or confidence drops
* `a_i` high during stable ride, low near corners
* `a_b` high when `a_junction` or `a_intent` is high
* `a_h` high when side ambiguity is high and side-lock exists
* `a_u` high in clutch or low-confidence regimes

## 6.10 Coefficient pseudocode

```ts
function resolveRuntimeCoefficients(ctx: ParameterContext, curves: Curves): RuntimeCoefficients {
  const edgeGain = curves.edgeGain(ctx.edgeStrength, ctx.confidence, ctx.pressure, ctx.clutchActive);
  const tangentGain = curves.tangentGain(ctx.confidence, ctx.ambiguity.junction, ctx.pointerSpeed);
  const springGain = curves.springGain(ctx.cursorDistance, ctx.confidence, ctx.clutchActive);
  const inertiaGain = curves.inertiaGain(ctx.confidence, ctx.ambiguity.continuity, ctx.ambiguity.junction);
  const brakeGain = curves.brakeGain(ctx.ambiguity.junction, ctx.ambiguity.intentConflict);
  const hysteresisGain = curves.hysteresisGain(ctx.ambiguity.side, ctx.confidence);
  const userOverrideGain = curves.userOverrideGain(ctx.clutchActive, ctx.confidence, ctx.pressure);

  return {
    edgeGain,
    tangentGain,
    springGain,
    inertiaGain,
    brakeGain,
    hysteresisGain,
    userOverrideGain,
    maxSpeed: curves.maxSpeed(ctx.pointerSpeed, ctx.confidence),
    maxStepDistance: curves.maxStepDistance(ctx.pointerSpeed),
    reacquireRadius: curves.reacquireRadius(ctx.confidence, ctx.lostState),
    unzipReverseThreshold: curves.unzipReverseThreshold(ctx.pointerSpeed),
  };
}
```

# 7. Side-Lock and Contour-Family Detection

## 7.1 Goal

When the image contains parallel or closely adjacent contour families, the lasso should not chatter laterally between them every time the field wiggles. The system therefore maintains a weak side-consistency memory.

## 7.2 Family candidate extraction

At the current head position:

1. compute local signed tangent `t`
2. compute local normal `n = perp(t)`
3. sample field along `n` over offsets `d in [-r, r]`
4. identify local contour channels as minima of `Phi(d)` or maxima of `B_fused(d)`

Each channel candidate gets:

* offset `d_i`
* strength `s_i`
* tangent agreement `g_i`

## 7.3 Preferred side inference

When entering a contour ride, estimate which side the user intends to remain on.

Possible cues:

* initial approach direction relative to contour normal
* whether cursor is outside or inside the current valley channel
* recent signed offset from dominant channel centerline

If sign of preferred side is stable for `N` frames, set side-lock active.

## 7.4 Family switching cost

If current family `f_cur` is active and another family `f_alt` appears nearby, require a switching condition:

`switchAllowed if score(f_alt) > score(f_cur) + delta_switch`

### Family score

Possible score:

`score(f) = m1 * strength + m2 * tangentAgreement + m3 * userAlignment - m4 * lateralJumpCost`

where `lateralJumpCost` grows with |offset difference| and with side-lock persistence.

## 7.5 Side-lock persistence decay

Side-lock should not become a prison.

Let persistence `P` decay when:

* clutch active
* sustained intent conflict
* family evidence weakens
* branch commit is absent and ambiguity persists

## 7.6 Pseudocode

```ts
function updateSideLock(input: SideLockInput): SideLockState {
  const families = detectContourFamilies(input.fieldNeighborhood, input.signedTangent, input.radiusPx);
  if (families.length === 0) return { ...input.prev, active: false, familyId: null, strength: 0, persistence: 0 };

  const best = rankFamilies(families, input.userIntent, input.prev);

  if (!input.prev.active) {
    const prefNormal = inferPreferredNormal(best, input.approachHistory);
    const persistence = input.stableSideEvidence ? input.prev.persistence + 1 : 0;
    return {
      active: persistence >= input.acquireFrames,
      preferredNormal: prefNormal,
      familyId: best.id,
      strength: best.score,
      persistence,
    };
  }

  const currentFamily = families.find(f => f.id === input.prev.familyId) ?? null;
  const currentScore = currentFamily ? currentFamily.score + input.switchBias : -Infinity;
  const bestScore = best.score;

  if (best.id !== input.prev.familyId && bestScore <= currentScore + input.switchMargin) {
    return decaySideLock(input.prev, input.decayRate, input.clutchActive, input.intentConflict);
  }

  return {
    active: true,
    preferredNormal: input.prev.preferredNormal ?? inferPreferredNormal(best, input.approachHistory),
    familyId: best.id,
    strength: bestScore,
    persistence: Math.min(input.prev.persistence + 1, input.maxPersistence),
  };
}
```

# 8. Junction Brake Cones

## 8.1 Goal

At sharp corners and branch-heavy regions, continued momentum is often wrong. The system must actively damp overshoot.

## 8.2 Cone definition

At position `x`, if junctionness exceeds threshold, define one or more unstable outgoing directions from the neighborhood orientation histogram.

For each unstable branch direction `b_i`, define an angular cone:

`Cone_i = { v : angle(v, b_i) <= theta_i }`

The current velocity or intended continuation vector is tested against these cones.

## 8.3 Brake magnitude

If current continuation direction lies inside an active cone:

`brakeConeFactor = smoothstep(theta_i, 0, angle(v, b_i))`

Then raise brake gain and lower tangent/inertia gains.

## 8.4 Simplified implementation

A cheaper implementation uses only cornerness and heading-change prediction:

* estimate upcoming turn severity from local tangent variation ahead
* if severity high, raise brake and reduce inertial continuation

This is less explicit than full cones but still valid if benchmarked.

## 8.5 Pseudocode

```ts
function evaluateBrakeCone(input: BrakeConeInput): BrakeConeOutput {
  if (input.junctionness < input.junctionThreshold) {
    return { active: false, factor: 0 };
  }

  let bestFactor = 0;
  for (const dir of input.branchDirections) {
    const ang = angleBetween(input.continuationDir, dir);
    if (ang <= input.maxConeAngle) {
      const factor = smoothstep(input.maxConeAngle, 0, ang);
      bestFactor = Math.max(bestFactor, factor);
    }
  }

  return {
    active: bestFactor > 0,
    factor: bestFactor,
  };
}
```

# 9. Lost-State and Re-Acquisition Logic

## 9.1 Lost-state entry

The lasso is “lost” when the local field no longer supports coherent contour guidance.

### Suggested entry conditions

Enter lost-state if for `M` consecutive frames:

* `confidence` in LOW regime
* `edgeStrength < E_lost`
* `coherence < K_lost`
* no stable family candidate exists

This avoids toggling on transient noise.

## 9.2 Lost-state behavior

When lost-state is active:

* clamp tangent gain toward zero
* reduce edge attraction strongly
* increase spring and user override dominance
* optionally allow small-radius re-acquisition attempts
* never snap long distances

The tool should become nearly manual, not clever.

## 9.3 Re-acquisition

Attempt re-acquisition only within radius `r_reacq` around cursor/head corridor.

A candidate contour is valid if:

* edge strength high enough
* coherence adequate
* score exceeds current empty-state baseline by margin
* candidate is not grossly inconsistent with user motion

## 9.4 Pseudocode

```ts
function updateLostState(input: LostStateInput): LostStateResult {
  const weakNow = input.confidenceRegime === 'LOW'
    && input.edgeStrength < input.edgeLostThreshold
    && input.coherence < input.cohLostThreshold
    && !input.hasStableFamily;

  let lostFrames = weakNow ? input.prevLostFrames + 1 : 0;
  let lostState = input.prevLostState;

  if (!lostState && lostFrames >= input.enterFrames) lostState = true;

  if (lostState) {
    const candidate = findReacquireCandidate(input.neighborhoodSamples, input.userIntent, input.reacquireRadius);
    if (candidate && candidate.score >= input.reacquireScoreThreshold) {
      lostState = false;
      lostFrames = 0;
      return { lostState, lostFrames, reacquired: true, candidate };
    }
  }

  return { lostState, lostFrames, reacquired: false, candidate: null };
}
```

# 10. Motion Integration

## 10.1 Recommended integrator

A bounded semi-implicit Euler step is usually adequate and simple.

### Update

`v_next = v + dt * F_total`
`v_next = clampNorm(v_next, maxSpeed)`

`x_next = x + dt * v_next`

If `|x_next - x| > maxStepDistance`, clamp displacement.

## 10.2 Stability rules

Always enforce:

* max speed cap
* max displacement cap
* finite acceleration cap if needed
* tangent sign hysteresis
* corner braking before velocity explode-o-rama

## 10.3 Optional projection

If the head is extremely close to a stable valley centerline and confidence is high, a slight projection toward the valley floor may improve ride smoothness. This must remain local and gentle.

## 10.4 Pseudocode

```ts
function integrateHead(head: HeadState, force: Vec2, coeffs: RuntimeCoefficients, dt: number): HeadKinematics {
  const accel = force;
  let velocity = add(head.kinematics.velocity, scale(accel, dt));
  velocity = clampNorm(velocity, coeffs.maxSpeed);

  let displacement = scale(velocity, dt);
  displacement = clampNorm(displacement, coeffs.maxStepDistance);

  const position = add(head.kinematics.position, displacement);
  const heading = length(velocity) > 1e-6 ? normalize(velocity) : head.kinematics.heading;

  return {
    position,
    velocity,
    acceleration: accel,
    heading,
    speed: length(velocity),
  };
}
```

# 11. Floating Tail, Unzip Corridor, and Commit Logic

## 11.1 Tail philosophy

The floating tail is the mutable recent history of the path. It exists so the user can revise by motion, not by bureaucratic ritual.

## 11.2 Tail insertion

Insert a new tail point when one of the following is true:

* distance from last tail point exceeds `d_sample`
* heading change exceeds `theta_sample`
* regime change occurred
* anchor event occurred
* time since last point exceeds `t_sample`

## 11.3 Unzip corridor

To reverse through recent path safely, define a corridor around the tail polyline.

### Corridor test

For cursor/head position `x`, compute minimum distance to tail segments:

`distTail(x) = min_j distPointToSegment(x, seg_j)`

Unzip candidate if:

* `distTail(x) <= r_unzip`
* reverse intent active
* projection onto recent tail ordering indicates backward traversal

## 11.4 Reverse intent test

Let `d_ptr = pointer delta` and `t_tail` be direction of most recent tail segment.

Reverse intent if:

`dot(n(d_ptr), t_tail) < -tau_reverse`

where `tau_reverse` is positive.

## 11.5 Unzip operation

When active:

* suppress tangent gain
* increase spring/user dominance
* pop or mark-retract tail points until cursor/head reaches appropriate rollback point
* do not damage committed body unless explicit rules permit stepping across a commit barrier

## 11.6 Commit barrier

The boundary between tail and committed body is sacred unless:

* explicit user action allows crossing
* hard anchor semantics permit rollback
* project mode intentionally supports body retreat with confirmation

## 11.7 Pseudocode

```ts
function updateFloatingTail(state: TemporalState, input: TailUpdateInput): TemporalUpdateResult {
  let tail = cloneTail(state.tail);
  let body = cloneBody(state.body);
  const events: TemporalEvent[] = [];

  if (shouldInsertTailPoint(input, tail)) {
    const point = makeTailPoint(input.headPosition, input.timestampMs, input.confidence, input.cornerness, input.source);
    tail.points.push(point);
    events.push({ type: 'TAIL_POINT_ADDED', pointId: point.id });
  }

  if (isUnzipActive(input, tail)) {
    let removed = 0;
    while (tail.points.length > 0 && shouldRetractLastPoint(input, tail.points[tail.points.length - 1])) {
      tail.points.pop();
      removed++;
    }
    if (removed > 0) events.push({ type: 'TAIL_UNZIPPED', removedCount: removed });
  }

  const commitCount = computeTailCommitCount(tail, input.commitRules);
  if (commitCount > 0) {
    const commitSlice = tail.points.splice(0, commitCount).map(p => ({ ...p, mutable: false }));
    body.committed.push(...commitSlice);
    events.push({ type: 'TAIL_COMMITTED', committedCount: commitCount });
  }

  return { state: { ...state, body, tail }, events };
}
```

# 12. Anchor Semantics Algorithms

## 12.1 Soft anchor

A soft anchor lightly stabilizes local history.

Algorithmic effects may include:

* immediate tail insertion at anchor point
* increased local commit weight for nearby tail points
* temporary reduction of unzip penetration past the anchor

## 12.2 Hard anchor

A hard anchor creates a commit barrier.

Algorithmic effects:

* insert point into committed body
* mark rollback barrier
* strengthen branch persistence if near family ambiguity

## 12.3 Corner pin

A corner pin preserves a user-authored corner.

Algorithmic effects:

* commit point
* reduce subsequent smoothing over a small arc neighborhood
* preserve high local curvature in postprocessing

## 12.4 Branch commit

A branch commit says: this local route is deliberate.

Algorithmic effects:

* commit point
* set side-lock family strongly
* raise switch margin for nearby family swaps for a limited future arc-length

# 13. Exact Path Rasterization and Parity Logic

## 13.1 Goal

The mask must reflect path geometry exactly and deterministically.

## 13.2 Polyline closure

Given committed body plus optional tail at closure time, construct closed polygonal chain.

Rules:

* ensure last point connects to first point exactly once
* collapse duplicate consecutive vertices within epsilon
* preserve anchor-marked corners

## 13.3 Fill rule

Use a declared fill rule consistently:

* even-odd parity, or
* non-zero winding

For selection tools, even-odd parity is often acceptable and easy to reason about. The choice must remain explicit and consistent across platforms.

## 13.4 Scanline parity-safe rasterization

For each scanline:

1. compute intersections with non-horizontal edges
2. apply consistent inclusion rule for vertices to avoid double counting
3. sort intersections by x
4. fill alternating spans for even-odd parity

### Vertex handling rule

Use the classic half-open edge convention:

* include edges where `y_min <= y < y_max`
* exclude upper endpoint from intersection count

This avoids double-counting shared vertices.

## 13.5 Anti-aliased boundary coverage

For better mask fidelity, compute sub-pixel coverage at edges.

Options:

* supersampled coverage
* analytical segment coverage for edge pixels
* distance-based AA approximation

The implementation may choose based on performance budget, but it must be deterministic.

## 13.6 Pseudocode

```ts
function rasterizeClosedPath(path: Vec2[], width: number, height: number): FloatMask {
  const mask = createFloatMask(width, height, 0);

  for (let y = 0; y < height; y++) {
    const scanY = y + 0.5;
    const xs: number[] = [];

    forEachEdge(path, (a, b) => {
      if (a.y === b.y) return;
      const yMin = Math.min(a.y, b.y);
      const yMax = Math.max(a.y, b.y);
      if (!(scanY >= yMin && scanY < yMax)) return;
      const t = (scanY - a.y) / (b.y - a.y);
      const x = a.x + t * (b.x - a.x);
      xs.push(x);
    });

    xs.sort((m, n) => m - n);
    for (let i = 0; i + 1 < xs.length; i += 2) {
      fillSpanAA(mask, y, xs[i], xs[i + 1]);
    }
  }

  return mask;
}
```

## 13.7 Boolean composition

Given base mask `M_base` and incoming mask `M_new`:

* Replace: `M = M_new`
* Union: `M = max(M_base, M_new)` for coverage masks or boolean OR for binary
* Intersect: `M = min(M_base, M_new)` or boolean AND
* Subtract: `M = clamp(M_base - M_new, 0, 1)` for coverage-like masks
* XOR: symmetric difference rule

If using soft coverage masks, document exact semantics to avoid later confusion.

# 14. Morphology and Feather Algorithms

## 14.1 Expand/contract

For binary masks, morphology may use dilation/erosion.

For soft coverage masks, one may approximate expand/contract via signed distance transform and threshold shift.

### Preferred higher-quality route

1. build signed distance field from mask
2. shift threshold by `deltaPx`
3. reconstruct coverage

This is more stable than repeated kernel stomping.

## 14.2 Feather

Feather radius `r_f` should produce smooth boundary attenuation.

If using signed distance `D(x)` where negative is inside:

`alpha(x) = saturate( 0.5 - D(x) / (2 r_f) )`

or use a smoother kernel as desired.

# 15. Benchmark Metric Algorithms

## 15.1 Mean contour deviation

Given produced contour `P(s)` and ground truth centerline `G`, compute nearest distance from sampled points on `P` to `G` or tolerance band.

`meanDev = mean_i dist(P_i, G)`

If a tolerance band exists, distances inside the band may count as zero.

## 15.2 Max contour deviation

`maxDev = max_i dist(P_i, G)`

## 15.3 Corner overshoot distance

For benchmark scenes with known corner apex `x_corner`, measure maximum forward excursion past the intended turning region before the path settles onto the correct outgoing edge.

This directly measures the embarrassing slingshot problem.

## 15.4 Branch-switch frequency

Count number of family-ID changes in regions labeled as parallel-family tests or branch-stable zones.

`switchRate = switches / pathLength`

## 15.5 Zipper chatter rate

Count repeated lateral oscillations between contour families over short arc windows.

A simple detector:

* family sequence alternates A-B-A-B within short distance
* count as zipper chatter episode

## 15.6 Correction burden

In replay or user sessions, compute:

* number of unzip events
* total rollback distance
* anchor count
* clutch usage count

These are not purely “bad.” But relative reductions under equal accuracy can indicate superior control.

## 15.7 Latency metric

Measure end-to-end latency from pointer sample timestamp to rendered head/path update on screen. Also measure simulation-only latency.

Both matter. A mathematically fancy tool that moves like a sleepy walrus is still a failure.

# 16. Deterministic Replay Specification

## 16.1 Replay bundle contents

A replay must capture enough data to reproduce a session materially.

### Required contents

* source image asset hash
* field build configuration
* parameter preset ID or serialized preset
* timestamped pointer samples
* clutch events
* anchor events
* closure / commit events
* target frame timing profile or normalized dt sequence
* implementation version hash

## 16.2 Replay rule

The replay engine must feed the exact same event sequence into the same state machine ordering. Any nondeterministic sources must be disabled or controlled.

## 16.3 Tolerance

Replayed geometry should remain within a small tolerance of original authored path. Exact bitwise equality is not always realistic across devices, but material equivalence should be enforced.

# 17. Reference End-to-End Step Pseudocode

```ts
function stepBoundaryInstrument(ctx: RuntimeContext): RuntimeContext {
  const pointer = readPointer(ctx.input);
  const intent = updateIntentProxy(pointer, ctx.intentState, ctx.intentConfig);

  const sample = ctx.fieldEngine.sample(ctx.fieldSet, ctx.head.kinematics.position);
  const neighborhood = ctx.fieldEngine.sampleNeighborhood(
    ctx.fieldSet,
    ctx.head.kinematics.position,
    ctx.neighborhoodRadiusPx,
    ctx.neighborhoodSampleCount
  );

  const signedTangent = resolveSignedTangent({
    unsignedTangent: sample.tangent,
    prevStableTangent: ctx.head.lastStableTangent,
    intentDir: normalizeOrNull(intent.forwardIntent),
    velocityDir: normalizeOrNull(ctx.head.kinematics.velocity),
    wPrev: ctx.tangentSignWeights.prev,
    wIntent: ctx.tangentSignWeights.intent,
    wVel: ctx.tangentSignWeights.vel,
    hysteresisMargin: ctx.tangentSignWeights.margin,
  });

  const sideLock = updateSideLock(buildSideLockInput(ctx, neighborhood, signedTangent, intent));
  const ambiguity = estimateAmbiguity(buildAmbiguityInputs(ctx, sample, neighborhood, signedTangent, sideLock, intent));
  const lost = updateLostState(buildLostStateInput(ctx, sample, neighborhood, sideLock));

  const confidence = composeConfidence(buildConfidenceInputs(ctx, sample, signedTangent, sideLock, ambiguity, lost));
  const coeffs = resolveRuntimeCoefficients(buildParameterContext(ctx, pointer, confidence, ambiguity, lost));

  const brakeCone = evaluateBrakeCone(buildBrakeConeInput(ctx, neighborhood, signedTangent));

  const forces = evaluateForces({
    head: ctx.head,
    pointer,
    intent,
    sample,
    signedTangent,
    sideLock,
    confidence,
    ambiguity,
    brakeCone,
    coeffs,
  });

  const nextKinematics = integrateHead(ctx.head, forces.total, coeffs, ctx.dt);

  const nextHead: HeadState = {
    ...ctx.head,
    kinematics: nextKinematics,
    confidence,
    ambiguity,
    sideLock,
    currentFieldSample: sample,
    previousFieldSample: ctx.head.currentFieldSample,
    lastStableTangent: signedTangent,
    lostState: lost.lostState,
    lostStateFrames: lost.lostFrames,
  };

  const temporalUpdate = updateFloatingTail(ctx.temporalState, {
    headPosition: nextHead.kinematics.position,
    timestampMs: pointer.timestampMs,
    confidence: confidence.value,
    cornerness: sample.cornerness,
    source: 'AUTO_SAMPLE',
    pointer,
    intent,
    commitRules: ctx.commitRules,
    unzipRules: ctx.unzipRules,
  });

  ctx.diagnostics.pushFrame(makeFrameTelemetry(ctx, pointer, nextHead, forces, temporalUpdate.state));

  return {
    ...ctx,
    pointerState: pointer,
    intentState: intent,
    head: nextHead,
    temporalState: temporalUpdate.state,
  };
}
```

# 18. Tuning Doctrine for Engineers

## 18.1 Tune by benchmark family, not by ego

Do not tune the system by staring at one pretty demo and congratulating yourselves like Victorian phrenologists with a new caliper. Tune by benchmark families.

* corners tune brake logic
* zipper tests tune side-lock and switching margin
* texture traps tune ambiguity and field fusion
* low-contrast scenes tune lost-state and manual dominance
* stylus sessions tune clutch grammar and spring balance

## 18.2 Tune factors before tuning products

If confidence feels wrong, do not immediately crank a giant global multiplier like a barbarian. Inspect:

* edge factor
* coherence factor
* alignment factor
* roughness penalty
* corner penalty
* continuity factor

Likewise for ambiguity. Tune the decomposition before you tune the final blend.

## 18.3 Preserve doctrine under tuning

A tuned system that becomes more autonomous at ambiguity is a regression even if the demo looks slick. The whole point is lawful assistance.

The codebase must preserve the following truths:

1. The Boundary Instrument is a **human-led contour authoring system**.

## 1.3 Coding philosophy

Prefer:

* small pure functions for estimators
* explicit typed state objects
* deterministic replayable state transitions
* benchmark-driven iteration
* per-factor telemetry for composite heuristics
* file ownership clarity over clever abstraction soup

Avoid:

* giant monolithic controllers

# 18. Coding Agent Rules

## 18.1 Mandatory instructions for AI coding agents

When generating or editing code for this project:

1. Keep subsystem boundaries intact.
2. Do not replace explicit logic with semantic shortcuts.
3. Do not remove diagnostics hooks for convenience.
4. Keep pure computation separate from UI and rendering glue.
5. Prefer small named helpers over giant opaque pipeline blobs.
6. Preserve deterministic replay behavior.
7. Comment invariants and behavioral contracts, not obvious syntax.

## 18.2 Forbidden agent behaviors

* adding “smart object snapping” without explicit doctrine approval
* silently importing ML or segmentation libraries into core path logic
* replacing typed state machines with ad hoc booleans scattered across components
* deleting benchmark failures instead of fixing them
* rewriting a subsystem boundary because “it felt simpler” without updating contracts and tests

## 18.3 Agent output expectations

For each task, a coding agent should provide:

* files created/edited
* summary of logic added
* invariants preserved
* tests added
* known gaps
* benchmark impact if measurable

```ts
}

export function sub(a: Vec2, b: Vec2): Vec2 {
  return { x: a.x - b.x, y: a.y - b.y };
}

export function scale(v: Vec2, s: number): Vec2 {
  return { x: v.x * s, y: v.y * s };
}

export function dot(a: Vec2, b: Vec2): number {
  return a.x * b.x + a.y * b.y;
}

export function cross2(a: Vec2, b: Vec2): number {
  return a.x * b.y - a.y * b.x;
}

export function length(v: Vec2): number {
  return Math.hypot(v.x, v.y);
}

export function safeDiv(a: number, b: number): number {
  return a / Math.max(Math.abs(b), EPS);
}

export function normalize(v: Vec2): Vec2 {
  const len = length(v);
  if (len < EPS) return { x: 0, y: 0 };
  return scale(v, 1 / len);
}

export function normalizeOrNull(v: Vec2): Vec2 | null {
  const len = length(v);
  if (len < EPS) return null;
  return scale(v, 1 / len);
}

export function clamp(value: number, min: number, max: number): number {
  return Math.max(min, Math.min(max, value));
}

export function saturate(value: number): number {
  return clamp(value, 0, 1);
}

export function remap(value: number, inMin: number, inMax: number): number {
  return safeDiv(value - inMin, inMax - inMin);
}

export function smoothstep(edge0: number, edge1: number, x: number): number {
  const t = saturate((x - edge0) / (edge1 - edge0 || EPS));
  return t * t * (3 - 2 * t);
}

export function perp(v: Vec2): Vec2 {
  return { x: -v.y, y: v.x };
}

export function distance(a: Vec2, b: Vec2): number {
  return length(sub(a, b));
}

export function clampNorm(v: Vec2, maxLen: number): Vec2 {
  const len = length(v);
  if (len <= maxLen) return v;
  return scale(v, maxLen / Math.max(len, EPS));
}

export function angleBetween(a: Vec2, b: Vec2): number {
  const na = normalize(a);
  const nb = normalize(b);
  return Math.acos(clamp(dot(na, nb), -1, 1));
}

export function lerp(a: number, b: number, t: number): number {
  return a + (b - a) * t;
}

export function lerpVec2(a: Vec2, b: Vec2, t: number): Vec2 {
  return { x: lerp(a.x, b.x, t), y: lerp(a.y, b.y, t) };
}
```

```ts
export function distPointToSegment(p: Vec2, a: Vec2, b: Vec2): number {
  const ab = sub(b, a);
  const ap = sub(p, a);
  const t = clamp(dot(ap, ab) / Math.max(dot(ab, ab), 1e-6), 0, 1);
  const q = add(a, scale(ab, t));
  return distance(p, q);
}

export function polylineLength(points: Vec2[]): number {
  let total = 0;
  for (let i = 1; i < points.length; i++) {
    total += distance(points[i - 1], points[i]);
  }
  return total;
```

## `src/core/motion/tangentSign.ts`

```ts
import type { Vec2 } from '../common/types';
import { dot, normalize, scale } from '../common/math';

export type TangentSignInput = {
  unsignedTangent: Vec2;
  prevStableTangent: Vec2 | null;
  intentDir: Vec2 | null;
  velocityDir: Vec2 | null;
  wPrev: number;
  wIntent: number;
  wVel: number;
  hysteresisMargin: number;
};

export function resolveSignedTangent(input: TangentSignInput): Vec2 {
  const tPlus = normalize(input.unsignedTangent);
  const tMinus = scale(tPlus, -1);

  const score = (t: Vec2): number =>
    input.wPrev * dot(t, input.prevStableTangent ?? t) +
    input.wIntent * dot(t, input.intentDir ?? t) +
    input.wVel * dot(t, input.velocityDir ?? t);

  const sPlus = score(tPlus);
  const sMinus = score(tMinus);

  const current = input.prevStableTangent ?? tPlus;
  const sCurrent = score(current);
  const candidate = sPlus >= sMinus ? tPlus : tMinus;
  const sCandidate = Math.max(sPlus, sMinus);

  if (dot(candidate, current) < 0 && sCandidate < sCurrent + input.hysteresisMargin) {
    return current;
  }

  return candidate;
}
```

## `src/core/motion/forceModel.ts`

```ts
import type { ForceBreakdown, HeadState } from './motionTypes';
import type { RuntimeCoefficients } from '../params/paramTypes';
import type { Vec2 } from '../common/types';
import { add, scale, sub, vec2 } from '../common/math';

export type ForceModelInput = {
  head: HeadState;
  pointerProxy: Vec2;
  signedTangent: Vec2;
  coeffs: RuntimeCoefficients;
};

export function evaluateForces(input: ForceModelInput): ForceBreakdown {
  const edge = scale(input.head.currentFieldSample?.terrainGrad ?? vec2(0, 0), -input.coeffs.edgeGain);
  const tangent = scale(input.signedTangent, input.coeffs.tangentGain);
  const spring = scale(sub(input.pointerProxy, input.head.kinematics.position), input.coeffs.springGain);
  const inertia = scale(input.head.kinematics.heading, input.coeffs.inertiaGain);
  const brake = scale(input.head.kinematics.heading, -input.coeffs.brakeGain);
  const hysteresis = vec2(0, 0);
  const userOverride = vec2(0, 0);

  const total = add(add(add(edge, tangent), add(spring, inertia)), add(add(brake, hysteresis), userOverride));

  return { edge, tangent, spring, inertia, brake, hysteresis, userOverride, total };
}
```

## `src/core/motion/integrator.ts`

```ts
import type { HeadState, HeadKinematics } from './motionTypes';
import type { RuntimeCoefficients } from '../params/paramTypes';
import type { Vec2 } from '../common/types';
import { add, clampNorm, length, normalize, scale } from '../common/math';

export function integrateHead(
  head: HeadState,
  force: Vec2,
  coeffs: RuntimeCoefficients,
  dt: number,
): HeadKinematics {
  const accel = force;
  let velocity = add(head.kinematics.velocity, scale(accel, dt));
  velocity = clampNorm(velocity, coeffs.maxSpeed);

  let displacement = scale(velocity, dt);
  displacement = clampNorm(displacement, coeffs.maxStepDistance);

  const position = add(head.kinematics.position, displacement);
  const heading = length(velocity) > 1e-6 ? normalize(velocity) : head.kinematics.heading;

  return {
    position,
    velocity,
    acceleration: accel,
    heading,
    speed: length(velocity),
  };
}
```

## `src/core/motion/motionEngine.ts`

```ts
import { BoundaryFieldEngine } from '../field/fieldEngine';
import { resolveSignedTangent } from './tangentSign';
import { evaluateForces } from './forceModel';
import { integrateHead } from './integrator';
import type { MotionStepRequest, MotionStepResult } from './motionTypes';
import { normalizeOrNull } from '../common/math';

const fieldEngine = new BoundaryFieldEngine();

export class MotionEngine {
  step(request: MotionStepRequest): MotionStepResult {
    const sample = fieldEngine.sample(request.field, request.head.kinematics.position);

    const signedTangent = resolveSignedTangent({
      unsignedTangent: sample.tangent,
      prevStableTangent: request.head.lastStableTangent,
      intentDir: normalizeOrNull(request.intent.forwardIntent),
      velocityDir: normalizeOrNull(request.head.kinematics.velocity),
      wPrev: 0.7,
      wIntent: 0.2,
      wVel: 0.1,
      hysteresisMargin: 0.05,
    });

    const forces = evaluateForces({
      head: { ...request.head, currentFieldSample: sample },
      pointerProxy: request.intent.proxyPosition,
      signedTangent,
      coeffs: request.coeffs,
    });

    const nextKinematics = integrateHead(
      { ...request.head, currentFieldSample: sample },
      forces.total,
      request.coeffs,
      request.dtSec,
    );

    const nextHead = {
      ...request.head,
      kinematics: nextKinematics,
      currentFieldSample: sample,
      previousFieldSample: request.head.currentFieldSample,
      lastStableTangent: signedTangent,
    };

    return {
      nextHead,
      forces,
      events: [],
    };
  }
}
```

## `src/core/mask/rasterize.ts`

```ts
import type { Vec2 } from '../common/types';
import type { RasterMaskHandle } from './maskTypes';

export function rasterizeClosedPath(path: Vec2[], width: number, height: number): RasterMaskHandle {
  const data = new Float32Array(width * height);

  for (let y = 0; y < height; y++) {
    const scanY = y + 0.5;
    const xs: number[] = [];

    for (let i = 0; i < path.length; i++) {
      const a = path[i];
      const b = path[(i + 1) % path.length];
      if (a.y === b.y) continue;
      const yMin = Math.min(a.y, b.y);
      const yMax = Math.max(a.y, b.y);
      if (!(scanY >= yMin && scanY < yMax)) continue;
      const t = (scanY - a.y) / (b.y - a.y);
      xs.push(a.x + t * (b.x - a.x));
    }

    xs.sort((m, n) => m - n);
    for (let i = 0; i + 1 < xs.length; i += 2) {
      const start = Math.max(0, Math.floor(xs[i]));
      const end = Math.min(width - 1, Math.ceil(xs[i + 1]));
      for (let x = start; x <= end; x++) {
        data[y * width + x] = 1;
      }
    }
  }

  return {
    id: `mask-${Date.now()}`,
    width,
    height,
    coverageEncoding: 'FLOAT',
    data,
  };
}
```

## `src/core/mask/maskEngine.ts`

```ts
import type { Vec2 } from '../common/types';
import type { RasterMaskHandle, SelectionOp } from './maskTypes';
import { rasterizeClosedPath } from './rasterize';

export class MaskEngine {
  rasterizePath(path: Vec2[], width: number, height: number): RasterMaskHandle {
    return rasterizeClosedPath(path, width, height);
  }

  applySelectionOp(base: RasterMaskHandle | null, incoming: RasterMaskHandle, op: SelectionOp): RasterMaskHandle {
    if (!base || op === 'REPLACE') return incoming;

    const data = new Float32Array(base.data.length);
    for (let i = 0; i < data.length; i++) {
      const a = base.data[i] ?? 0;
      const b = incoming.data[i] ?? 0;
      switch (op) {
        case 'UNION': data[i] = Math.max(a, b); break;
        case 'INTERSECT': data[i] = Math.min(a, b); break;
        case 'SUBTRACT': data[i] = Math.max(0, a - b); break;
        case 'XOR': data[i] = (a > 0) !== (b > 0) ? 1 : 0; break;
        default: data[i] = b;
      }
    }

    return { ...incoming, id: `mask-${Date.now()}`, data };
  }
}
```

## `src/input/pointerNormalizer.ts`

```ts
import type { PointerState } from './pointerTypes';
import { distance, lerpVec2, sub, vec2 } from '../core/common/math';

export function normalizePointerEvent(
  ev: PointerEvent,
  prev: PointerState | null,
  rect: DOMRect,
): PointerState {
  const rawPosition = vec2(
    (ev.clientX - rect.left) / Math.max(rect.width, 1),
    (ev.clientY - rect.top) / Math.max(rect.height, 1),
  );

  const filteredPosition = prev ? lerpVec2(prev.filteredPosition, rawPosition, 0.35) : rawPosition;
  const delta = prev ? sub(filteredPosition, prev.filteredPosition) : vec2(0, 0);
  const dtMs = prev ? Math.max(1, ev.timeStamp - prev.timestampMs) : 1;
  const speed = prev ? distance(filteredPosition, prev.filteredPosition) / dtMs : 0;

  return {
    deviceType: ev.pointerType === 'pen' ? 'PEN' : ev.pointerType === 'touch' ? 'TOUCH' : 'MOUSE',
    rawPosition,
    filteredPosition,
    delta,
    speed,
    pressure: ev.pressure ?? 0.5,
    modifiers: {
      shift: ev.shiftKey,
      alt: ev.altKey,
      ctrlOrCmd: ev.ctrlKey || ev.metaKey,
      clutch: ev.altKey,
    },
    isDown: ev.buttons !== 0,
    timestampMs: ev.timeStamp,
  };
}
```

```ts
export function renderPath(ctx: CanvasRenderingContext2D, temporal: TemporalState, width: number, height: number): void {
  const points = [...temporal.body.committed, ...temporal.tail.points].map((p) => p.position);
  if (points.length < 2) return;

  ctx.save();
  ctx.strokeStyle = '#00ffcc';
  ctx.lineWidth = 2;
  ctx.beginPath();
  ctx.moveTo(points[0].x * width, points[0].y * height);
  for (let i = 1; i < points.length; i++) {
    ctx.lineTo(points[i].x * width, points[i].y * height);
  }
  ctx.stroke();
  ctx.restore();
}
```

```ts
import type { Vec2 } from '../../core/common/types';
import { distance } from '../../core/common/math';

export function meanContourDeviation(produced: Vec2[], truth: Vec2[]): number {
  if (produced.length === 0 || truth.length === 0) return Infinity;
  let total = 0;
  for (const p of produced) {
    let best = Infinity;
    for (const g of truth) {
      best = Math.min(best, distance(p, g));
    }
    total += best;
  }
  return total / produced.length;
}
```

# 4. Field Implementation — Real Phase 1 Logic

## 4.1 `src/core/field/preprocess.ts`

### Responsibility

Convert `ImageBitmap` to a luminance field.

### Implementation notes

* render `ImageBitmap` to an offscreen canvas
* read `ImageData`
* compute luminance per pixel
* store normalized luminance in `[0,1]`

### File content

```ts
import type { ScalarField2D } from './fieldTypes';

export async function extractLuminanceField(imageBitmap: ImageBitmap): Promise<ScalarField2D> {
  const width = imageBitmap.width;
  const height = imageBitmap.height;

  const canvas = new OffscreenCanvas(width, height);
  const ctx = canvas.getContext('2d', { willReadFrequently: true });
  if (!ctx) throw new Error('Failed to create 2D context for luminance extraction');

  ctx.drawImage(imageBitmap, 0, 0, width, height);
  const imageData = ctx.getImageData(0, 0, width, height);
  const src = imageData.data;
  const data = new Float32Array(width * height);

  for (let i = 0; i < width * height; i++) {
    const r = src[i * 4 + 0] / 255;
    const g = src[i * 4 + 1] / 255;
    const b = src[i * 4 + 2] / 255;
    data[i] = 0.2126 * r + 0.7152 * g + 0.0722 * b;
  }

  return { width, height, data };
}
```

## 4.2 `src/core/field/gradients.ts`

### Responsibility

Compute luminance gradients and edge magnitude.

### Implementation notes

Use a Sobel kernel for Phase 1. Scharr may be better later, but Sobel is fine for the first true pass.

### File content

```ts
import type { ScalarField2D } from './fieldTypes';

function idx(x: number, y: number, width: number): number {
  return y * width + x;
}

function clampCoord(v: number, max: number): number {
  return Math.max(0, Math.min(max, v));
}

export function computeGradients(field: ScalarField2D): {
  gradX: ScalarField2D;
  gradY: ScalarField2D;
  edgeMagnitude: ScalarField2D;
} {
  const { width, height, data } = field;
  const gx = new Float32Array(width * height);
  const gy = new Float32Array(width * height);
  const mag = new Float32Array(width * height);

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      const xm1 = clampCoord(x - 1, width - 1);
      const xp1 = clampCoord(x + 1, width - 1);
      const ym1 = clampCoord(y - 1, height - 1);
      const yp1 = clampCoord(y + 1, height - 1);

      const tl = data[idx(xm1, ym1, width)];
      const tc = data[idx(x,   ym1, width)];
      const tr = data[idx(xp1, ym1, width)];
      const ml = data[idx(xm1, y,   width)];
      const mr = data[idx(xp1, y,   width)];
      const bl = data[idx(xm1, yp1, width)];
      const bc = data[idx(x,   yp1, width)];
      const br = data[idx(xp1, yp1, width)];

      const sx = -tl + tr - 2 * ml + 2 * mr - bl + br;
      const sy = -tl - 2 * tc - tr + bl + 2 * bc + br;

      const i = idx(x, y, width);
      gx[i] = sx;
      gy[i] = sy;
      mag[i] = Math.hypot(sx, sy);
    }
  }

  return {
    gradX: { width, height, data: gx },
    gradY: { width, height, data: gy },
    edgeMagnitude: { width, height, data: mag },
  };
}
```

## 4.3 `src/core/field/tensor.ts`

### Responsibility

Compute a Phase-1 coherence field and tangent vector field.

### Implementation notes

Use a small local tensor window. Keep it CPU-side and explicit.

### File content

```ts
import type { ScalarField2D, VectorField2D } from './fieldTypes';

function idx(x: number, y: number, width: number): number {
  return y * width + x;
}

function clampCoord(v: number, max: number): number {
  return Math.max(0, Math.min(max, v));
}

export function computeCoherenceAndTangent(
  gradX: ScalarField2D,
  gradY: ScalarField2D,
): {
  coherence: ScalarField2D;
  tangent: VectorField2D;
} {
  const { width, height } = gradX;
  const coh = new Float32Array(width * height);
  const tx = new Float32Array(width * height);
  const ty = new Float32Array(width * height);

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      let j00 = 0;
      let j01 = 0;
      let j11 = 0;

      for (let oy = -1; oy <= 1; oy++) {
        for (let ox = -1; ox <= 1; ox++) {
          const sx = clampCoord(x + ox, width - 1);
          const sy = clampCoord(y + oy, height - 1);
          const k = idx(sx, sy, width);
          const gx = gradX.data[k];
          const gy = gradY.data[k];
          j00 += gx * gx;
          j01 += gx * gy;
          j11 += gy * gy;
        }
      }

      const trace = j00 + j11;
      const detTerm = Math.sqrt(Math.max(0, (j00 - j11) * (j00 - j11) + 4 * j01 * j01));
      const l1 = 0.5 * (trace + detTerm);
      const l2 = 0.5 * (trace - detTerm);
      const coherence = trace > 1e-6 ? (l1 - l2) / trace : 0;

      const vx = j01;
      const vy = l1 - j00;
      const len = Math.hypot(vx, vy);
      const nx = len > 1e-6 ? vx / len : 1;
      const ny = len > 1e-6 ? vy / len : 0;

      const i = idx(x, y, width);
      coh[i] = coherence;
      tx[i] = -ny;
      ty[i] = nx;
    }
  }

  return {
    coherence: { width, height, data: coh },
    tangent: { width, height, x: tx, y: ty },
  };
}
```

### File content

```ts
import type { ScalarField2D, TerrainConfig, VectorField2D } from './fieldTypes';

function idx(x: number, y: number, width: number): number {
  return y * width + x;
}

function clampCoord(v: number, max: number): number {
  return Math.max(0, Math.min(max, v));
}

function blur3(field: ScalarField2D): ScalarField2D {
  const { width, height, data } = field;
  const out = new Float32Array(width * height);

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      let sum = 0;
      let count = 0;
      for (let oy = -1; oy <= 1; oy++) {
        for (let ox = -1; ox <= 1; ox++) {
          const sx = clampCoord(x + ox, width - 1);
          const sy = clampCoord(y + oy, height - 1);
          sum += data[idx(sx, sy, width)];
          count++;
        }
      }
      out[idx(x, y, width)] = sum / count;
    }
  }

  return { width, height, data: out };
}

export function computeTerrain(
  edgeMagnitude: ScalarField2D,
  cornerness: ScalarField2D,
  config: TerrainConfig,
): {
  scalarTerrain: ScalarField2D;
  terrainGrad: VectorField2D;
} {
  const blurredEdge = blur3(edgeMagnitude);
  const blurredCorner = blur3(cornerness);
  const { width, height } = edgeMagnitude;
  const terrain = new Float32Array(width * height);

  for (let i = 0; i < terrain.length; i++) {
    terrain[i] = 1
      - config.boundaryGain * blurredEdge.data[i]
      + config.cornerPenaltyGain * blurredCorner.data[i];
  }

  const gx = new Float32Array(width * height);
  const gy = new Float32Array(width * height);

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      const xm1 = clampCoord(x - 1, width - 1);
      const xp1 = clampCoord(x + 1, width - 1);
      const ym1 = clampCoord(y - 1, height - 1);
      const yp1 = clampCoord(y + 1, height - 1);

      const i = idx(x, y, width);
      gx[i] = 0.5 * (terrain[idx(xp1, y, width)] - terrain[idx(xm1, y, width)]);
      gy[i] = 0.5 * (terrain[idx(x, yp1, width)] - terrain[idx(x, ym1, width)]);
    }
  }

  return {
    scalarTerrain: { width, height, data: terrain },
    terrainGrad: { width, height, x: gx, y: gy },
  };
}
```

```ts
import type { BoundaryFieldSet, FieldSample, ScalarField2D, VectorField2D } from './fieldTypes';
import type { Vec2 } from '../common/types';

function sampleScalar(field: ScalarField2D, uv: Vec2): number {
  const x = uv.x * (field.width - 1);
  const y = uv.y * (field.height - 1);

  const x0 = Math.floor(x);
  const y0 = Math.floor(y);
  const x1 = Math.min(field.width - 1, x0 + 1);
  const y1 = Math.min(field.height - 1, y0 + 1);
  const tx = x - x0;
  const ty = y - y0;

  const i00 = y0 * field.width + x0;
  const i10 = y0 * field.width + x1;
  const i01 = y1 * field.width + x0;
  const i11 = y1 * field.width + x1;

  const a = field.data[i00] * (1 - tx) + field.data[i10] * tx;
  const b = field.data[i01] * (1 - tx) + field.data[i11] * tx;
  return a * (1 - ty) + b * ty;
}

function sampleVector(field: VectorField2D, uv: Vec2): Vec2 {
  const x = uv.x * (field.width - 1);
  const y = uv.y * (field.height - 1);

  const x0 = Math.floor(x);
  const y0 = Math.floor(y);
  const x1 = Math.min(field.width - 1, x0 + 1);
  const y1 = Math.min(field.height - 1, y0 + 1);
  const tx = x - x0;
  const ty = y - y0;

  const i00 = y0 * field.width + x0;
  const i10 = y0 * field.width + x1;
  const i01 = y1 * field.width + x0;
  const i11 = y1 * field.width + x1;

  const ax = field.x[i00] * (1 - tx) + field.x[i10] * tx;
  const bx = field.x[i01] * (1 - tx) + field.x[i11] * tx;
  const ay = field.y[i00] * (1 - tx) + field.y[i10] * tx;
  const by = field.y[i01] * (1 - tx) + field.y[i11] * tx;

  return {
    x: ax * (1 - ty) + bx * ty,
    y: ay * (1 - ty) + by * ty,
  };
}

export function sampleField(field: BoundaryFieldSet, uv: Vec2): FieldSample {
  return {
    uv,
    edgeStrength: sampleScalar(field.edgeMagnitude, uv),
    tangent: sampleVector(field.tangent, uv),
    tangentValid: true,
    terrain: sampleScalar(field.scalarTerrain, uv),
    terrainGrad: sampleVector(field.terrainGrad, uv),
    coherence: sampleScalar(field.coherence, uv),
    cornerness: sampleScalar(field.cornerness, uv),
    fusedBoundary: sampleScalar(field.edgeMagnitude, uv),
  };
}

export function sampleNeighborhood(field: BoundaryFieldSet, uv: Vec2, radiusPx: number, count: number): FieldSample[] {
  const out: FieldSample[] = [];
  const rx = radiusPx / Math.max(field.width - 1, 1);
  const ry = radiusPx / Math.max(field.height - 1, 1);

  for (let i = 0; i < count; i++) {
    const t = count === 1 ? 0 : i / (count - 1);
    const dx = (t * 2 - 1) * rx;
    const dy = 0;
    const s = {
      x: Math.max(0, Math.min(1, uv.x + dx)),
      y: Math.max(0, Math.min(1, uv.y + dy)),
    };
    out.push(sampleField(field, s));
  }

  return out;
}
```

```ts
import type { ConfidenceRegime, HeadConfidence } from './motionTypes';

function regimeFromValue(value: number, prev: ConfidenceRegime): ConfidenceRegime {
  if (prev === 'HIGH' && value >= 0.63) return 'HIGH';
  if (prev === 'LOW' && value <= 0.36) return 'LOW';
  if (value >= 0.72) return 'HIGH';
  if (value <= 0.28) return 'LOW';
  return 'MID';
}

export function composePhase1Confidence(input: {
  edgeStrength: number;
  coherence: number;
  alignment: number;
  cornerness: number;
  prevRegime: ConfidenceRegime;
}): HeadConfidence {
  const edgeFactor = Math.max(0, Math.min(1, input.edgeStrength));
  const coherenceFactor = Math.max(0, Math.min(1, input.coherence));
  const alignmentFactor = Math.max(0, Math.min(1, input.alignment));
  const cornerPenalty = Math.max(0, Math.min(1, input.cornerness));

  const value = Math.max(
    0,
    Math.min(1, edgeFactor * 0.45 + coherenceFactor * 0.35 + alignmentFactor * 0.20 - cornerPenalty * 0.25),
  );

  return {
    value,
    regime: regimeFromValue(value, input.prevRegime),
    edgeFactor,
    coherenceFactor,
    alignmentFactor,
    roughnessPenalty: 0,
    cornerPenalty,
    continuityFactor: 0,
  };
```

## 7.2 Upgrade `forceModel.ts`

### Responsibility

The skeleton used terrain gradient, tangent, spring, and inertia. Keep that, but scale them with sampled field quality and interim confidence.

### Replacement content

```ts
import type { ForceBreakdown, HeadState } from './motionTypes';
import type { RuntimeCoefficients } from '../params/paramTypes';
import type { Vec2 } from '../common/types';
import { add, scale, sub, vec2, normalize, clampNorm } from '../common/math';

export type ForceModelInput = {
  head: HeadState;
  pointerProxy: Vec2;
  signedTangent: Vec2;
  coeffs: RuntimeCoefficients;
};

export function evaluateForces(input: ForceModelInput): ForceBreakdown {
  const sample = input.head.currentFieldSample;
  const terrainGrad = sample?.terrainGrad ?? vec2(0, 0);
  const edgeDir = normalize(scale(terrainGrad, -1));
  const edgeStrength = sample?.edgeStrength ?? 0;
  const coherence = sample?.coherence ?? 0;
  const confidence = input.head.confidence.value;

  const edge = scale(edgeDir, input.coeffs.edgeGain * edgeStrength);
  const tangent = scale(normalize(input.signedTangent), input.coeffs.tangentGain * coherence * Math.max(0.25, confidence));
  const spring = scale(sub(input.pointerProxy, input.head.kinematics.position), input.coeffs.springGain);
  const inertia = scale(normalize(input.head.kinematics.heading), input.coeffs.inertiaGain * Math.max(0, 1 - (sample?.cornerness ?? 0)));
  const brake = scale(normalize(input.head.kinematics.heading), -input.coeffs.brakeGain * (sample?.cornerness ?? 0));
  const hysteresis = vec2(0, 0);
  const userOverride = vec2(0, 0);

  const total = clampNorm(
    add(add(add(edge, tangent), add(spring, inertia)), add(add(brake, hysteresis), userOverride)),
    input.coeffs.maxSpeed,
  );

  return { edge, tangent, spring, inertia, brake, hysteresis, userOverride, total };
}
```

```ts
import type { Vec2 } from '../common/types';
import type { FloatingTail } from './temporalTypes';
import { distance, dot, normalize } from '../common/math';

export function shouldInsertTailPoint(
  position: Vec2,
  currentHeading: Vec2,
  tail: FloatingTail,
  sampleDistancePx: number,
  sampleAngleRad: number,
): boolean {
  const last = tail.points[tail.points.length - 1];
  if (!last) return true;

  const d = distance(last.position, position);
  if (d >= sampleDistancePx) return true;

  if (tail.points.length >= 2) {
    const prev = tail.points[tail.points.length - 2];
    const prevHeading = normalize({
      x: last.position.x - prev.position.x,
      y: last.position.y - prev.position.y,
    });
    const h = normalize(currentHeading);
    const angleLike = Math.acos(Math.max(-1, Math.min(1, dot(prevHeading, h))));
    if (angleLike >= sampleAngleRad) return true;
  }

  return false;
}
```

# 13. Immediate Engineering Notes

## 13.1 The normalized/pixel split is a trap

Your path state uses normalized coordinates. Your field arrays use pixel grids. Your sampling thresholds may be thought of in pixels. If you do not keep conversions explicit, the lasso will start behaving differently per image size like a cursed tape measure.

Keep these rules strict:

* simulation position in normalized UV
* field storage in pixel-indexed arrays
* all user-facing radii/thresholds explicitly converted

## 13.2 Sobel is not sacred

Phase 1 uses Sobel because it is simple and readable. Later you may replace it with Scharr or a GPU pass. Do not waste time ceremonializing the gradient operator until the control loop is alive.

## 13.3 Coherence is already a huge win

Even this simple tensor/coherence pass gives you something legacy magnetic lassos often handle badly: a distinction between strong organized contour and mushy isotropic nonsense. That’s already real progress.

## 13.4 Cornerness is still crude here

Yes, `cornerness = edgeMagnitude * (1 - coherence)` is a blunt instrument. That is acceptable in Phase 1. Its job is only to start teaching the motion model humility near turns.

# 2. Temporal Model Redesign

## 2.1 The fundamental split

The path is divided into two zones:

### Committed Body

The stable historical section of the path. This represents authored history that is no longer freely retractable by casual reverse motion.

### Floating Tail

The recent mutable section of the path. This is the live, revisable chronology zone. It should feel fluid and retractable.

This split is the heart of Phase 2.

## 2.2 Temporal state invariants

1. The committed body is **append-only** during ordinary forward drawing.
2. The floating tail is **mutable** and may retract under reverse motion.
3. Reverse motion may only damage committed history when an explicit policy or anchor semantics allow it.
4. Anchor types alter the commit and rollback grammar, not just the stored points.
5. Closure must be a temporal act, not a hidden geometric accident.

# 3. Floating Tail Semantics

## 3.1 Design principle

The floating tail is not just “the last few points.” It is a designated mutable chronological zone with special behavior.

## 3.2 Tail insertion rules

A tail point should be inserted when any of the following are true:

* arc-length since last tail point exceeds threshold
* heading changed enough to matter
* confidence changed substantially
* an anchor was inserted
* closure suggestion state changed

Phase 2 may still use the Phase 1 sampling logic as a base, but the tail now carries revision meaning.

## 3.3 Tail commit rules

Tail points migrate into the committed body when:

* the oldest tail segment exceeds a commit distance from the live head
* a hard anchor or branch commit declares commitment
* closure is accepted
* tail length exceeds safe mutable budget and the oldest region must age into history

### Important

Commit should not happen every frame. Overcommitting destroys reversibility and turns the lasso back into primitive bureaucracy.

## 3.4 Tail mutability rule

Only the floating tail is ordinarily retractable. Once points are committed into the body, rollback may not cross barriers unless explicit policy allows it.

## 3.5 `src/core/temporal/floatingTail.ts`

```ts
import type {
  FloatingTail,
  PathPoint,
  TailPoint,
  TemporalState,
} from './temporalTypes';
import { distance } from '../common/math';

export function createFloatingTail(maxPointCount = 96, maxArcLength = 0.18): FloatingTail {
  return {
    points: [],
    maxPointCount,
    maxArcLength,
  };
}

export function appendTailPoint(state: TemporalState, point: PathPoint): TemporalState {
  const tailPoint: TailPoint = { ...point, mutable: true };
  const nextTail = [...state.tail.points, tailPoint];

  while (nextTail.length > state.tail.maxPointCount) {
    nextTail.shift();
  }

  return {
    ...state,
    tail: {
      ...state.tail,
      points: nextTail,
    },
  };

export function computeTailArcLength(points: Array<{ position: { x: number; y: number } }>): number {
  let total = 0;
  for (let i = 1; i < points.length; i++) {
    total += distance(points[i - 1].position, points[i].position);
  }
  return total;
}

export function commitOldestTailPoints(
  state: TemporalState,
  livePosition: { x: number; y: number },
  commitDistanceNorm: number,

  const tail = [...state.tail.points];
  const committed = [...state.body.committed];
  let committedCount = 0;

  while (tail.length > 1) {
    const oldest = tail[0];
    if (distance(oldest.position, livePosition) < commitDistanceNorm) break;

        committed,

      tail: {
        ...state.tail,
        points: tail,
      },
```

# 4. Reverse Intent Detection

## 4.1 Goal

The user must be able to reverse along recent path history and retract the floating tail naturally.

Reverse intent is not simply “cursor moved backward a little.” It should be meaningful and robust.

## 4.2 Reverse intent conditions

A strong Phase 2 reverse-intent detector should consider:

* current pointer delta
* current head heading
* direction of the most recent tail segment
* local proximity to the tail corridor

A minimal rule:

> Reverse intent is active when the cursor/head motion projects strongly opposite the most recent tail direction.

# 5. Unzip Corridor Logic

## 5.1 Concept

The user should not need a special “undo mode.” If they physically reverse along the recent tail and remain near it, the tail should unzip.

That requires two tests:

1. **reverse intent is active**
2. **current position lies within an unzip corridor around the floating tail**

## 5.2 Corridor definition

The unzip corridor is a tubular region around the polyline formed by the floating tail.

For current head or pointer position `x`, compute:

`distTail(x) = min distance from x to each tail segment`

If `distTail(x) <= unzipRadiusNorm`, the user is “on the tail corridor.”

## 5.3 `src/core/temporal/unzip.ts` (new)

```ts
import type { CommitBarrier, TemporalState } from './temporalTypes';
import { distPointToSegment } from '../common/geometry';
import { distance } from '../common/math';

function canCrossBarrier(
  barrier: CommitBarrier,
  allowRollbackPastSoftBarrier: boolean,
  allowRollbackPastHardBarrier: boolean,
): boolean {
  if (barrier.type === 'SOFT') return allowRollbackPastSoftBarrier;
  if (barrier.type === 'HARD' || barrier.type === 'CORNER_PIN' || barrier.type === 'BRANCH_COMMIT') {
    return allowRollbackPastHardBarrier;
  }
  return true;
}

export function tailCorridorDistance(
  state: TemporalState,
  position: { x: number; y: number },
): number {
  const pts = state.tail.points;
  if (pts.length === 0) return Infinity;
  if (pts.length === 1) return distance(position, pts[0].position);

  let best = Infinity;
  for (let i = 1; i < pts.length; i++) {
    best = Math.min(best, distPointToSegment(position, pts[i - 1].position, pts[i].position));
  }
  return best;
}

export function unzipTail(
  state: TemporalState,
  position: { x: number; y: number },
  unzipRadiusNorm: number,
  reverseIntent: boolean,
  allowRollbackPastSoftBarrier: boolean,
  allowRollbackPastHardBarrier: boolean,
): { state: TemporalState; removedCount: number } {
  if (!reverseIntent) return { state, removedCount: 0 };
  if (tailCorridorDistance(state, position) > unzipRadiusNorm) return { state, removedCount: 0 };

  const tail = [...state.tail.points];
  let removedCount = 0;

  while (tail.length > 0) {
    const last = tail[tail.length - 1];
    if (distance(position, last.position) <= unzipRadiusNorm) break;

    const barrier = state.barriers.find((b) => b.pointId === last.id);
    if (barrier && !canCrossBarrier(barrier, allowRollbackPastSoftBarrier, allowRollbackPastHardBarrier)) {
      break;
    }

    tail.pop();
    removedCount++;
  }

  return {
    state: {
      ...state,
      tail: {
        ...state.tail,
        points: tail,
      },
    },
    removedCount,
  };
}
```

## 5.4 Important behavioral note

Unzip should act on the **floating tail first**. It should not greedily tear into the committed body just because the user wagged the pointer like an agitated metronome.

# 6. Commit Barriers

## 6.1 Why they matter

Without barriers, rollback can become too loose and erase authored intent too easily. With too many barriers, the tool becomes rigid and bureaucratic.

The answer is a hierarchy:

* no barrier
* soft barrier
* hard barrier
* corner pin barrier
* branch commit barrier

## 6.2 Barrier semantics

### No barrier

Ordinary tail points. Freely retractable.

### Soft barrier

Inserted by soft anchor. May block casual rollback but may be crossed under permissive policy.

### Hard barrier

Inserted by hard anchor. Should not be crossed during ordinary rollback.

### Corner pin barrier

Protects an authored corner from casual re-smoothing or rollback.

### Branch commit barrier

Protects a deliberate branch choice.

## 6.3 `src/core/temporal/anchors.ts` (new)

```ts
import type {
  AnchorEvent,
  CommitBarrier,
  CommitBarrierType,
  TemporalState,
} from './temporalTypes';

function barrierTypeFromAnchor(anchorType: AnchorEvent['type']): CommitBarrierType {
  switch (anchorType) {
    case 'SOFT': return 'SOFT';
    case 'HARD': return 'HARD';
    case 'CORNER_PIN': return 'CORNER_PIN';
    case 'BRANCH_COMMIT': return 'BRANCH_COMMIT';
  }
}

export function applyAnchor(state: TemporalState, anchor: AnchorEvent): {
  state: TemporalState;
  barrier: CommitBarrier | null;
} {
  const anchors = [...state.anchors, anchor];
  const tail = [...state.tail.points];
  const body = [...state.body.committed];

  const pointId = tail.length > 0 ? tail[tail.length - 1].id : body.length > 0 ? body[body.length - 1].id : null;
  if (!pointId) {
    return {
      state: { ...state, anchors },
      barrier: null,
    };
  }

  const barrier: CommitBarrier = {
    pointId,
    type: barrierTypeFromAnchor(anchor.type),
    timestampMs: anchor.timestampMs,
  };

  return {
    state: {
      ...state,
      anchors,
      barriers: [...state.barriers, barrier],
    },
    barrier,
  };
}
```

# 7. Anchor Semantics

## 7.1 Soft anchor

A soft anchor does three things:

* records authored intent at a location
* places a soft rollback barrier
* encourages nearby tail commitment without freezing everything

## 7.2 Hard anchor

A hard anchor:

* records authored intent
* inserts a hard barrier
* should immediately move the local point into committed history if present in the tail

### Add helper to `anchors.ts`

```ts
export function forceCommitTailTip(state: TemporalState): TemporalState {
  const tail = [...state.tail.points];
  const body = [...state.body.committed];
  if (tail.length === 0) return state;

  const tip = tail.pop()!;
  body.push(tip);

  return {
    ...state,
    body: { ...state.body, committed: body },
    tail: { ...state.tail, points: tail },
  };
}
```

# 8. Closure Behavior

## 8.1 Philosophy

Closure is not a semantic region guess. It is a temporal act the user can accept or ignore.

## 8.2 Closure suggestion

Suggest closure when:

* head is near the start point
* current heading is not wildly inconsistent
* at least a minimal path length exists

Do **not** auto-close from fantasy. The tool is not a psychic polygon butler.

## 8.3 `src/core/temporal/closure.ts` (new)

```ts
import type { TemporalState } from './temporalTypes';
import { distance, dot, normalize } from '../common/math';

export function getPathStart(state: TemporalState): { x: number; y: number } | null {
  if (state.body.committed.length > 0) return state.body.committed[0].position;
  if (state.tail.points.length > 0) return state.tail.points[0].position;
  return null;
}

export function shouldSuggestClosure(
  state: TemporalState,
  position: { x: number; y: number },
  heading: { x: number; y: number },
  closureRadiusNorm: number,
): boolean {
  const start = getPathStart(state);
  if (!start) return false;

  const totalPoints = state.body.committed.length + state.tail.points.length;
  if (totalPoints < 3) return false;

  const near = distance(position, start) <= closureRadiusNorm;
  if (!near) return false;

  const toStart = normalize({ x: start.x - position.x, y: start.y - position.y });
  const headingN = normalize(heading);
  const alignment = dot(headingN, toStart);

  return alignment > -0.5;
}

export function acceptClosure(state: TemporalState): TemporalState {
  const start = getPathStart(state);
  if (!start) return state;

  const closurePointId = state.body.committed[0]?.id ?? state.tail.points[0]?.id ?? null;

  return {
    ...state,
    body: {
      ...state.body,
      closed: true,
      closurePointId,
    },
    closure: {
      suggested: false,
      suggestedPoint: null,
      accepted: true,
    },
  };
}
```

## 9.1 Replace `src/core/temporal/temporalEngine.ts` with:

```ts
import type {
  AnchorEvent,
  TemporalState,
  TemporalUpdateInput,
  TemporalUpdateResult,
} from './temporalTypes';
import { createFloatingTail, appendTailPoint, commitOldestTailPoints } from './floatingTail';
import { shouldInsertTailPoint } from './pathSampling';
import { unzipTail } from './unzip';
import { applyAnchor, forceCommitTailTip } from './anchors';
import { shouldSuggestClosure, acceptClosure } from './closure';

export class TemporalEngine {
  beginPath(): TemporalState {
    return {
      body: {
        committed: [],
        closed: false,
        closurePointId: null,
      },
      tail: createFloatingTail(),
      barriers: [],
      anchors: [],
      closure: {
        suggested: false,
        suggestedPoint: null,
        accepted: false,
      },
    };
  }

  update(state: TemporalState, input: TemporalUpdateInput): TemporalUpdateResult {
    const events: TemporalUpdateResult['events'] = [];

    if (shouldInsertTailPoint(input.position, input.heading, state.tail, input.sampleDistanceNorm, input.sampleAngleRad)) {
      const id = `${input.timestampMs}-${state.tail.points.length}`;
      state = appendTailPoint(state, {
        id,
        position: input.position,
        timestampMs: input.timestampMs,
        confidenceAtInsert: input.confidence,
        cornernessAtInsert: input.cornerness,
        source: 'AUTO_SAMPLE',
      });
      events.push({ type: 'TAIL_POINT_ADDED', pointId: id });
    }

    const unzip = unzipTail(
      state,
      input.position,
      input.unzipRadiusNorm,
      input.reverseIntent,
      input.allowRollbackPastSoftBarrier,
      input.allowRollbackPastHardBarrier,
    );
    state = unzip.state;
    if (unzip.removedCount > 0) {
      events.push({ type: 'TAIL_UNZIPPED', removedCount: unzip.removedCount });
    }

    const commit = commitOldestTailPoints(state, input.position, input.commitDistanceNorm);
    state = commit.state;
    if (commit.committedCount > 0) {
      events.push({ type: 'TAIL_COMMITTED', committedCount: commit.committedCount });
    }

    if (input.pendingAnchor) {
      const anchorResult = applyAnchor(state, input.pendingAnchor as AnchorEvent);
      state = anchorResult.state;
      events.push({ type: 'ANCHOR_INSERTED', anchorId: input.pendingAnchor.id });
      if (anchorResult.barrier) {
        events.push({
          type: 'BARRIER_CREATED',
          pointId: anchorResult.barrier.pointId,
          barrierType: anchorResult.barrier.type,
        });
      }

      if (input.pendingAnchor.type === 'HARD' || input.pendingAnchor.type === 'CORNER_PIN' || input.pendingAnchor.type === 'BRANCH_COMMIT') {
        state = forceCommitTailTip(state);
      }
    }

    const suggestClosure = shouldSuggestClosure(state, input.position, input.heading, input.commitDistanceNorm);
    if (suggestClosure && !state.closure.suggested) {
      state = {
        ...state,
        closure: {
          suggested: true,
          suggestedPoint: state.body.committed[0]?.position ?? state.tail.points[0]?.position ?? null,
          accepted: false,
        },
      };
      events.push({ type: 'CLOSURE_SUGGESTED' });
    } else if (!suggestClosure && state.closure.suggested) {
      state = {
        ...state,
        closure: {
          suggested: false,
          suggestedPoint: null,
          accepted: false,
        },
      };
      events.push({ type: 'CLOSURE_REVOKED' });
    }

    return { state, events };
  }

  closePath(state: TemporalState): TemporalState {
    return acceptClosure(state);
  }

  reset(): TemporalState {
    return this.beginPath();
  }
}
```

## 9.2 Note on ordering

The order matters:

1. tail insertion
2. unzip
3. commit aging
4. anchor handling
5. closure suggestion

This order keeps the live chronology honest. If you anchor before unzip, or commit before checking rollback conditions, temporal weirdness starts breeding in the walls.

# 2. Confidence Model Upgrade

## 2.1 Why Phase 1 confidence is no longer enough

The Phase 1 confidence score was intentionally crude. It proved the loop. But Phase 3 needs confidence that can answer *why* the system is confident or not.

The factors should remain explicit and individually inspectable.

## 2.2 Confidence factors

We will use these factors:

* **edge factor**: strength of boundary evidence under the head
* **coherence factor**: how organized the local contour structure is
* **alignment factor**: agreement between user motion and signed tangent direction
* **roughness penalty**: instability in recent path heading changes
* **corner penalty**: local cornerness / junction pressure
* **continuity factor**: whether the current path continues the recent contour family
* **lost-state penalty**: hard authority collapse when contour guidance is gone

## 2.3 Upgrade `motionTypes.ts`

The existing `HeadConfidence` type is already close enough, but add continuity as a first-class live contributor and ensure the ambiguity vector is always present.

No schema change is strictly required if you kept the Phase 2 structure.

## 2.4 `src/core/motion/confidence.ts`

### Replace with:

```ts
import type { ConfidenceRegime, HeadConfidence } from './motionTypes';

function clamp01(v: number): number {
  return Math.max(0, Math.min(1, v));
}

function regimeFromValue(value: number, prev: ConfidenceRegime): ConfidenceRegime {
  if (prev === 'HIGH' && value >= 0.63) return 'HIGH';
  if (prev === 'LOW' && value <= 0.36) return 'LOW';
  if (value >= 0.72) return 'HIGH';
  if (value <= 0.28) return 'LOW';
  return 'MID';
}

export type ConfidenceInputs = {
  edgeStrength: number;
  coherence: number;
  alignment: number;
  roughness: number;
  cornerness: number;
  continuity: number;
  lostState: boolean;
  prevRegime: ConfidenceRegime;
};

export function composeConfidence(input: ConfidenceInputs): HeadConfidence {
  const edgeFactor = clamp01(input.edgeStrength);
  const coherenceFactor = clamp01(input.coherence);
  const alignmentFactor = clamp01(input.alignment);
  const roughnessPenalty = clamp01(input.roughness);
  const cornerPenalty = clamp01(input.cornerness);
  const continuityFactor = clamp01(input.continuity);
  const lostPenalty = input.lostState ? 1 : 0;

  const positive = Math.pow(edgeFactor, 0.9)
    * Math.pow(coherenceFactor, 0.9)
    * Math.pow(alignmentFactor, 0.7)
    * Math.pow(Math.max(0.05, continuityFactor), 0.8);

  const penalties = clamp01(
    0.20 * roughnessPenalty +
    0.25 * cornerPenalty +
    0.80 * lostPenalty,
  );

  const value = clamp01(positive * (1 - penalties));
  const regime = regimeFromValue(value, input.prevRegime);

  return {
    value,
    regime,
    edgeFactor,
    coherenceFactor,
    alignmentFactor,
    roughnessPenalty,
    cornerPenalty,
    continuityFactor,
  };
}
```

## 2.5 Roughness estimator

Add a helper to compute short-horizon heading instability from recent tail geometry.

### `src/core/motion/roughness.ts` (new)

```ts
import type { TemporalState } from '../temporal/temporalTypes';
import { dot, normalize } from '../common/math';

export function computeTailRoughness(state: TemporalState): number {
  const pts = state.tail.points;
  if (pts.length < 3) return 0;

  let total = 0;
  let count = 0;

  for (let i = 2; i < pts.length; i++) {
    const a = normalize({
      x: pts[i - 1].position.x - pts[i - 2].position.x,
      y: pts[i - 1].position.y - pts[i - 2].position.y,
    });
    const b = normalize({
      x: pts[i].position.x - pts[i - 1].position.x,
      y: pts[i].position.y - pts[i - 1].position.y,
    });
    total += 1 - Math.max(-1, Math.min(1, dot(a, b)));
    count++;
  }

  return count > 0 ? Math.max(0, Math.min(1, total / count)) : 0;
}
```

```ts
import type { AmbiguityVector } from './motionTypes';

function clamp01(v: number): number {
  return Math.max(0, Math.min(1, v));
}

export type AmbiguityInputs = {
  sample: FieldSample;
  parallelPeakScore: number;
  continuityDrop: number;
  intentAlignment: number;

    continuity,
    intentConflict,
```

# 4. Contour-Family Extraction and Side-Lock

## 4.1 Goal

Prevent chatter between adjacent contour channels while still allowing deliberate user-led switching.

This is the first serious defense against the classic zipper-edge clown show.

## 4.2 Phase 3 simplification

We do not yet need a perfect family model. A good first version is:

* sample boundary strength along the local normal line
* detect multiple strong candidates
* score their spacing and similarity
* use that to estimate side ambiguity and maintain persistence

## 4.3 `src/core/motion/sideLock.ts` (new)

```ts
import type { BoundaryFieldSet, FieldSample } from '../field/fieldTypes';
import type { SideLockState } from './motionTypes';
import type { Vec2 } from '../common/types';
import { dot, normalize, perp } from '../common/math';
import { sampleField } from '../field/sampler';

function clamp01(v: number): number {
  return Math.max(0, Math.min(1, v));
}

export type FamilyCandidate = {
  id: string;
  offset: number;
  strength: number;
  score: number;
};

export type SideLockUpdateInput = {
  field: BoundaryFieldSet;
  headUv: Vec2;
  signedTangent: Vec2;
  prev: SideLockState;
  userIntent: Vec2;
  sampleRadiusPx: number;
  samplesPerSide: number;
  switchMargin: number;
  clutchActive: boolean;
};

function sampleCandidates(
  field: BoundaryFieldSet,
  headUv: Vec2,
  signedTangent: Vec2,
  sampleRadiusPx: number,
  samplesPerSide: number,
): FamilyCandidate[] {
  const n = normalize(perp(signedTangent));
  const stepNorm = sampleRadiusPx / Math.max(field.width, field.height);
  const candidates: FamilyCandidate[] = [];

  for (let i = -samplesPerSide; i <= samplesPerSide; i++) {
    const offset = i * stepNorm;
    const uv = {
      x: Math.max(0, Math.min(1, headUv.x + n.x * offset)),
      y: Math.max(0, Math.min(1, headUv.y + n.y * offset)),
    };
    const s: FieldSample = sampleField(field, uv);
    candidates.push({
      id: `cand-${i}`,
      offset,
      strength: s.edgeStrength,
      score: s.edgeStrength * (0.6 + 0.4 * s.coherence),
    });
  }

  return candidates.sort((a, b) => b.score - a.score);
}

export function estimateParallelPeakScore(candidates: FamilyCandidate[]): number {
  if (candidates.length < 2) return 0;
  const a = candidates[0];
  const b = candidates[1];
  const similarity = 1 - Math.abs(a.score - b.score);
  const spacing = Math.abs(a.offset - b.offset);
  return clamp01(similarity * Math.min(1, spacing * 40));
}

export function updateSideLock(input: SideLockUpdateInput): {
  next: SideLockState;
  parallelPeakScore: number;
} {
  const candidates = sampleCandidates(
    input.field,
    input.headUv,
    input.signedTangent,
    input.sampleRadiusPx,
    input.samplesPerSide,
  );

  const parallelPeakScore = estimateParallelPeakScore(candidates);
  const best = candidates[0] ?? null;
  if (!best) {
    return {
      next: {
        active: false,
        preferredNormal: null,
        familyId: null,
        strength: 0,
        persistence: 0,
      },
      parallelPeakScore,
    };
  }

  const currentScore = input.prev.familyId === best.id ? best.score + 0.15 : best.score;
  const shouldRelease = input.clutchActive || best.score < 0.05;
  if (shouldRelease) {
    return {
      next: {
        active: false,
        preferredNormal: null,
        familyId: null,
        strength: 0,
        persistence: 0,
      },
      parallelPeakScore,
    };
  }

  if (!input.prev.active) {
    const normal = normalize(perp(input.signedTangent));
    const normalSign = dot(normal, input.userIntent) >= 0 ? normal : { x: -normal.x, y: -normal.y };
    return {
      next: {
        active: true,
        preferredNormal: normalSign,
        familyId: best.id,
        strength: best.score,
        persistence: 1,
      },
      parallelPeakScore,
    };
  }

  if (input.prev.familyId !== best.id && best.score < input.prev.strength + input.switchMargin) {
    return {
      next: {
        ...input.prev,
        persistence: Math.max(0, input.prev.persistence - 1),
      },
      parallelPeakScore,
    };
  }

  return {
    next: {
      ...input.prev,
      active: true,
      familyId: best.id,
      strength: best.score,
      persistence: Math.min(255, input.prev.persistence + 1),
    },
    parallelPeakScore,
  };
}
```

## 4.4 Notes

This is still intentionally local and mechanical. No semantic nonsense. No object guesses. Just normal-line evidence and switching resistance.

# 5. Continuity Estimation

## 5.1 Purpose

Continuity is the difference between “still plausibly on the same contour” and “the field is becoming mush or branch soup.”

## 5.2 `src/core/motion/continuity.ts` (new)

```ts
import type { HeadState } from './motionTypes';

function clamp01(v: number): number {
  return Math.max(0, Math.min(1, v));
}

export function estimateContinuity(head: HeadState, currentEdgeStrength: number, currentCoherence: number): {
  continuity: number;
  continuityDrop: number;
} {
  const prevEdge = head.currentFieldSample?.edgeStrength ?? currentEdgeStrength;
  const prevCoh = head.currentFieldSample?.coherence ?? currentCoherence;

  const edgeDrop = Math.max(0, prevEdge - currentEdgeStrength);
  const cohDrop = Math.max(0, prevCoh - currentCoherence);
  const drop = clamp01(0.6 * edgeDrop + 0.4 * cohDrop);

  return {
    continuity: clamp01(1 - drop),
    continuityDrop: drop,
  };
}
```

# 6. Corner Braking

## 6.1 Goal

Reduce overshoot near turns and junction-like regions.

Phase 3 does not need the full branch-cone system yet. A good first step is a corner braking factor driven by cornerness + junction ambiguity.

```ts
export function computeCornerBrakeFactor(cornerness: number, junctionAmbiguity: number): number {
  const raw = 0.65 * cornerness + 0.35 * junctionAmbiguity;
  return Math.max(0, Math.min(1, raw));
}
```

This factor will modulate inertia downward and brake upward.

# 7. Early Lost-State Behavior

## 7.1 Goal

When evidence collapses, the tool should stop pretending. It should retreat toward manual dominance.

## 7.2 Entry rule

Enter lost-state if all are true for a short persistence window:

* confidence regime is LOW
* edge strength below threshold
* coherence below threshold
* no strong side-lock candidate or active contour family confidence

## 7.3 Exit rule

Exit lost-state when a nearby sample recovers adequate edge + coherence.

## 7.4 `src/core/motion/lostState.ts` (new)

```ts
import type { BoundaryFieldSet } from '../field/fieldTypes';
import type { Vec2 } from '../common/types';
import { sampleField } from '../field/sampler';

export type LostStateUpdateInput = {
  prevLostState: boolean;
  prevLostFrames: number;
  field: BoundaryFieldSet;
  headUv: Vec2;
  confidenceValue: number;
  edgeStrength: number;
  coherence: number;
  reacquireRadiusPx: number;
};

export type LostStateUpdateResult = {
  lostState: boolean;
  lostFrames: number;
};

export function updateLostState(input: LostStateUpdateInput): LostStateUpdateResult {
  const weakNow = input.confidenceValue <= 0.28 && input.edgeStrength < 0.10 && input.coherence < 0.20;

  let lostFrames = weakNow ? input.prevLostFrames + 1 : 0;
  let lostState = input.prevLostState;

  if (!lostState && lostFrames >= 3) {
    lostState = true;
  }

  if (lostState) {
    const r = input.reacquireRadiusPx / Math.max(input.field.width, input.field.height);
    const offsets = [
      { x: 0, y: 0 },
      { x: r, y: 0 },
      { x: -r, y: 0 },
      { x: 0, y: r },
      { x: 0, y: -r },
    ];

    for (const off of offsets) {
      const s = sampleField(input.field, {
        x: Math.max(0, Math.min(1, input.headUv.x + off.x)),
        y: Math.max(0, Math.min(1, input.headUv.y + off.y)),
      });
      if (s.edgeStrength >= 0.15 && s.coherence >= 0.25) {
        return { lostState: false, lostFrames: 0 };
      }
    }
  }

  return { lostState, lostFrames };
}
```

## 7.5 Behavior rule

When lost-state is active:

* lower edge gain
* lower tangent gain
* raise spring dominance
* do not snap long distances

This is not “be smarter.” It is “admit you’re blind and let the human drive.” Very important distinction, unless one enjoys building liars.

## 9.1 Replace `src/core/motion/motionEngine.ts` with:

```ts
import { resolveSignedTangent } from './tangentSign';
import { evaluateForces } from './forceModel';
import { integrateHead } from './integrator';
import { composeConfidence } from './confidence';
import { estimateAmbiguity } from './ambiguity';
import { updateSideLock } from './sideLock';
import { estimateContinuity } from './continuity';
import { computeTailRoughness } from './roughness';
import { computeCornerBrakeFactor } from './braking';
import { updateLostState } from './lostState';
import type { MotionStepRequest, MotionStepResult } from './motionTypes';
import { dot, normalize, normalizeOrNull } from '../common/math';

export class MotionEngine {
  step(request: MotionStepRequest): MotionStepResult {
    const actualSample = request.fieldSampler(request.field, request.head.kinematics.position);

    const signedTangent = resolveSignedTangent({
      unsignedTangent: actualSample.tangent,
      prevStableTangent: request.head.lastStableTangent,
      intentDir: normalizeOrNull(request.intent.forwardIntent),
      velocityDir: normalizeOrNull(request.head.kinematics.velocity),
      wPrev: 0.7,
      wIntent: 0.2,
      wVel: 0.1,
      hysteresisMargin: 0.05,
    });

    const intentDir = normalize(request.intent.forwardIntent);
    const tangentDir = normalize(signedTangent);
    const alignment = 0.5 + 0.5 * dot(intentDir, tangentDir);

    const sideLockResult = updateSideLock({
      field: request.field,
      headUv: request.head.kinematics.position,
      signedTangent,
      prev: request.head.sideLock,
      userIntent: request.intent.forwardIntent,
      sampleRadiusPx: 6,
      samplesPerSide: 3,
      switchMargin: 0.08,
      clutchActive: request.intent.clutchActive,
    });

    const continuity = estimateContinuity(
      request.head,
      actualSample.edgeStrength,
      actualSample.coherence,
    );

    const ambiguity = estimateAmbiguity({
      sample: actualSample,
      parallelPeakScore: sideLockResult.parallelPeakScore,
      continuityDrop: continuity.continuityDrop,
      intentAlignment: alignment,
    });

    const lost = updateLostState({
      prevLostState: request.head.lostState,
      prevLostFrames: request.head.lostStateFrames,
      field: request.field,
      headUv: request.head.kinematics.position,
      confidenceValue: request.head.confidence.value,
      edgeStrength: actualSample.edgeStrength,
      coherence: actualSample.coherence,
      reacquireRadiusPx: request.coeffs.reacquireRadius * Math.max(request.field.width, request.field.height),
    });

    const confidence = composeConfidence({
      edgeStrength: Math.min(1, actualSample.edgeStrength),
      coherence: Math.min(1, actualSample.coherence),
      alignment,
      roughness: computeTailRoughness(request.temporalState),
      cornerness: Math.min(1, actualSample.cornerness),
      continuity: continuity.continuity,
      lostState: lost.lostState,
      prevRegime: request.head.confidence.regime,
    });

    const cornerBrakeFactor = computeCornerBrakeFactor(actualSample.cornerness, ambiguity.junction);

    const nextHeadSeed = {
      ...request.head,
      confidence,
      ambiguity,
      sideLock: sideLockResult.next,
      currentFieldSample: actualSample,
      previousFieldSample: request.head.currentFieldSample,
      lastStableTangent: signedTangent,
      lostState: lost.lostState,
      lostStateFrames: lost.lostFrames,
    };

    const forces = evaluateForces({
      head: nextHeadSeed,
      pointerProxy: request.intent.proxyPosition,
      signedTangent,
      coeffs: request.coeffs,
      cornerBrakeFactor,
    });

    const nextKinematics = integrateHead(nextHeadSeed, forces.total, request.coeffs, request.dtSec);

    const nextHead = {
      ...nextHeadSeed,
      kinematics: nextKinematics,
    };

    return {
      nextHead,
      forces,
      events: [],
    };
  }
}
```

# 10. Force Model Upgrade

## 10.1 Replace `src/core/motion/forceModel.ts` with:

```ts
import type { ForceBreakdown, HeadState } from './motionTypes';
import type { RuntimeCoefficients } from '../params/paramTypes';
import type { Vec2 } from '../common/types';
import { add, clampNorm, normalize, perp, scale, sub, vec2 } from '../common/math';

export type ForceModelInput = {
  head: HeadState;
  pointerProxy: Vec2;
  signedTangent: Vec2;
  coeffs: RuntimeCoefficients;
  cornerBrakeFactor: number;
};

export function evaluateForces(input: ForceModelInput): ForceBreakdown {
  const sample = input.head.currentFieldSample;
  const terrainGrad = sample?.terrainGrad ?? vec2(0, 0);
  const edgeDir = normalize(scale(terrainGrad, -1));
  const edgeStrength = sample?.edgeStrength ?? 0;
  const coherence = sample?.coherence ?? 0;
  const confidence = input.head.confidence.value;
  const side = input.head.sideLock;

  const edge = scale(edgeDir, input.coeffs.edgeGain * edgeStrength);
  const tangent = scale(normalize(input.signedTangent), input.coeffs.tangentGain * coherence * Math.max(0.15, confidence));
  const spring = scale(sub(input.pointerProxy, input.head.kinematics.position), input.coeffs.springGain);
  const inertia = scale(normalize(input.head.kinematics.heading), input.coeffs.inertiaGain * (1 - input.cornerBrakeFactor));
  const brake = scale(normalize(input.head.kinematics.heading), -input.coeffs.brakeGain * input.cornerBrakeFactor);

  let hysteresis = vec2(0, 0);
  if (side.active && side.preferredNormal) {
    hysteresis = scale(normalize(side.preferredNormal), input.coeffs.hysteresisGain * 0.05 * Math.min(1, side.persistence / 8));
  }

  const userOverride = vec2(0, 0);

  const total = clampNorm(
    add(add(add(edge, tangent), add(spring, inertia)), add(add(brake, hysteresis), userOverride)),
    input.coeffs.maxSpeed,
  );

  return { edge, tangent, spring, inertia, brake, hysteresis, userOverride, total };
}
```

This is still intentionally modest. It is enough to bias behavior without turning the side-lock into a tyrant in a cheap plastic crown.

# 13. Metrics

## 13.1 `src/bench/metrics/familySwitchRate.ts` (new)

```ts
export function familySwitchRate(familyIds: Array<string | null>): number {
  if (familyIds.length < 2) return 0;
  let switches = 0;
  let steps = 0;
  for (let i = 1; i < familyIds.length; i++) {
    if (familyIds[i - 1] == null || familyIds[i] == null) continue;
    steps++;
    if (familyIds[i - 1] !== familyIds[i]) switches++;
  }
  return steps > 0 ? switches / steps : 0;
}
```

```ts
export function lostStateRate(states: boolean[]): number {

  return count / states.length;
}
```

# 15. Phase 3 Acceptance Gates

A Phase 3 branch is acceptable only if all of the following are true:

# 16. Engineering Notes

## 16.1 Side-lock should be weakly persuasive, not authoritarian

If side-lock makes it difficult for the user to intentionally switch contours, you built a small tyrant instead of a lasso helper. The user must remain the boss.

## 16.2 Lost-state should feel honest, not dramatic

When the field goes weak, the system should quietly hand back control. No desperate leaps. No ghost-snapping to faraway edges like it’s chasing rumors in a tavern.

## 16.3 Confidence factors are tuning gold

Do not bury them. They are exactly how you learn whether the tool is wrong because the field is weak, coherence is low, continuity broke, or the user is simply steering elsewhere.

## 16.4 Ambiguity is not failure

Ambiguity is the point where the tool proves whether it is lawful. A system that becomes more forceful at ambiguity is not smart. It is just arrogant with a nicer résumé.
