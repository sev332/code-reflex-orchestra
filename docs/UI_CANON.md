# THE DEFINITIVE CANON OF VISUAL INTERFACE DESIGN FOR AI-ASSISTED BROWSER APPLICATIONS

## A Total Doctrine of Persistent Shells, Living Workspaces, Visual Instruments, and Humane Power

---

## Dedication

This canon exists for software that refuses to be mediocre.

It exists for interfaces that do not reduce making, thinking, simulating, composing, designing, editing, directing, researching, or orchestrating into dead forms and obedient little number boxes.

It exists for browser applications that aspire to become true environments: places where intelligence is present, time is visible, structure is legible, tools are embodied, and the user’s thought is given form instead of friction.

This is not a style guide for pretty screens.

This is a doctrine for building high-agency software.

# 1. The Domain of the Canon

This canon does **not** claim to govern every application.

It is not written for tiny utilities, one-shot flows, checkout pages, basic dashboards, or trivial mobile interactions whose main job is to be quick and leave.

This canon is written for a specific class of software:

**AI-assisted browser applications of high agency.**

That means applications where users are doing one or more of the following:

* creating
* editing
* sequencing
* simulating
* directing
* orchestrating
* designing
* comparing
* inspecting
* researching
* iterating
* branching
* versioning
* collaborating with intelligent assistance
* moving between different but related forms of work without losing cognitive continuity

This canon governs software where the user is not merely consuming outcomes.

The user is steering reality.

# 2. The Prime Thesis

## The interface must become a living workspace.

The old browser-app model treated each page like a separate room with partial amnesia.

The user moved from one route to another and lost context.
The assistant disappeared.
The history disappeared.
The process disappeared.
The ontology changed shape.
The software kept asking the user to reorient.

That is primitive.

The new model is different.

A true AI-assisted browser application must provide a **persistent cognitive shell** inside which page-specific systems may change while intelligence, time, continuity, and spatial trust remain intact.

The user should feel that they are inside **one environment** with multiple territories, not inside a bag of disconnected pages.

---

# 3. The First Distinction: Shell, Page, Instrument

This distinction is absolute. Confusing these layers causes nearly all interface decay.

## 3.1 The Shell

The shell is the persistent architecture that survives page changes.
It is the app’s enduring body.
It holds the faculties that must remain present across the environment.

The shell governs:

* global navigation
* persistent AI presence
* global history and process continuity
* spatial trust
* stable edges and their meanings

## 3.2 The Page

The page is a domain territory within the shell.
Each page has its own ontology, local systems, local tools, local submodes, and local interactions.

A page may be:

* an image editor
* a 3D scene workspace
* a script editor
* a planning environment
* a simulation page
* a knowledge page
* a graph page
* a composition board
* a code environment

## 3.3 The Instrument

The instrument is the actual control surface through which a domain concept is manipulated.

An instrument is not a “settings panel.”
An instrument is a visual, interactive, embodied control for a specific phenomenon.

Examples:

* a curve editor
* a morphology editor
* a trajectory graph
* a cloth interaction surface
* a timeline clip editor
* a layering system
* a force balance display
* a node graph
* a gradient editor
* a spatial handle system

## 3.4 The Central Rule

**The shell provides continuity. The page provides ontology. The instrument provides agency.**

If any of these are confused, the experience becomes mush.

# 4. The Great Spatial Order

Every edge of the screen must have a stable semantic role.

This is one of the deepest laws in the entire canon.

Not because users consciously recite it, but because their nervous system learns it.

The canonical arrangement is:

* **Top** = global identity, global navigation, project scope, global actions
* **Second top row** = internal page modes, sub-pages, local route families
* **Left** = page ontology, local systems, local tools, object families, structural categories
* **Center** = the thing itself; the workspace; the artifact; the scene; the canvas; the graph; the text; the world
* **Right** = persistent intelligence and, where needed, explicitly separated contextual inspection
* **Bottom** = time, history, process, jobs, sequences, traces, branches, variants, outputs in motion

This order is not arbitrary.

It is cognitively natural because it aligns with how many complex acts of making actually work:

1. What world am I in?
2. What submode of that world am I in?
3. What systems live here?
4. What am I directly touching?
5. What is the assistant saying or understanding?
6. What happened over time, what is running, what changed, what branches exist?

This is why the arrangement feels calm even when it is dense.

It maps meaning to territory.

# 5. The Persistent Cognitive Shell

The shell is the master invention.

It is what elevates the application from a collection of routes into a continuous thought-space.

The shell must satisfy five properties:

## 5.1 Persistence

Its core faculties remain available across all main pages.

## 5.2 Stability

Its regions do not change meaning arbitrarily.

## 5.3 Adaptability

Pages may express different local systems without breaking the shell’s semantic order.

## 5.4 Continuity

The user should retain cognitive orientation when moving between pages.

## 5.5 Latent Power

The shell must feel capable without being visually noisy.
It should suggest depth without screaming it.

The shell is not decorative chrome.
It is the **metabolism** of the environment.

# 6. The Top Bar: Global Identity and World Switching

The top bar defines the world-level structure of the application.

It is where the user understands:

* what main page they are in
* what project or workspace is open
* what global commands are available
* what the broad sections of the environment are

The top bar must therefore feel like **geography**, not like a random toolbar.

## 6.1 The Top Bar Governs Main Worlds

Top-bar items should represent the application’s primary domains.
For example:

* Create
* Image
* 3D
* Script
* Graph
* Plan
* Assets
* Research
* Code
* Publish

These should not be ephemeral toggles.
They should feel like territories within one civilization.

## 6.2 The Top Bar Must Be Sparse and Decisive

Do not flood the top bar with local toggles, minor inspectors, or one-off page actions.
That is not its job.

The top bar is for:

* world switching
* project scope
* identity
* global state
* universal actions

## 6.3 The Top Bar Must Signal Persistent Context

It must preserve the sense that the project, conversation, history, and assistant continue even as the user moves between pages.

The application is one being with many chambers.

---

# 7. The Sub-Page Bar: Local Modes and Internal Territories

Below the top bar lives a second bar for **internal page territories**.

This row is essential in sophisticated pages because a page often contains multiple internally coherent modes.

Examples:

Within Image:

* Edit
* Mask
* Layers
* Retouch
* Effects
* Output

Within Script:

* Outline
* Draft
* Timeline
* Variants
* Notes

Within 3D:

* Scene
* Materials
* Physics
* Lighting
* Render

Within Research:

* Sources
* Synthesis
* Notes
* Claims
* Graph

## 7.1 The Distinction Between Main Page and Sub-Page Must Be Crystal Clear

If the top bar and sub-page bar feel like competing navigation systems, the user experiences double steering-wheel syndrome.

The distinction must be:

* top bar = major world
* sub-page bar = internal district of the current world

## 7.2 Sub-Pages Must Preserve Local Continuity

Switching sub-pages should not feel like leaving the domain.
It should feel like turning toward another face of the same subject.

## 7.3 Sub-Pages Must Share Local Context

The active asset, selection, or working set should persist whenever possible across sub-page switches.

# 8. The Left Rail: Page Ontology and Local Systems

The left side is not universally “tools.”
This is crucial.

The left side is the home of the **current page’s ontology**.

That means it contains the main categories through which that page organizes its local systems.

On one page, those categories may be tools.
On another, they may be object families.
On another, they may be layers of simulation.
On another, they may be structural systems.

## 8.1 The Left Side Represents the Page’s Native Grammar

Ask of any page:

“What are the major classes of thing this page contains?”

The answer belongs on the left.

Examples:

### Image page

* Select
* Paint
* Mask
* Transform
* Retouch
* Effects
* Crop
* Color

### Scene page

* Camera
* Lights
* Atmosphere
* Terrain
* Props
* Characters
* Physics
* Post

### Simulation page

* Emitters
* Constraints
* Materials
* Forces
* Boundaries
* Solvers
* Debug

### Knowledge page

* Outline
* Sources
* Arguments
* References
* Notes
* Claims
* Relations

## 8.2 The Left Rail Must Be Stable in Behavior Even When Its Content Changes

The categories may differ from page to page.
The behavior pattern must remain familiar.

Users should learn that the left edge always means:

“Here are the local systems of this page.”

## 8.3 Mini Bar to Full Drawer Expansion

For dense pages, especially editors, the left side may contain:

* a compact icon rail
* a mini settings strip
* an expandable full drawer

This is a highly canonical pattern.

The small form offers constant reachability.
The larger form offers depth.
The transition preserves anchor.

This is not merely a space-saving trick.
It is a law of civilized power:

**compact presence, expanded depth, preserved location.**

# 9. The Center: The Thing Itself

The center is sacred.

It must show the thing the user is actually working on.

Not the metadata of the thing.
Not an apologetic summary of the thing.
Not a list of settings for the thing.

The thing.

The center may be:

* the image
* the canvas
* the scene
* the document
* the script
* the graph
* the world
* the simulation
* the composition board
* the model
* the waveform
* the timeline composition

## 9.1 Center Must Remain Primary

No shell component may psychologically overpower the center.
The shell exists to support the work, not to become the work.

## 9.2 Center Must Be Interactive, Not Merely Displayed

Whenever possible, the primary artifact should be directly manipulable.
The user should touch the phenomenon, not only its shadow.

## 9.3 Center Must Feel Like an Instrument Chamber

The center is where the app proves it is not a control spreadsheet with fancy borders.

# 10. The Right Side: Persistent Intelligence and Contextual Inspection

This is one of the great revelations of the canon.

The right side is no longer a generic inspector strip.
It is divided into two conceptual species:

## 10.1 Persistent Intelligence

The right side contains a **persistent AI faculty** that survives page changes.

This region may house:

* AI chat
* contextual assistance
* generation prompts
* transformation commands
* explanations
* diagnostics
* command interpretation
* project memory
* selection-aware help
* autonomous tasks and agent actions
* summaries and proposals
* reasoning about the current artifact

This makes intelligence a resident faculty of the environment rather than a modal gimmick.

The assistant is not a page.
The assistant is a power of the application.

## 10.2 Contextual Inspection

Some pages also require local right-side inspection surfaces:

* layers
* selection inspectors
* property panels
* local object detail
* mode-specific side editors
* semantic breakdowns

These are allowed **only when explicitly distinguished** from the persistent intelligence region.

## 10.3 The Critical Distinction

If persistent AI and contextual inspection are mixed into one ambiguous soup, the right side becomes semantically unstable.

Therefore the right side must follow one of these canonical strategies:

### Strategy A: Single persistent AI drawer, with local inspectors elsewhere

Use when the page already has strong left-side systems or center overlays.

### Strategy B: Vertical distinction within the right side

For example:

* upper right = persistent intelligence
* lower right = page-local inspection

### Strategy C: Toggled domain separation

A very clear mode system that allows switching between:

* AI
* Inspect
* Layers
* Analyze

Only valid if the visual distinction is unmistakable and behavior remains stable.

## 10.4 The Right Side Must Feel Trustworthy

The assistant must know where the user is, what is selected, what mode is active, what changed recently, and what the user is trying to do.

AI without context is a confident raccoon with a flashlight.
The canon rejects this creature.

# 11. The Bottom Dock: Time, Process, History, and Becoming

The bottom dock is one of the most powerful pieces of the entire system.

In older software, time appeared only in media tools.
In canonical high-agency software, time becomes a first-class dimension of interface.

The bottom dock is the region of:

* timeline
* history
* undo/redo navigation
* job queue
* background processes
* versions
* variants
* branches
* activity traces
* execution logs
* event sequences
* animation tracks
* state evolution
* process outcomes
* generated artifacts over time

## 11.1 The Bottom Dock Is the Home of Becoming

The left side says what systems exist.
The center shows what is.
The bottom shows what **has happened, is happening, and may happen next**.

That is why it feels alive.

## 11.2 Persistent Bottom, Page-Specific Sections

The bottom dock itself should persist across the app.
But its interior sections may vary by page.

This is the correct refinement.

For example:

### Global bottom capabilities

* history
* tasks
* queue
* activity
* versions

### Image page additions

* edit timeline
* masks timeline
* snapshots

### Script page additions

* scene timing
* beat strip
* draft variants

### Simulation page additions

* state curves
* telemetry log
* playhead
* event markers

### Research page additions

* claim history
* source ingestion queue
* synthesis passes

## 11.3 The Bottom Must Not Become a Junkyard

Everything that has temporal relevance will try to crawl down here like a crab to the sea.
Discipline is mandatory.

The bottom dock must be organized into crisp internal categories, not endless tabs of unrelated debris.

# 12. The Laws of Drawers

Drawers are not hidden closets.
They are **latent fields of power**.

A good drawer is spatially present even when visually minimized.
The user always knows it is there, what category it belongs to, and how to reveal it.

## 12.1 Every Drawer Must Have One Sentence of Meaning

If you cannot describe the drawer’s purpose in one sentence, it is already a mess.

## 12.2 Drawers Must Open From Stable Edges

The edge matters.
It is part of spatial memory.

## 12.3 Drawers Must Preserve Anchor on Expansion

A mini strip expanding into a full drawer must keep the same origin and semantic identity.
It must feel like the same faculty unfolding.

## 12.4 Drawers Must Not Be Storage Units for Miscellaneous Shame

If the drawer contains random controls simply because the team had nowhere else to put them, it has failed.

## 12.5 Drawers Must Be Structured Internally

A high-functioning drawer can have layers:

* layer 1: category selection
* layer 2: subcategory selection
* layer 3: visual editor or contextual surface
* layer 4: optional precision inputs hidden by default

This remains a deeply canonical pattern.

## 12.6 Drawers Must Reveal Depth Without Punishing Discovery

The user should sense that more exists, but not be overwhelmed.

# 13. The Laws of Icon Grammar

Icons are not decoration.
They are a compact language.

Poor icon systems create mysticism and suffering.
Good icon systems create motor memory.

## 13.1 Icons Must Represent Stable Categories

The same icon in the same position should mean the same class of thing over time.

## 13.2 Icons Require Grouping

Ungrouped icon fields become soup.

Canonical grouping may include:

* selection and navigation
* creation and manipulation
* structure and organization
* simulation and behavior
* output and publishing
* AI and meta-tools

## 13.3 Icons Require Active State Clarity

The user must always know:

* what is selected
* what is armed
* what is transiently hovered
* what is disabled
* what belongs to another mode

## 13.4 Icons Require Tooltips or Secondary Labels

The canon forbids symbol priesthood.
Users are not required to enter a secret order to know what a glyph means.

## 13.5 Icons Must Support Motor Memory

The goal is not just recognition.
The goal is embodied reach.
The hand learns. The eye confirms. The mind remains free.

# 13A. The Laws of Iconography and Glyphcraft

Iconography is not garnish.
It is not a decorative sticker set applied after the real design is done.
It is one of the core languages by which the interface teaches the user what kinds of power exist.

The canon therefore rejects cheap, generic, over-colored, over-rounded, obviously AI-generated icon sets that scream synthetic novelty while contributing nothing to clarity.

Such icons are not futuristic.
They are merely noisy.

## 13A.1 The Canon Prefers a Unified Native Icon Language

A serious environment should not feel like it borrowed half its symbols from five unrelated design kits and the other half from a startup pitch deck.

The icon family must feel native to the shell.

That means consistency of:

* optical weight
* corner treatment
* stroke logic
* fill logic
* proportion
* grid discipline
* negative space
* terminal shape
* visual tension
* animation behavior

A true icon canon is a species, not a zoo.

## 13A.2 Baseline Libraries Are Acceptable, But Not Sovereign

A clean system such as Lucide-class iconography is a respectable baseline because it is restrained, legible, and quiet.

But the highest form of the canon is not permanent dependence on a generic external library.

The highest form is a bespoke icon language authored specifically for the environment.

That bespoke language should emerge once the shell, page systems, and domain ontology are mature enough to justify it.

## 13A.3 Color Must Not Be the Primary Identity of Icons

The canon rejects rainbow icon kits whose main job is to shout “look, an icon!” from across the room like caffeinated parrots.

Icons should usually derive their identity from:

* form
* silhouette
* stroke rhythm
* spacing
* grouping
* active state treatment

Color may be used for:

* state
* risk
* category accents
* live conditions
* special semantic escalation

But color must not be used as compensation for weak form.

## 13A.4 Every Icon Must Survive in Monochrome

If an icon only works because it is brightly colored, its form is weak.

The icon must remain understandable in grayscale, low-saturation, and disabled states.

This is the same law as strong object form in material rendering: the thing must still read when chroma is removed.

## 13A.5 Canonical Icon Families Must Be Authored on a Grid But Corrected by Eye

Perfect geometry is not enough.
Icons require optical correction.

This includes:

* apparent centering, not merely mathematical centering
* stroke balancing across complex and simple glyphs
* consistent perceived scale
* corrected diagonals
* corrected circular weight
* stable empty-space relationships

The user reads icons with the eye, not with a coordinate inspector.

## 13A.6 Custom SVG Is the Canonical Medium

The canonical icon substrate is custom SVG.

SVG provides:

* crisp scaling
* direct authorial control
* semantic structure
* stylable strokes and fills
* animation hooks
* compatibility with design-system pipelines

This allows the icon system to become part of the app’s native language rather than a pasted asset pack.

## 13A.7 Animation Must Be Subtle, Semantic, and State-Bound

Animated icons are permitted.
In advanced systems they can be excellent.

But they must obey severe discipline.

Animation is justified when it clarifies:

* active state
* mode transition
* loading or processing
* spatial direction
* live recording or simulation
* object relationship

Animation is not justified merely because the machine can wiggle.

Subtle pulsing, rotational confirmation, morphing between related states, tiny parallax, or restrained material shimmer may be canonical.

Attention-harvesting twitchiness is forbidden.

## 13A.8 Hero Icons and 2.5D Icons Are Special Species

The canon allows advanced icons with:

* subtle depth
* lighting response
* micro-material cues
* shader-based polish
* parallax response
* glass or metallic accents
* low-amplitude 3D presentation

But these belong to special situations:

* hero navigation landmarks
* flagship tools
* premium creation surfaces
* showcase interactions
* core brand primitives

Not every icon deserves to be a tiny opera.

Most icons should remain calm and structural.

## 13A.9 Icon Hierarchy Must Match Functional Hierarchy

The more global the function, the more stable and restrained the icon should be.
The more special the function, the more expressive it may become.

Thus:

* global shell icons should be disciplined and enduring
* page ontology icons may be more domain-specific
* instrument icons may become more descriptive
* hero/brand icons may carry richer material identity

## 13A.10 The Icon Canon Is a Language System, Not a Bag of Symbols

A proper icon canon should eventually specify:

* stroke widths
* radius rules
* fill-versus-stroke rules
* filled-state rules
* active-state rules
* badge logic
* animation grammar
* semantic color rules
* size tiers
* grid sizes
* export rules
* SVG authoring rules
* shader or 2.5D usage rules

Once this exists, iconography becomes architecture rather than decoration.

# 14. The Laws of Visual Instruments

This section carries forward the original prime directive and places it inside the larger shell canon.

## 14.1 The Foundational Commandment

**Never build generic slider/input settings panels as the primary interface for meaningful creative or simulation parameters.**

Numeric inputs are allowed.
But they are not the primary language of control.

## 14.2 Every Important Parameter Must Have a Visible Phenomenon

If a parameter matters, the user should be able to perceive:

* what it changes
* where it acts
* how far it has moved
* what the safe range is
* what the resulting behavior looks like

## 14.3 Direct Manipulation Outranks Abstract Entry

Users should drag the curve, the handle, the node, the force vector, the region, the layer order, the trajectory, the light, the shape, the mask, the anchor point, the timeline clip.

Not because sliders are evil little vegetables.
Because direct manipulation preserves causal understanding.

## 14.4 Visual Instruments Must Be Embodied

An instrument must feel like:

* touching the thing
* steering a behavior
* shaping a relationship
* sculpting a condition

not like typing into a tax form.

## 14.5 Precision Inputs Are Secondary

Numeric inputs belong in:

* advanced foldouts
* precision popovers
* secondary inspector tiers
* keyboard nudges
* explicit expert panels

They must never replace visual agency.

## 14.6 Every Instrument Must Carry Feedback

A proper instrument includes, where relevant:

* live preview
* active drag feedback
* current value display
* bounded zone coloring
* presets
* reset gestures
* keyboard support
* explanatory hint text
* semantic labels
* visual evidence of consequence

# 15. The Laws of AI Presence

Persistent AI is not a gimmick. It is an architectural faculty.

## 15.1 AI Must Be Native to the Shell

The assistant must not feel like an embedded customer support widget wearing a fake moustache.
It must feel like part of the application’s operating system.

## 15.2 AI Must Be Context-Aware

The assistant should understand:

* current page
* current sub-page
* current selection
* current mode
* recent actions
* open artifact
* project-wide intent
* queued jobs
* user goal when inferable

## 15.3 AI Must Be Reachable Without Taking Over

Persistent does not mean dominant.
The assistant should be calm, available, intelligent, and properly bound to context.

## 15.4 AI Must Preserve Conversation Across Page Switches

Because the shell persists, the assistant must remain continuous.
The app should not suffer from route-induced amnesia.

## 15.5 AI Must Produce Actions, Not Just Words

The assistant may:

* explain the current system
* manipulate settings on request
* generate alternatives
* summarize changes
* create assets
* critique the current artifact
* transform the current selection
* search the workspace
* compare variants
* prepare exports
* reason about history

## 15.6 AI Must Never Be Semantically Homeless

Do not scatter AI into unrelated page panels with incompatible identities.
The user should always know where intelligence lives.

# 16. The Laws of Temporal Continuity

Time must not be homeless.

Anything that evolves, branches, queues, runs, simulates, versions, or remembers belongs in the temporal order of the interface.

## 16.1 Every Meaningful Process Has a Temporal Representation

If the system can change over time, that evolution deserves structure.

## 16.2 Undo Is Not Enough

True temporal continuity includes:

* undo/redo
* branching history
* snapshots
* versions
* activity streams
* generated outputs over time
* process telemetry
* asynchronous jobs
* playheads where relevant

## 16.3 The User Must Be Able to Reconstruct Causality

What changed?
When?
Why?
By whom?
Because of what operation?
From what prompt?
Toward what result?

The bottom dock is where this story lives.

## 16.4 Temporal Systems Must Support Both Linear and Branching Thought

Users do not always work in a straight line.
A canonical system must allow:

* timeline sequences
* branches
* alternates
* forks
* variants
* snapshots
* returns

---

# 17. The Laws of State, Selection, and Context

A high-agency application lives or dies by selection intelligence.

## 17.1 Selection Is a First-Class State

The shell, page, assistant, and instruments should all know what the user has selected.

## 17.2 Context Must Cascade Coherently

Global context -> page context -> sub-page context -> local selection -> active instrument.

The user should never feel the app is confused about what object is active.

## 17.3 Selection Must Be Visually Legible

The interface must clearly distinguish:

* hovered
* selected
* active for editing
* focused for keyboard input
* pinned
* referenced
* locked
* ghosted

## 17.4 Context Switching Must Preserve the Right Things

Not everything should persist.
But the right things should:

* current project
* assistant memory
* recent history
* main selection where appropriate
* page-local mode when returning

---

# 18. The Laws of Progressive Disclosure

Progressive disclosure is not hiding things to be clever.
It is a way to preserve legibility while retaining depth.

## 18.1 The User Must Feel the Shape of the Hidden

Hidden depth should be discoverable, not mysterious.

## 18.2 Primary Controls Must Be Visible

Do not hide what is needed for common work.

## 18.3 Advanced Controls Must Be Reachable Without Ritual

A few clicks is acceptable.
A pilgrimage is not.

## 18.4 The Canonical Disclosure Stack

Often the right stack is:

1. visual instrument
2. contextual annotations and presets
3. advanced foldouts
4. numeric precision controls
5. raw technical details

This order preserves intuition while allowing mastery.

---

# 19. The Laws of Motion and Expansion

Motion is not cosmetic. It teaches the user how the world is organized.

## 19.1 Motion Must Explain Structure

When a drawer opens, the animation should clarify where it came from and what changed.

## 19.2 Expansion Must Preserve Identity

A mini strip expanding into a full drawer should feel like growth, not replacement.

## 19.3 Motion Must Be Quick but Legible

Too abrupt and the structure is unclear.
Too slow and the app feels like it is moving through syrup.

## 19.4 Active Indicators Must Be Alive

Pulse, glow, drag feedback, live dots, motion traces, and process indicators may all be used to show living state.

But always with meaning.
Never for casino glitter.

---

# 20. The Laws of Density, Calm, and Power

The highest quality interfaces are often dense.
Density is not the enemy.
Chaos is the enemy.

## 20.1 Dense Interfaces Must Be Semantically Ordered

If every region has a role, density can feel calm.

## 20.2 Calm Comes from Meaningful Boundaries

Sections, rails, tabs, drawers, and docks should create compositional peace.

## 20.3 Power Must Feel Reachable

The user should sense there is depth immediately.
But it should feel graspable, not hostile.

## 20.4 Decorative Minimalism Is Not the Goal

A stark, barren, underpowered interface is not enlightened. It is merely afraid of complexity.

The canon embraces complexity **when the task demands it**, but gives it shape.

---

# 21. The Laws of Visual Semantics

Visual language must communicate hierarchy and meaning with consistency.

## 21.1 Color Must Carry Role, Not Random Mood

Use color to mark:

* active state
* system category
* risk zones
* live telemetry
* selected object family
* semantic grouping

## 21.2 Bounded Parameters Should Use Zone Semantics

Many instruments benefit from explicit regions such as:

* comfort
* strain
* danger

This is especially true for simulations, physics, and constraint-heavy systems.

## 21.3 Signals Must Be Sparse and Interpretable

If every region is vivid, none of them speak.

## 21.4 Surface Hierarchy Must Clarify Depth

Background layers, cards, drawer surfaces, overlays, and workspaces must be materially distinct enough to orient the eye.

# 21A. The Laws of Materiality and Surface Engines

The canon does not stop at flat composition.

A truly advanced browser application may also possess a disciplined material language: surfaces that feel machined, inset, frosted, rubberized, glassy, ceramic, metallic, or softly luminous.

But this must never degenerate into random ornamental gloss.

The mature path is not “add more shadows.”
The mature path is to treat interface surfaces as a **2.5D material-and-light system**.

## 21A.1 Stop Painting Effects. Start Modeling Causes.

Most weak so-called skeuomorphic work paints results directly:

* random highlights
* random shadows
* random gloss bands
* random blur

The canon instead begins from causes:

* shape
* height
* curvature
* material response
* light direction
* contact occlusion
* transmission
* motion state

A surface should be modeled as though it has an implied height field, even when rendered in 2D.

## 21A.2 Height-First UI

Every serious surface should be understandable in terms of relief:

* raised
* inset
* beveled
* filleted
* domed
* concave
* ridged
* chamfered
* micro-textured

From this implied height come:

* normals
* cavity darkness
* highlight compression
* edge brightening
* rim logic
* press deformation
* transmission behavior

This is how the interface avoids shadow soup.

## 21A.3 One Light Rig, Not Rogue Highlight Goblins

A canonical material system requires a coherent light rig.

At minimum it should define:

* key light
* fill light
* ambient lift
* contact occlusion
* optional transmission glow
* optional environment bands for glossy materials

All highlights and shadows must agree about where the light lives.

The canon rejects stray highlight goblins wandering the UI with independent suns.

## 21A.4 Material Taxonomy Must Be Explicit

Not every surface should look like the same glossy candy.
A serious system defines material families with distinct laws.

Canonical families may include:

* soft polymer
* anodized metal
* acrylic glass
* ceramic/enamel
* rubber tactile
* gel capsule
* machined chassis surfaces

Each material family should specify:

* highlight character
* shadow character
* roughness
* transmission
* deformation behavior
* microtexture
* edge response

## 21A.5 Surface Engines Need Tiers

Not every control requires heroic rendering.

The canon therefore recognizes multiple render tiers:

### Tier 1: CSS material composition

For broad deployment and everyday premium surfaces.

### Tier 2: Procedural surface assets

For generated grain, brushed metal, bevel masks, and reusable texture atlases.

### Tier 3: WebGPU hero rendering

For flagship panels, premium toggles, live glass, transmission, caustic approximation, and pointer-reactive material response.

The shell should be intelligent enough to use richer rendering where it matters, not everywhere all at once.

## 21A.6 CSS Surfaces Must Remain Disciplined

In CSS mode, a premium control should normally be composed from a restrained stack such as:

* layered backgrounds for base material and sheen
* inset shadows for cavity or bevel logic
* one contact shadow
* one cast shadow
* optional pseudo-element for polish, specular, or transmission haze

Not fifty unrelated shadows.
That is decorative tax fraud.

## 21A.7 Contact Shadow and Cast Shadow Are Different Species

This law is absolute.

* contact shadow = tight, dark, local, grounding
* cast shadow = broader, softer, spatial, projecting away from the object

If they are confused, the surface loses physical credibility.

## 21A.8 Motion Must Reinforce Material and Mass

A premium surface is not fully defined until it moves.

Press, hover, drag, toggle, and release should affect:

* compression
* highlight drift
* contact shadow tightness
* cast shadow length
* caustic energy where relevant
* slight deformation where justified

Motion must make the object feel weighted.
Not haunted.

## 21A.9 Caustics Must Be Understood Honestly

True physically correct caustics are not the aim of normal browser UI.
What the canon seeks is **plausible concentrated light behavior** under transmissive materials.

For glassy or gel-like surfaces, the system may generate:

* underside light pooling
* focused refractive glow
* curvature-driven highlight concentration
* subtle animated light drift

In CSS this is an approximation.
In GPU-backed hero rendering it may become a true dedicated pass.

The goal is not a cathedral window.
The goal is a believable luminous interface surface.

## 21A.10 The Highest Form Is a Material-and-Light Graph

At the highest level, surfaces should be authored not as disconnected CSS hacks but as a graph of:

* 2D geometry
* implied height
* material model
* light rig
* shadow rig
* interaction state
* motion behavior
* post-finish treatment

Such a graph may then compile into:

* CSS
* procedural assets
* WebGPU render passes

This is the mature future of advanced browser interface rendering.

## 21A.11 Surface Richness Must Serve Meaning

Material richness is justified when it helps communicate:

* hierarchy
* affordance
* premium control feel
* active state
* danger or fragility
* spatial depth
* the identity of flagship actions

It is not justified merely to show that the machine can impersonate a glass marble.

## 21A.12 The Best Surface Systems Feel Luxurious But Still Quiet

The interface may become richer than most contemporary web chrome.
It may even rival premium game UI in moments of emphasis.

But it must remain controlled, legible, and structurally honest.

The purpose of advanced material rendering is not spectacle alone.
It is to deepen the felt reality of the workspace.

# 23. The Laws of Modes, Tools, and Subsystems

Modes are dangerous but sometimes necessary.

## 23.1 Modes Must Be Explicit

If the user is in a tool mode, edit mode, selection mode, playback mode, or agent mode, the interface must announce it.

## 23.2 Page Subsystems Must Be Locally Coherent

Each page can have multiple subsystems, but they must form a comprehensible family.

## 23.3 Tools Must Not Hijack the Whole Environment Without Cause

A brush tool changing the entire app’s global identity is absurd.
A page-local tool should remain page-local unless it truly changes the environment’s broader state.

# 24. The Laws of Multi-Panel Pages

Some pages need more complexity than others.
The image editor is a classic stress test.

## 24.1 Multiple Side Systems Are Allowed

For dense pages, it is acceptable to have:

* left rail
* left expanding settings drawer
* center workspace
* persistent AI on the right
* local right contextual surfaces
* bottom temporal dock

This is valid **only if the meaning of each region is disciplined**.

## 24.2 Separation Must Be Visible

When the right side contains both persistent AI and page-local inspection, the distinction must be architecturally obvious.

## 24.3 The User Must Never Ask “What Kind of Panel Is This?”

That question is a sign of semantic collapse.

# 25. The Laws of Editors, Simulators, and Worldbuilding Tools

These domains are natural citizens of the canon.

## 25.1 Editors Must Expose Causality

A good editor lets the user see what causes what.

## 25.2 Simulators Must Show Living State

Simulation tooling should reveal:

* current runtime condition
* trajectory
* stress zones
* active forces
* telemetry
* thresholds
* present state versus configured state

## 25.3 Worldbuilding Tools Must Preserve Layered Ontology

Worldbuilding often spans:

* terrain
* materials
* weather
* characters
* logic
* animation
* narrative
* systems
* ecological layers
* lighting
* physics

The shell-plus-page model is exceptionally suited to this because each world can have its own ontology while sharing the persistent shell.

## 25.4 The Canon Prefers Instrument Panels Over Generic Settings Forms

If a cloth simulator is mostly sliders, it is under-evolved.
If a planing physics editor exposes live graphs, interactive handles, and telemetry, it is on the canonical path.

# 26. The Laws of Documents, Planning, and Knowledge Workspaces

The canon is not only for visual art tools.
It also applies to serious thinking environments.

## 26.1 Knowledge Work Needs Spatial Order Too

A knowledge page may use:

* left = structure, sources, graph, argument modules
* center = current text, synthesis, or canvas
* right = persistent AI critique and help
* bottom = revision history, claims timeline, research queue

## 26.2 Documents Can Also Be Instrumental

Even text can be treated as a living artifact when accompanied by:

* structure maps
* relation views
* source traces
* variant drafts
* AI reasoning assistance
* chronology of revisions

## 26.3 Planning Systems Benefit from Temporal Docking

Plans, production boards, scenario maps, and story systems gain enormous power when time/history/process are explicit.

# 27. The Laws of AI-Native Creation Environments

An AI-native environment is not one that merely contains a chatbot.

It is one in which intelligence is woven into the structure of the shell and the flow of work.

## 27.1 AI Must Interact with Artifacts, Not Float Above Them

The assistant should be able to reference:

* the selected layer
* the active region of the image
* the current node group
* the active simulation subsystem
* the selected paragraph or scene
* the currently visible timeline segment

## 27.2 AI Must Be Able to Traverse the Shell

It should understand and possibly act across:

* pages
* sub-pages
* instruments
* history
* queued jobs
* versions

## 27.3 AI Must Help Maintain Continuity

The assistant can become the thread that remembers:

* why a variant was created
* why settings were changed
* what the user was exploring
* what branch is promising
* what failed earlier

This is not cosmetic. It changes the character of software.

# 28. The Laws of Error, Risk, Safety, and Recovery

High-agency software must not merely enable action. It must support survival.

## 28.1 Errors Must Be Locatable

When something fails, the user must know:

* what failed
* where it failed
* whether it was local or global
* whether data is at risk
* what can be retried

## 28.2 Destructive Acts Must Have Temporal Dignity

Deletion, overwrite, replacement, and large transformations should be recoverable through strong history systems.

## 28.3 Dangerous Parameter Zones Should Be Legible

When domains include physical, visual, computational, or logical danger ranges, the user should be able to see them.

## 28.4 AI Actions Must Be Explainable and Reversible

If the assistant changes the artifact, the user should be able to inspect and undo the change.

---

# 29. The Laws of Accessibility and Humane Reachability

The canon is not complete unless it is humane.

## 29.1 Keyboard Reach Must Exist

Dense interfaces require excellent keyboard reachability.

## 29.2 Focus States Must Be Visible

Never let focus vanish into the dark like a frightened mouse.

## 29.3 Icons Must Be Named

Tooltips, labels, accessibility names, and semantic clarity are required.

## 29.4 Hit Targets Must Respect Human Hands

Tiny targets are not elite. They are rude.

## 29.5 Visual Meaning Must Not Depend on Color Alone

Use shape, contrast, label, motion, and hierarchy alongside color.

---

# 30. The Laws of Responsiveness and Screen Adaptation

The canon originates in rich desktop-class browser environments, but it must adapt intelligently.

## 30.1 Responsiveness Must Preserve Semantics, Not Just Rearrange Boxes

When screen size changes, the meaning of regions should remain legible.

## 30.2 Compression Order Must Be Deliberate

On narrower screens, the likely order of compression is:

1. collapse local drawers to icon rails
2. reduce persistent dock height
3. move some local panels into toggled overlays
4. preserve center workspace as strongly as possible
5. keep AI reachable but not dominant

## 30.3 Mobile Is Not the Primary Canonical Form

This doctrine is natively aimed at advanced browser environments where real work occurs.
Small-screen adaptations are descendants, not the sovereign form.

---

# 31. The Laws of Implementation Architecture

A profound interface requires corresponding architectural discipline.

## 31.1 State Must Reflect the Shell/Page/Instrument Hierarchy

At minimum, state should be separable into:

* shell state
* page state
* local selection state
* instrument state
* temporal state
* AI context state

## 31.2 Cross-Page Faculties Must Be Truly Global

Persistent AI, job queue, history, and project identity cannot be faked as local page components.
They must actually live above page routing.

## 31.3 Context Plumbing Must Be Intentional

The assistant, dock, and local inspectors need access to the correct slices of state without degenerating into spaghetti horror.

## 31.4 Instrument Components Must Be Reusable but Domain-Aware

The shell can be shared broadly.
Instruments must remain specific to their phenomenon.

## 31.5 Versioning and Actions Must Be Structured

A canonical application benefits from explicit action records, snapshots, and event streams.
This feeds both history and AI reasoning.

# 32. Anti-Patterns and Heresies

These are forbidden patterns or severe warnings.

## 32.1 The Everything Inspector

A giant right panel where every possible setting is dumped.
This is not power. It is landfill.

## 32.2 The Semantic Shape-Shifter Drawer

A drawer that means AI on one page, export on another, layers on another, and random controls on Tuesday.
This destroys spatial trust.

## 32.3 The Top-Bar Junk Parade

Stuffing global and local actions into the top bar until it becomes a panicked airport departures board.

## 32.4 The Symbol Cult

All-icon navigation with no labels, no grouping, no hints, no mercy.

## 32.5 The Spreadsheet of Doom

A “pro interface” composed mostly of sliders, number boxes, and collapsible accordions.
This is anti-canonical unless the domain is genuinely tabular.

## 32.6 The Bottom Junk Peninsula

Logs, notifications, exports, layers, comments, versioning, AI, analytics, and jobs all stuffed into the bottom without ontology.

## 32.7 The Overeager AI Court Jester

An assistant that constantly interrupts, autoscrolls, suggests nonsense, or speaks without clear context.
Persistent intelligence must be dignified.

## 32.8 The Page Reset Curse

Changing pages and losing the sense of where you were, what was selected, what the assistant knew, and what process was underway.

# 34. The Test of Legibility

A page passes the test of legibility if a user can answer, without hesitation:

* What world am I in?
* What submode am I in?
* What do the left-side systems represent?
* What is the center artifact?
* What kind of thing lives on the right right now?
* What does the bottom region represent here?
* What is selected?
* What is happening live?

If these answers are muddy, the design is not mature yet.

---

# 35. The Test of Continuity

A canonical application preserves continuity across movement.

A page switch should not cause identity collapse.

The user should feel:

* same project
* same assistant
* same temporal continuity
* same shell
* new domain territory

That is the signature of the system.

---

# 36. The Test of Agency

A canonical interface does not merely let the user configure things.
It lets the user **act with understanding**.

An interface passes the test of agency when the user can:

* directly manipulate important phenomena
* see the consequences of action immediately
* move across pages without losing mental orientation
* use AI as a true collaborator
* reconstruct what happened over time
* reach both common and advanced actions fluidly
* maintain flow under complexity

---

# 37. The Test of Beauty

Beauty in this canon is not prettiness.
It is the felt experience of ordered power.

A beautiful interface is one in which:

* every region has meaning
* no region lies about its purpose
* the center remains sacred
* drawers feel latent, not cluttered
* icons feel grammatical, not mystical
* AI feels present, not noisy
* time feels housed, not scattered
* tools feel embodied, not bureaucratic
* complexity feels orchestrated, not dumped

Beauty is what happens when ontology, interaction, and form stop arguing.

# 38. Final Commandments

1. **Build for high-agency browser work, not generic app fashion.**
2. **Preserve a persistent cognitive shell across the environment.**
3. **Assign stable semantic roles to the edges of the screen.**
4. **Use the top bar for worlds, not local clutter.**
5. **Use the second top row for local territories within the active world.**
6. **Use the left side for the page’s ontology and local systems.**
7. **Keep the center devoted to the thing itself.**
8. **Make the right side the home of persistent intelligence, with explicit distinction for local inspection when needed.**
9. **Make the bottom dock the home of time, history, process, jobs, and becoming.**
10. **Treat drawers as latent fields of power, not storage bins.**
11. **Treat icons as grammar, not decoration.**
12. **Never make important control surfaces generic slider forms.**
13. **Make visual instruments the primary means of manipulation.**
14. **Allow precision inputs, but demote them beneath direct manipulation.**
15. **Ensure AI is contextual, calm, persistent, and capable of action.**
16. **Ensure time is visible and reconstructable.**
17. **Preserve continuity of project, context, and assistant across page switches.**
18. **Allow page-specific complexity inside a globally stable shell.**
19. **Reject semantic ambiguity at all screen edges.**
20. **Build a bespoke icon language; never rely on cheap synthetic icon noise as identity.**
21. **Treat premium surfaces as a disciplined material-and-light system, not random shadow decoration.**
22. **Design for ordered power.**

## Appendix C: The Icon Canon Specification

This appendix translates the doctrine of iconography into a design-system spec.

### C.1 Icon Roles

The icon system should be divided into four role classes:

#### C.1.1 Shell Icons

Used for global navigation, universal shell faculties, persistent AI, project scope, and enduring world landmarks.

Properties:

* maximally stable
* quiet
* highly legible
* restrained
* low novelty

#### C.1.2 Page Ontology Icons

Used for left-rail categories and page-native system families.

Properties:

* domain-specific
* still governed by the global icon grammar
* slightly more descriptive than shell icons

#### C.1.3 Instrument Icons

Used inside tools, inspectors, mode bars, and local editor subsystems.

Properties:

* compact
* semantically tight
* optimized for repeated expert use

#### C.1.4 Hero or Landmark Icons

Used sparingly for flagship actions, premium moments, major AI faculties, or brand primitives.

Properties:

* may carry subtle depth
* may use richer finish or animation
* must still remain structurally coherent with the main family

### C.2 Canonical Construction Rules

A native icon language should define, at minimum:

* base grid size
* size tiers
* stroke widths by tier
* stroke cap style
* stroke join style
* corner radius rules
* diagonal treatment rules
* circle and curve correction rules
* fill-versus-stroke rules
* terminal treatment rules
* cutout logic
* negative-space minimums

### C.3 Recommended Grid and Tier Strategy

A strong starting system for browser-class high-agency UIs is:

* 16px micro tier for dense inspector use
* 18px or 20px standard tier for default rails and controls
* 24px emphasis tier for major navigation or key actions
* 32px+ landmark tier for hero moments

All icons should be authored against a common grid but optically corrected by eye.

### C.4 Stroke and Form Doctrine

The system should choose one dominant family and commit:

#### Option A: Stroke-led family

* calm
* technical
* airy
* excellent for shell and ontology rails

#### Option B: Hybrid family

* stroke-led by default
* selective fills for active states and high-salience functions
* often the best choice for premium browser applications

#### Option C: Fill-led family

* stronger silhouette
* higher density
* useful in some compact contexts
* riskier for large systems unless extremely disciplined

The canon generally favors **stroke-led or hybrid** families for complex AI-assisted environments.

### C.5 Active-State Rules

A complete icon system must define:

* inactive state
* hovered state
* pressed state
* active/selected state
* disabled state
* destructive state
* live-processing state
* notification/badged state

The active state should normally change by a combination of:

* background shape or chip
* line or fill emphasis
* semantic accent color
* subtle material elevation
* optional micro-animation

Do not rely on color alone.

### C.6 Badge Grammar

Badges must be standardized.
Possible canonical badges:

* dot = unread, unsaved, or pending
* count pill = quantitative state
* pulse ring = live or recording
* spark marker = AI-suggested or generated content
* warning wedge = caution or conflict

Badges must never corrupt the icon silhouette beyond recognition.

### C.7 Icon Animation Grammar

Permitted icon motions include:

* soft pulse for live state
* rotational settle for confirmation
* tiny directional slide for reveal/collapse
* morph between two related semantic states
* shimmer or highlight drift for hero landmarks
* orbit or signal wave for intelligence, connectivity, or processing

Animation must be:

* low amplitude
* state-driven
* sparse
* interruptible
* never ornamental noise

### C.8 Monochrome Test

Every icon must pass all of the following:

* readable at 1x and 2x scale
* readable in monochrome
* readable in disabled opacity
* readable in active chip state
* readable against both dark and light shell variants

### C.9 SVG Authoring Rules

Each SVG should satisfy:

* clean viewBox discipline
* no accidental fractional chaos unless intentional for optical correction
* named layers where useful
* reusable path structure where practical
* stroke expansion policy defined for export
* no unnecessary clipping or masking complexity for standard icons

### C.10 Icon System Deliverables

A mature icon canon package should include:

* master SVG library
* React icon wrapper components
* sizing tokens
* state tokens
* animation primitives
* badge primitives
* documentation plates
* do/don’t examples
* optical-correction notes

## Appendix D: The Surface Engine Specification

This appendix translates materiality doctrine into an implementation-grade system.

### D.1 The Surface Stack

Every major interface surface should be describable through a stack of causes:

1. geometry
2. implied height field
3. material family
4. light rig response
5. occlusion/contact behavior
6. motion behavior
7. interaction state modifiers
8. optional post-finish treatment

This replaces ad hoc styling with a coherent rendering model.

### D.2 Canonical Material Families

A premium UI environment may define a limited set of material species such as:

#### D.2.1 Soft Polymer

Use for calm controls, chips, toggles, modern soft industrial surfaces.

Traits:

* broad highlight
* soft cavity shading
* restrained specular
* low visual aggression

#### D.2.2 Anodized Metal

Use for chassis surfaces, precision controls, pro-grade toolbars, premium housings.

Traits:

* directional sheen
* firmer edge response
* controlled reflection banding
* stronger premium technical character

#### D.2.3 Acrylic Glass

Use for overlays, floating inspectors, AI capsules, high-end panels, transmission surfaces.

Traits:

* transmissive blur
* refractive edge energy
* subtle underside light pooling
* luminous premium feel

#### D.2.4 Ceramic / Enamel

Use for flagship buttons, focus controls, deliberate hero actions.

Traits:

* clean glossy shell
* elegant highlight compression
* dense finished presence

#### D.2.5 Rubber / Tactile Matte

Use for drag handles, scrubbers, simulation grips, tactile manipulation anchors.

Traits:

* low sheen
* strong affordance
* press credibility
* grounded feel

### D.3 Light Rig Tokens

The system should define explicit light tokens rather than improvising per component.

Core tokens:

* key light direction
* key light intensity
* fill lift
* ambient lift
* contact occlusion strength
* cast shadow softness
* cast shadow offset logic
* transmission glow strength
* highlight sharpness by material

### D.4 Height and Edge Tokens

A real surface engine must define shape primitives such as:

* base height
* press depth
* bevel depth
* bevel softness
* corner radius family
* dome curvature
* cavity depth
* rim lift
* inset strength

These tokens allow shared material logic across buttons, chips, drawers, cards, and hero controls.

### D.5 Render Tiers

#### D.5.1 Tier One: CSS Material System

Use for most shell surfaces and everyday controls.

Typical techniques:

* layered gradients
* restrained inset and outset shadows
* pseudo-elements for sheen or transmission haze
* backdrop blur where appropriate
* composited material tokens

#### D.5.2 Tier Two: Procedural Texture and Surface Assets

Use for:

* brushed metal
* micrograin
* authored bevel masks
* reusable polish maps
* subtle noise or anisotropic response

This tier bridges pure CSS and GPU hero rendering.

#### D.5.3 Tier Three: WebGPU Hero Surfaces

Use selectively for:

* flagship navigation landmarks
* premium hero controls
* live 2.5D icons
* high-end glass capsules
* pointer-reactive transmission
* physically suggestive lighting effects
* surface editors and showcase panels

The shell should escalate to this tier only where payoff is high.

### D.6 Shadow Doctrine

A surface system must separate:

#### Contact shadow

* darkest
* smallest
* local grounding
* strongest during press or near-surface contact

#### Cast shadow

* broader
* softer
* spatially displaced
* communicates lift and hierarchy

#### Interior shadow

* inset
* communicates cavity, bevel, or recess

These must not be randomly blended into one muddy fog.

### D.7 Motion and Material Coupling

Material behavior must respond to interaction:

* hover may shift highlight and light pressure
* press may compress height and tighten contact shadow
* drag may stretch highlight direction or deepen grip state
* toggle may move luminous emphasis across the object
* loading may create slow intelligent circulation or energy flow

### D.8 Caustic and Transmission Approximation

For premium glass or gel-like surfaces, the system may include:

* underside glow accumulation
* edge-bright refractive suggestion
* curved light pooling
* subtle chromatic separation at hero tier only
* pointer-reactive internal highlight drift

These effects must remain plausible and restrained.

### D.9 Surface Engine Deliverables

A mature surface-engine package should include:

* material taxonomy sheet
* light-rig tokens
* height tokens
* component recipes
* CSS utility patterns
* shader escalation rules
* WebGPU hero component library
* performance budgets
* accessibility contrast notes
* do/don’t comparison plates

## Appendix F: The Implementation Mapping

This appendix turns the canon into architectural guidance for modern browser application development.

### F.1 State Domains

The app should separate state into at least:

* shell state
* page route state
* sub-page state
* selection state
* drawer state
* dock state
* AI context state
* temporal/history state
* instrument-local state
* surface/theme state

### F.2 Component Domains

A mature codebase should likely distinguish:

* shell frame components
* page layout components
* ontology rail components
* drawer and dock primitives
* right-side intelligence components
* instrument components
* timeline/process components
* icon system components
* surface engine primitives

### F.3 React Design-System Deliverables

The canonical implementation should eventually include:

* shell layout primitives
* animated drawer primitives
* tab and sub-page systems
* AI dock primitives
* bottom temporal dock primitives
* icon package
* material surface package
* instrument composition kit
* blueprint/demo playground pages

### F.4 Performance Doctrine

The highest-quality environment must remain fast.
Therefore:

* premium materials should be tokenized and reused
* GPU hero rendering must be isolated to high-payoff moments
* expensive blur and shadow stacks must be audited
* animation must avoid layout thrash
* state subscriptions must remain disciplined
* context propagation must be deliberate

### F.5 The Final Engineering Principle

**The shell should be universal, the pages should be domain-native, the instruments should be specific, and the rendering richness should scale according to semantic importance and performance budget.**

## Appendix G: Canonical Token Tables

This appendix provides a concrete baseline token system for implementing the canon in a modern browser application.

These values are not sacred constants.
They are a disciplined starting lattice.
A mature product may tune them, but it should do so systematically rather than improvisationally.

### G.1 Spacing Scale

The canonical spacing system should be compact enough for high-agency work while preserving breathability.

Recommended base unit: **4px**

Recommended scale:

* 2 = 8px micro gap
* 3 = 12px tight gap
* 4 = 16px default interior spacing
* 5 = 20px comfortable spacing
* 6 = 24px major control spacing
* 8 = 32px section spacing
* 10 = 40px panel macro spacing
* 12 = 48px major shell rhythm

Design rule:

* use 8–16px inside dense controls
* use 16–24px inside drawer sections
* use 24–40px between major layout bands

### G.2 Radius Scale

The application should define a clear radius family.

Recommended baseline:

* r1 = 6px for compact controls
* r2 = 10px for chips and minor panels
* r3 = 14px for drawers and cards
* r4 = 18px for premium shell surfaces
* r5 = 24px for hero controls or landmark modules
* r6 = 32px+ for special showcase capsules only

Rule:
small controls should not become cartoon pills unless the material language explicitly calls for it.

### G.3 Typography Scale

A practical canonical type ladder for dense browser environments:

* t0 = 11px metadata / micro labels
* t1 = 12px compact control labels
* t2 = 13px default UI labels
* t3 = 14px standard panel text
* t4 = 16px emphasized labels / compact headings
* t5 = 18px section headers
* t6 = 22px page headers
* t7 = 28px major page hero title

Recommended weights:

* 450–500 for default labels
* 550–600 for active section labels
* 650–700 for page or panel headers

### G.4 Icon Size Tiers

Recommended icon ladder:

* i0 = 14px micro state or inline support icons
* i1 = 16px dense inspector tier
* i2 = 18px default action tier
* i3 = 20px shell and left-rail default tier
* i4 = 24px emphasis tier
* i5 = 32px hero landmark tier

### G.5 Shell Width and Height Budgets

Recommended desktop shell baselines:

* top bar height = 52px to 60px
* sub-page bar height = 36px to 44px
* left icon rail width = 52px to 64px
* left expanded drawer width = 280px to 420px
* right persistent AI drawer width = 320px to 440px
* lower-right contextual drawer width = 280px to 380px
* bottom dock collapsed height = 44px to 56px
* bottom dock standard open height = 180px to 260px
* bottom dock expanded analytical height = 300px to 420px

These are not random numbers.
They reflect a balance between density, readability, and center-workspace preservation.

### G.6 Center Preservation Rule

On desktop-class layouts, the center workspace should ideally retain at least:

* 45% of total width in complex pages
* 55%+ of total width in standard pages
* 65%+ in immersive world, canvas, or viewport pages

If the center falls below this for routine interaction, the shell is over-consuming the artifact.

### G.7 Drawer Interior Rhythm

A strong default drawer rhythm may use:

* 16px outer padding on dense drawers
* 20px outer padding on standard drawers
* 24px outer padding on premium drawers
* 10–12px between compact controls
* 16px between control groups
* 24px between drawer sections

### G.8 Interactive Height Tokens

Recommended control heights:

* h1 = 28px compact utility control
* h2 = 32px dense control
* h3 = 36px standard control
* h4 = 40px major action control
* h5 = 48px hero action control

### G.9 Elevation Tiers

A mature surface system should define consistent elevation tiers.

Example:

* e0 = base shell plane
* e1 = raised controls and local chips
* e2 = cards and floating inspectors
* e3 = active drawers and dock overlays
* e4 = high-priority transient overlays
* e5 = modal interruption layer
* e6 = debug/emergency overlay tier

Each elevation tier should define:

* shadow recipe
* border/edge treatment
* blur/transmission rules if applicable
* motion response

### G.10 Z-Index Strata

The system must establish real strata rather than ad hoc z-index numerology.

Example strata:

* z0 = base workspace
* z10 = local ornament and minor overlays
* z20 = drawers and standard floating panels
* z30 = dock overlays and prominent inspectors
* z40 = shell-level transient surfaces
* z50 = command palette / AI action sheets
* z60 = modals / confirmations
* z70 = emergency blockers / global recoveries

Never let teams invent one-off z-index values like cursed numerological graffiti.

### G.11 Motion Timing Tokens

Recommended motion tokens:

* fast-1 = 90ms for tiny state flicks
* fast-2 = 140ms for compact hover/press transitions
* standard-1 = 180ms for chips, tabs, and local state shifts
* standard-2 = 220ms for drawer section changes
* major-1 = 280ms for rail-to-drawer expansion
* major-2 = 360ms for dock expansion or shell-level reveals
* slow-1 = 500ms max for deliberate hero material responses

Easing recommendations:

* use snappy ease-out for local control response
* use gentler deceleration for shell movements
* use spring-like motion only when it supports tactility rather than bounce theatrics

### G.12 Blur, Noise, and Sheen Limits

A premium shell must constrain these effects.

Recommended policy:

* only one or two blur strengths for major surfaces
* only one subtle grain family for premium materials
* sheen layers limited to material families that truly need them
* never stack blur, gloss, large shadows, and animated gradients without semantic reason

### G.13 Status Color Roles

The system should assign durable semantic color roles, such as:

* neutral
* active/focus
* success
* caution
* danger
* AI/generated
* live/recording
* disabled
* ghost/reference

These roles must apply consistently across icons, badges, telemetry, docks, and instruments.

### G.14 State Opacity Rules

Recommended baseline opacity logic:

* disabled controls = 35–50%
* ghost/reference surfaces = 45–70%
* inactive secondary metadata = 60–75%
* active primary content = 100%

Opacity should not be used to solve every hierarchy problem.
Shape, size, spacing, and material must also participate.

## Appendix I: Region Admissibility Tables

This appendix makes the canon brutally explicit by stating what belongs where.

### I.1 Top Bar Admissibility

Belongs here:

* main world navigation
* project identity
* global search or command access
* user/account access
* high-level publish/share/export if truly global

Does not belong here:

* page-specific sliders
* local inspector toggles
* random minor formatting controls
* deep tool settings

### I.2 Sub-Page Bar Admissibility

Belongs here:

* internal modes of the active page
* stable local route families
* view-family switches within a page

Does not belong here:

* global navigation
* ephemeral tool states
* settings that should live inside a drawer

### I.3 Left Side Admissibility

Belongs here:

* local systems
* ontology categories
* tool families
* structure groups
* domain-native object classes

Does not belong here:

* universal AI chat
* global notifications
* temporal job queues
* unrelated export functions

### I.4 Center Admissibility

Belongs here:

* primary artifact
* direct manipulation overlays
* live viewport or canvas
* central work surface

Does not belong here:

* giant permanent settings walls
* decorative shell clutter competing with the artifact

### I.5 Right Side Admissibility

Belongs here:

* persistent AI
* contextual inspection
* local semantic detail
* assistant actions and reasoning

Does not belong here:

* random overflow for controls no one placed thoughtfully
* global history or jobs that belong in the bottom dock
* page ontology categories that belong on the left

### I.6 Bottom Dock Admissibility

Belongs here:

* history
* versions
* activity
* jobs
* queues
* traces
* timeline systems
* variants
* outputs over time

Does not belong here:

* arbitrary navigation
* generic inspector clutter
* unrelated static settings

## Appendix K: Canonical Performance Budgets

A premium interface must remain fast enough to feel inevitable.

### K.1 Material Budget Rules

* use GPU hero surfaces only for high-value landmarks or flagship controls
* keep heavy blur regions limited in count and area
* avoid stacking multiple expensive filters in nested surfaces
* prefer tokenized shared shadow and gradient recipes

### K.2 Animation Budget Rules

* animate transform, opacity, filter only when justified
* avoid layout-affecting animation for shell structure when possible
* cap simultaneous ambient animations in dense pages
* reduce motion automatically in low-power conditions where feasible

### K.3 Drawer and Dock Budget Rules

* opening or switching a drawer should not cause full-page rerenders
* dock section switches should preserve offscreen state where useful
* persistent AI should stream context updates intelligently rather than reprocessing indiscriminately

### K.4 Reality Principle

The best interface in theory is worthless if it stutters like a haunted filing cabinet.

Performance is not a technical afterthought.
It is part of the aesthetic.
